⑥ 构建历程与工程方法

Engram 如何从一行 scaffold 提交成长为一个包含 10 个 Go 包、32 个源文件、29 个测试文件、2 个可运行二进制的生产级系统——以及背后那套严格的 spec→plan→subagent→review 工程规程。

1 · 路线图 M1–M5 与分层落地

2 · 工程规程（The Loop）

每一层的开发都严格遵循同一个五阶段循环，没有例外。

2.1 五阶段循环

flowchart LR
    A["brainstorming
意图 + 边界探索"]
    B["writing-plans
TDD 任务列表
bite-sized 每任务单独可绿"]
    C["subagent execution
每任务新鲜子 agent
test-red 到 test-green"]
    D["spec-review + code-review
合规审查 + 质量审查
opus final pass"]
    E["merge --no-ff
删分支"]

    A --> B --> C --> D --> E
    E -->|"下一层"| A

各阶段要点：

  • brainstorming（意图探索）：不急着写代码——先探清用例边界、接口形状、与已有包的关系。产物是一份 spec（docs/superpowers/specs/），每个 spec 章节必须含 Mermaid 架构图或时序图，不允许只有文字 + 代码片段。
  • writing-plans（实现计划）：spec 确认后，把实现拆成一个个 bite-sized TDD 任务（docs/superpowers/plans/）。每个任务的验收条件是"这个任务单独跑测试可以绿"——不允许"等后面任务一起测"。
  • subagent-driven execution（子 agent 执行）：每个任务派发给一个新鲜 subagent，没有前置会话上下文的干扰。TDD 顺序严格：先写测试（看着它失败），再写实现，再看测试变绿。每个任务对应独立的 feat/fix/refactor/test commit。
  • 两阶段审查：第一阶段 spec 合规审查（实现是否覆盖了 spec 的所有设计点），第二阶段代码质量审查（错误传播、%w 包装、接口大小、测试覆盖度）。通常合并为一次 opus 整体实现审查，产出一批 fix/refactor commit 然后才 merge。
  • 本地 --no-ff merge + 删分支：不用 GitHub PR 合并——本地执行，保留合并提交（主线可见分层历史），然后删除功能分支，不留悬空引用。

2.2 编码规范（TDD 五戒）

  1. test first, watch it fail — 任何新接口都先写测试，先跑失败，再写实现。
  2. %w 包装错误 — 所有 fmt.Errorf 必须用 %w，上层可用 errors.Is / errors.As 精确匹配。
  3. context.Context 第一参数 — 所有 I/O 函数第一参数为 ctx context.Context，无例外。
  4. 小接口 — 包边界只暴露最小接口；内部实现不泄漏到接口。
  5. 表驱动测试 — 多场景覆盖必须用 table-driven tests（[]struct{name, input, want}），不用大量重复函数。

3 · 审查链抓到的真实 bug

以下是审查或测试阶段捕获的具有代表性的缺陷，均有对应 fix commit 可溯源。

缺陷描述 问题根因 修复方式 Commit
Advisory lock pooled-conn 泄漏 WithGlobalLock 的 defer unlock 使用了 request ctx——若 ctx 已被取消，unlock SQL 无法执行，锁永久持有，连接归还连接池后仍锁住 unlock 改用 context.Background()，与 caller ctx 生命周期解耦 6f8c410

Reflect 提交空内容 / 传播读错误 读 resident 文件失败时 Reflect 静默忽略错误继续执行，提交空内容造成记忆损坏；或将空 resident 提交为"整合后"内容 读错误立即返回（不跳过）；resident 为空时直接 return，不发起 LLM 调用 d601f50

Defrag 收敛性注释错误 代码注释说 defrag 按"文件大小"判断是否可拆分（size-based），实际是按"结构"（是否有多个顶级标题）——大小判断无法保证收敛，小文件多标题仍会反复触发 注释改为正确的 structural 收敛性描述；isSplittable 确认只在有多个顶级 heading 时返回 true（拆后每文件只有一个，不再触发） 9d1688c
GC 扫描误删 embedding store 全局 mark-sweep GC 遍历对象存储时，embedding cache 的 ObjCache 与主 objstore 用了不同根路径，但 GC 缺少根路径区分，可能将 embedding 对象识别为不可达孤儿并删除 GC 流程增加 distinct-root guard：filepath.Clean(ENGRAM_EMB_OBJ) vs filepath.Clean(ENGRAM_OBJ) 不同时才允许共用存储路径；harden(cmd/maintenance) 用 filepath.Clean 比较 25286b8
CAS conflict 被静默覆盖 Session 早期实现在 CAS 失败时直接用本地视图覆盖 HEAD，不返回 ErrConflict——并发写者的提交会被无声丢弃 CAS 失败显式返回 ErrConflict；Session 捕获后重试（有上限），超出重试次数才报错 1d58581
read 工具 0-based vs 1-based 不一致 recall 工具返回的行号是 1-based（与 git diff / grep 一致），但 read 工具内部使用 0-based 偏移——模型传入 recall 的行号直接传给 read 时差一行 read 工具改为 1-based，与 recall、ToolDef 文档统一；测试用例同步修正（start=5 而非 start=4） 90c8b1d
规律

上表中有 3 个 bug（advisory lock 泄漏、ENGRAM_EMBEDDER 发散、GC 误删 embedding）是跨边界集成缺陷——单元测试无法发现，只有代码审查 + 集成测试才能抓到。这是两阶段审查规程存在的核心理由。

4 · 测试隔离的故事

Postgres 依赖测试在 go test ./... 并发执行时，早期版本存在间歇性 flake。根因是多个 _test.go 共享同一张 refs 表，某个测试的 DELETE 或断言会与其他测试的写入产生竞争。

4.1 方案 A：per-test 唯一 agent id + scoped DELETE

对于普通的 refs / MemStore 测试，修复方案（commit d3f4bba，merge 86f71dd）是：

  • 每个测试生成唯一 agent id（如 t.Name() 哈希 + 随机后缀），杜绝两个并发测试操作同一行。
  • 测试结束后只 DELETE 自己插入的行（WHERE agent_id = $1），而不是 TRUNCATE 整张表。
  • 移除所有全局 TRUNCATE：全局清空在并发测试下会把其他测试正在使用的数据一并删除。

4.2 方案 B：全局队列操作的 per-test schema 隔离

对于 ClaimJob、DrainJobs 这类不带 agent_id 过滤的全局扫描，仅靠 scoped DELETE 不够——扫描会看到其他测试插入的 pending jobs，破坏"claimed exactly N jobs"的断言。修复方案是 per-test pgx search_path schema 隔离：

  • 每个测试在事务内执行 SET search_path TO test_<unique_name>，所有 DDL 和 DML 都在独立 schema 下运行。
  • 全局扫描（SELECT ... FOR UPDATE SKIP LOCKED）只能看到同 schema 内的行——不同测试的数据完全不可见。
  • 测试完成后 DROP SCHEMA CASCADE 清理，无残留。
成果

隔离改造后，ENGRAM_TEST_DB="..." go test -race ./... 连续多次运行零 flake。29 个测试文件、跨 10 个包并发执行，无需 -p 1 串行化。

4b · 守则（Guardrails）

CLAUDE.md 列出了四条不得违反的设计约束，每条都有工程理由。

守则工程理由如果违反会怎样
不加 Temporal / Kafka 维护任务幂等、短小（秒级）、可重派；Postgres 同事务入队已消除 outbox 双写问题；reconcile loop 足够；River 是现成 drop-in（M5 计划） 引入 Temporal/Kafka 需独立运维，比 Postgres 更重，在 Agent 记忆规模下净收益为负
不可变对象 内容寻址 = 哈希 → 内容的映射永远不变 = 缓存永不失效（L3 LRU 无需 invalidation）；GC 靠可达性判断；append-only 写入崩溃安全 任何 mutation 破坏缓存正确性，引入 invalidation 复杂度，GC 无法做简单标记-清除
并发控制只有 ref CAS 一个点 单一序列化点使推理简单；所有写者在同一个原子操作竞争，输者重试，无两阶段提交、无分布式锁协议 第二个并发控制点引入分布式事务，概率指数级增长的"两者都持有、等对方"死锁
system/ 保持精小；缓存/索引/工作树是派生视图 system/ 每轮推理必须塞进 LLM 上下文，字节直接消耗 token；派生视图可随时重建，Pod 崩溃不丢数据，真相只在 objstore + Postgres system/ 膨胀导致每轮 token 爆炸；把缓存当主存会在 Pod 重启时丢数据

5.3 唯一延期的功能项

Embedding store 的驱逐（eviction）逻辑被有意推迟——不是技术问题，而是优先级决策：embedding 是内容寻址的派生视图，可从 HEAD 全量重建（maintenance.Reindex），且在 Agent 记忆规模下体积很小。实现 LRU 驱逐会增加实现复杂度，在需求出现前不值得。
