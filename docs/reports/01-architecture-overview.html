① 架构总览

Engram 是一个云端、多租户 Agent 记忆系统——用 Git 对象模型管理 Markdown 记忆，用 Postgres CAS 保证一致性，用无状态工作节点提供可伸缩性。

1. 我们在建什么

Engram 将 MemFS（一个 Git 支撑的本地记忆文件系统，源自 Letta 范式）搬到了服务器端，使得多个 Agent 各自拥有一个版本化的记忆仓库，由无状态工作节点按需服务。Agent 可以读取常驻记忆（system/）、在推理过程中按需召回长尾内容、并跨会话持久化编辑。

整体设计刻意保持轻量：无状态计算 + 内容寻址对象存储 + 单个 Postgres 作为协调枢纽，故意不引入 Temporal 或 Kafka（原因见§5）。

核心理念

记忆是主动式（agentic）的，而非 RAG 式的。推理前只加载常驻的 system/ 目录 + 轻量树索引（文件名 + description 字段）。长尾内容由模型在循环中途通过 recall / read 工具按需拉取，不会在推理前做自动 top-k 倾倒。这一原则显著降低了每轮 Token 消耗，同时保留了完整的记忆可达性。

2. 四个区（Four Zones）

架构在逻辑上分为四个区（docs/architecture.md §3）。读路径自上而下流经加速器层；写路径（commit）直达规范存储层；维护工作完全异步，不在关键路径上。

① 请求路径

Session Router → Agent Worker → LLM

Router 按 agent_id 串行化写操作（粘性路由或逐 Agent 锁）。Worker 是无状态 Pod：挂载工作树、组装上下文、运行工具、提交。LLM 纯推理，无状态。

更多细节见 ② 请求路径。

② 读加速层

SHA 缓存 + Trigram 搜索

Memory Cache 以 commit SHA（准确说是 system/ 子树哈希）为键，无需失效机制。Search Service 本地持有完整文件内容（Variant A），查询时零对象存储访问。

更多细节见 ④ 搜索与索引。

③ 规范存储层

Object Storage + Postgres

对象存储保存内容寻址的不可变对象（objects/<hash>）。Postgres 是唯一的强一致性协调点：持有 agent_id→HEAD CAS ref 和 memory_jobs 任务队列。

更多细节见 ③ 存储与一致性。

④ 维护层

单个 Go 维护 Worker

独立进程，周期性运行：GC（标记清除不可达对象）、Reflection（历史整合）、Defrag（拆分过大文件）、Reindex（增量触发）、Reaper（回收停滞任务）。

更多细节见 ⑤ 维护管线。

3. 三条核心不变量

以下三条不变量是整个架构的脊梁（docs/architecture.md §2，CLAUDE.md 核心不变量）。每一条都派生出下游的多项设计决策。

不变量 1 — 对象不可变（Objects are immutable）

文件内容、树对象、提交对象全部以其自身哈希作为内容寻址键（SHA）。永远不要修改对象，只追加新对象。修改记忆 = 产生新的对象 + 一次新的提交。

为什么这能派生出其他机制：

  • 按 SHA 缓存：sha → materialized content 是不可变映射，缓存永远不需要失效——"失效"退化为 LRU 淘汰以节省空间（§4.4）。
  • 跨版本去重：同一内容写入多次只存储一份对象，对象存储天然去重。
  • 标记清除 GC：哈希永远不会指向两份不同的内容，因此可以安全地从 HEAD 可达集之外标记"垃圾"，定期清除。
不变量 2 — Ref 可变且强一致（Refs mutable and strongly consistent）

系统中唯一可变的指针是 agent_id → HEAD，存储在 Postgres 中。所有并发控制、顺序保证、一致性都漏斗式地集中到这一个 CAS 操作上。

CAS 的 SQL 原语（docs/architecture.md §9）：

UPDATE agent_refs SET head=$new, updated_at=now()
WHERE agent_id=$id AND head=$expected_parent;

行数为 0 → HEAD 已经移动 → 调用方 reconcile（rebase/重试）。这一单点序列化取代了任何外部锁服务。对象存储不需要条件写，因为它只存不可变对象。

这与 lakeFS 的设计完全对应（§15.1 验证 3-0）：lakeFS 将已提交的元数据存储在对象存储中，但用 KV 存储中的 SetIf() CAS 保护分支指针，原因正是对象存储不具备条件写原语。

不变量 3 — 其他一切都可重建（Everything else is rebuildable）

缓存、trigram 索引、工作树副本都是从对象 + ref 推导的视图，因此是无状态的、可丢弃的。

  • 缓存：Pod 崩溃 → 重启后从对象存储重新填充，不丢任何数据。
  • 搜索索引：最终一致，刚写入的记忆可能几秒内不可搜——可接受，因为当前轮次的内容已在上下文中。
  • 工作树：临时目录，挂载到 worker scratch / tmpfs，可按 SHA 重新 materialize。
常见误区

不要将缓存、索引或工作树副本视为可信来源（source of truth）。它们是推导产物，是可丢弃的。对象 + ref 才是权威数据，这一点在 CLAUDE.md 禁止事项 中明确列出。

4. 数据流

以下序列图展示了从推理前组装、Agent 循环中途工具调用，到提交并触发维护的完整数据流（docs/architecture.md §8）。

sequenceDiagram
  participant Router as Session Router
  participant Worker as Agent Worker
  participant Cache as Memory Cache
  participant ObjStore as Object Storage
  participant PG as Postgres
  participant LLM as LLM Provider
  participant Maint as 维护 Worker

  Note over Router,Worker: 推理前组装 (pre-inference assemble)
  Router->>Worker: 按 agent_id 序列化，分配请求
  Worker->>PG: ResolveHead — 读取当前 HEAD
  Worker->>Cache: Get(agentID, sha) — 读 system/ + 树索引
  Cache-->>Worker: 命中 → 直接返回
  Cache->>ObjStore: 未命中 → 拉取对象
  ObjStore-->>Cache: 填充缓存
  Cache-->>Worker: 返回 Resident(system/ content, tree index)
  Worker->>LLM: 发送上下文(system/ + 树索引 + 用户消息)

  Note over Worker,LLM: Agent 循环 (mid-loop tools)
  LLM-->>Worker: 调用 recall 工具
  Worker->>Cache: Search.Recall(agentID, query)
  Cache-->>Worker: 返回匹配行范围
  LLM-->>Worker: 调用 read/edit 工具
  Worker->>ObjStore: 按需拉取或写入工作树文件

  Note over Worker,PG: 提交 (commit)
  Worker->>ObjStore: 写新内容寻址对象 (幂等 PUT，先写)
  Worker->>PG: 单事务: CAS HEAD + INSERT memory_jobs
  PG-->>Worker: 成功 (新 HEAD) 或 ErrCASConflict → 重试

  Note over Maint,PG: 异步维护 (async maintenance)
  Maint->>PG: 出队 memory_jobs (FOR UPDATE SKIP LOCKED)
  Maint->>ObjStore: GC 标记清除不可达对象
  Maint->>Cache: Reindex(agentID, from, to) — 增量更新索引
  Maint->>PG: 更新 maintenance_cursor / 任务状态

写顺序的关键：先对象，再 Ref

如 docs/architecture.md §6.2 所述：先将新对象写入对象存储（幂等），再在 Postgres 中执行 CAS。在 CAS 之前崩溃只留下不可达的垃圾对象（GC 会回收），不会产生指向不完整提交的悬空 ref。这是核心的崩溃安全保证。

5. 关键设计决策

决策 选择 放弃的备选方案 原因摘要
写并发 每个 agent 单写者 (Single-writer)，Router 串行化 乐观多写者 + 自动 merge Markdown prose 的自动 merge 有损。串行化简单、正确，且内存访问模式本就顺序化（§4.1）。
提交原子性 单个 Postgres 事务：CAS + 入队任务 Kafka outbox / 双写 原子性消除了"提交成功但任务未入队"的不一致。"任务入队 iff 提交已提交"（§8.2，§15.1 验证 3-0）。
写顺序 先写对象存储，再做 Ref CAS 先 CAS 再写对象 崩溃安全：CAS 之前崩溃只留垃圾对象（GC 回收），不留悬空 ref（§6.2）。
搜索变体 Variant A：索引存储文件内容 Variant B：索引存偏移量 + Range GET 查询时零对象存储访问；记忆是小型文本，内容冗余可忽略（§4.5）。L4 已实现混合 trigram + 嵌入向量。
任务队列 Postgres memory_jobs + reconcile loop Temporal / Kafka / River（计划 M5） 维护任务幂等、可从状态重推导、运行时间短——不需要耐久的中途 resume（§12）。SKIP LOCKED 使用得当可规避性能陷阱（§15.3）。
工作树 go-git + 自定义 Storer（不调用 git 二进制） shell out 到 git 服务中调用子进程不可靠，go-git 可在测试中替换为内存 Storer（CLAUDE.md 技术栈）。

为什么不用 Temporal 和 Kafka（§12）

Guardrail

不要添加 Temporal 或 Kafka。 Temporal 解决的是"长时运行、崩溃可恢复、多步骤工作流"的问题；我们的维护任务幂等、可从状态重推导、运行时间短，接受"从头重试"就不需要耐久的中途 resume。Kafka 解决的是"解耦写路径、扇出到多个消费者"的问题；我们只有一个消费者（维护 worker），Postgres 同一事务入队提供了解耦且无双写不一致。仅当任务变得真正长时复杂（→ Temporal）或消费者增多需要重放（→ Kafka）时再重新评估。

7. 技术栈

语言

Go

  • 所有 I/O 函数第一参数为 context.Context
  • 错误用 %w 包装（可追溯链）
  • 无全局可变状态
  • 包边界用小接口隔离
  • 测试用表驱动（table-driven tests）
数据库

Postgres + pgx

  • refs CAS（agent_refs）
  • 任务队列（memory_jobs，SKIP LOCKED）
  • 维护游标（maintenance_cursor）
  • 迁移：golang-migrate，扩展-收缩，从不破坏性单步
对象存储

S3 / 本地 FS

  • 始终隐藏在 ObjStore 接口后方
  • 生产用 S3/OSS；开发/测试用本地 FS 后端
  • 要求桶具备读后写强一致性（S3 自 2020 年起满足）
  • 嵌入向量 store 独立于 git 对象 store（GC 不扫描前者）
Git 层

go-git + 自定义 Storer

  • go-git 的 Storer 接口由 ObjStore 支撑
  • 服务内部不调用 git 二进制
  • 工作树 materialize 用 osfs；测试可换 memfs
  • 未来无本地磁盘路径：全内存 billy Filesystem（§10.1 提及，尚未实现）

核心接口速览（docs/architecture.md §11）

type MemStore interface {
    // ResolveHead 读取 agent 当前 HEAD（廉价 ref 读）
    ResolveHead(ctx context.Context, agentID string) (CommitHash, error)

    // Materialize 将 at 处的树检出到 dir（工作副本）
    Materialize(ctx context.Context, agentID string, at CommitHash, dir string) error

    // CommitWithCAS 写对象，构建提交，在同一 DB tx 中原子推进 HEAD 并入队任务
    // HEAD 已移动则返回 ErrCASConflict
    CommitWithCAS(ctx context.Context, agentID string, parent CommitHash,
        dir string, jobs []Job) (CommitHash, error)
}

type SystemCache interface {
    Get(ctx context.Context, agentID string, sha CommitHash) (*Resident, error)
}

type Search interface {
    Recall(ctx context.Context, agentID, query string, k int) ([]Hit, error)
    Reindex(ctx context.Context, agentID string, from, to CommitHash) error
}

8. 2026 最佳实践交叉验证摘要（§15）

一次深度、多源、对抗性验证的最佳实践审查（docs/architecture.md §15）对上述设计进行了交叉检验。结论摘要如下：

决策 验证结果 来源
不可变对象 + Postgres ref CAS 验证 3-0 与 lakeFS 设计完全对应（§15.1）
同一事务 CAS + 入队（无 Kafka/outbox） 验证 3-0 防止孤立任务和提交/发送竞争（§15.1）
纯词法 trigram 搜索（Variant A） A/B 候选 混合信号（trigram + 嵌入）在长尾召回质量上优于纯词法（§15.2）→ L4 已实现混合
Postgres SKIP LOCKED 任务队列 陷阱需规避 ORDER BY + 高并发可超线性退化；单写者天然低并发（§15.3）
Git 对象模型（小型 prose Markdown） 当前安全 大文件/深历史场景有风险，当前用例不受影响（§15.4）
小结

Engram 的三条核心不变量不仅在内部逻辑自洽，也被独立的 2026 行业实践所佐证：不可变内容寻址对象 + 强一致 CAS ref 是现代版本化数据湖（lakeFS 等）的共识方案；同一 Postgres 事务入队是消除双写不一致的最简路径。架构当前最大的可优化方向是搜索召回质量（L4 混合搜索已推进）和任务队列在更高并发下的健壮性（M5 River 升级候选）。
