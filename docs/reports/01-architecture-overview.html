架构总览

Engram 是给 agent 用的"云端笔记本"——用 git 对象模型存记忆，用 Postgres 的一把小锁保证一致性，用无状态工作节点做到横向扩展。这一页是全局地图；子系统细节见 请求路径、存储与一致性、搜索与索引、维护管线。

1. 我们在建什么

大模型每次对话都从空白开始——它不知道上次你说了什么，不记得上次它写了什么。Engram 要解决的就是这件事：给 agent 一个可以跨会话读写、版本化、多租户隔离的记忆仓库，部署在云端，由无状态工作节点按需服务。

出发点是 MemFS（Letta 项目里的 git 支撑本地记忆文件系统），但把它从"单机本地"搬到了"多租户云端"。每个 agent 各自拥有一个 git 仓库——文件就是 Markdown 记忆片段，历史就是 git commit，内容寻址就是 git 对象的哈希。云端的挑战是：多个 agent 并发写、pod 随时崩溃、worker 无状态、不能有单点瓶颈。

核心理念 — 记忆是主动去取，不是塞给模型

每轮推理前，只把 system/（常驻要点）和轻量树索引（文件名 + description 字段）加载进上下文。长尾内容——那些不常用但存在的记忆——由模型在推理循环中途通过 recall / read 工具按需拉取。不做自动 top-k 倾倒，不事先把"可能有用"的内容全部塞进 prompt。这一原则显著压低每轮 Token 消耗，同时保留了完整的记忆可达性。

整体设计刻意保持轻量：无状态计算 + 内容寻址对象存储 + 单个 Postgres 作为协调枢纽，不引入 Temporal 或 Kafka（原因见§5）。最简基础设施：一个对象存储桶 + Postgres + 无本地持久化的 pod。

2. 四个区

架构在逻辑上分为四个区（architecture.md §3）。读路径从上往下流经加速层；写路径（commit）直达规范存储；维护完全异步，不在关键路径上。

四区拓扑图。红色实线 = 写路径（先对象后 CAS）；绿色虚线 = 读路径（优先走缓存/索引）；橙色虚线 = 异步维护。

① 请求路径

Session Router → Agent Worker → LLM

Router 按 agent_id 串行化写操作（粘性路由或逐 Agent 锁）。Worker 是无状态 Pod：挂载工作树、组装上下文、运行工具循环、提交。LLM 纯推理，无状态。

→ 请求路径 深入解析。

② 读加速层

SHA 缓存 + 混合搜索

Memory Cache 以 system/ 子树哈希为键，无需失效（同一 SHA 永远映射同一内容）。Search Service 本地持有完整文件内容（Variant A），查询时零对象存储访问。

→ 搜索与索引 深入解析。

③ 规范存储层

Object Storage + Postgres

对象存储保存内容寻址的不可变对象（objects/<hash>）。Postgres 是系统唯一的强一致性协调点：持有 agent_id→HEAD CAS ref 和 memory_jobs 任务队列。

→ 存储与一致性 深入解析。

④ 维护层

单个 Go 维护 Worker

独立进程，周期性运行：GC（标记清除不可达对象）、Reflection（历史整合）、Defrag（拆分过大文件）、Reindex（增量更新索引）、Reaper（回收停滞任务）。完全异步，不阻塞前台。

→ 维护管线 深入解析。

3. 三条贯穿全局的规矩

下面三条不变量是整个架构的脊梁（architecture.md §2，CLAUDE.md 核心不变量）。它们彼此派生：第一条让缓存和 GC 成为可能，第二条让全系统有唯一的裁决点，第三条让你可以随时丢掉、重建一切派生物。

不变量 ① — 每块内容不可变，按指纹存

文件内容、树对象、提交对象——全部以自身内容的哈希（SHA）作为键，存在对象存储里。永远不修改对象，只追加新对象。修改记忆 = 产生新的对象 + 一次新的提交。

这条规矩派生出三件事：

  • 缓存永远不用失效。sha → materialized content 是不变映射，LRU 淘汰是为了省空间，不是为了保鲜。新内容 = 新 SHA = 新键 = 缓存未命中 = 重新填充，这是"天然失效"。
  • 跨版本去重。同一段内容写入两次只存一份对象，对象存储层天然去重。
  • 标记清除 GC 是安全的。同一个哈希不可能指向两段不同内容，因此"从 HEAD 可达集以外的对象"就是垃圾，定期清除，不会误删任何活跃对象。
不变量 ② — 唯一会变的是 HEAD 指针，用一把"改前先确认"的小锁守

系统里唯一可变的指针是 agent_id → HEAD，存在 Postgres 的 agent_refs 表。所有并发控制、顺序保证、一致性全都漏斗式地集中到这一个 CAS 操作上——这是全系统唯一的裁决点。

SQL 原语（architecture.md §9）：

UPDATE agent_refs SET head=$new, updated_at=now()
WHERE agent_id=$id AND head=$expected_parent;

返回 0 行 → HEAD 已经移动 → 调用方 reconcile 并重试。这一单点序列化取代了任何外部锁服务。对象存储不需要条件写，因为它只存不可变对象。

这与 lakeFS 的设计完全对应（architecture.md §15.1，验证 3-0）：lakeFS 用对象存储保存已提交元数据，用 KV 存储的 SetIf() CAS 保护分支指针，原因正是对象存储没有条件写原语。

不变量 ③ — 其余都是可丢弃、可重算的派生物

缓存、trigram 索引、嵌入向量索引、工作树副本——都是从对象 + ref 推导的视图，因此是无状态的、可丢弃的。

  • 缓存：Pod 崩溃 → 重启后从对象存储重新填充，零数据丢失。
  • 搜索索引：最终一致，刚写入的记忆可能数秒内不可搜——可接受，因为当前轮次的内容已在上下文中。
  • 工作树：临时目录，挂载在 worker scratch / tmpfs，可按 SHA 随时重新 materialize。
常见陷阱

不要把缓存、索引或工作树视为可信来源。它们是推导产物，是可丢弃的。对象 + ref 才是权威数据（CLAUDE.md 禁止事项）。一旦把派生物当成 source of truth，你就会开始担心"缓存和存储不一致怎么办"——而正确答案是：派生物不一致时，丢掉重建就好。

4. 一次请求到一次提交

从用户消息进来，到 agent 编辑记忆、提交、触发后台维护——完整生命周期走查（architecture.md §8）。

  1. 装载常驻要点 + 目录 Router 按 agent_id 序列化，将请求交给 Worker。Worker 从 Postgres 读当前 HEAD，向 Memory Cache 查 Get(agentID, systemTreeSHA)。命中 → 直接返回已物化的 system/ 内容 + 树索引（文件名、description）；未命中 → 从对象存储拉取对象、填充缓存。上下文组装：常驻要点 + 树索引 + 用户消息，发给 LLM。
  2. Agent 循环 — 模型自己决定去取什么 LLM 收到上下文后进入工具调用循环。它可以调用 recall(query)（向 Search Service 发 trigram+向量混合查询，返回匹配的行号范围，不是整个文件）、read(path)（从工作树或对象存储读一个文件）、edit(path, content)（在工作树上写入修改）。这一切都发生在推理过程中，模型自主决定拉取哪些记忆——不是预先顶满上下文。
  3. 提交 — 先写对象，再改 HEAD，同一事务入队任务 Worker 调用 CommitWithCAS：①把工作树里所有新内容写成内容寻址对象，幂等 PUT 到对象存储（这步可以崩溃，只留垃圾对象，GC 会清）；②在单个 Postgres 事务里执行 CAS HEAD: parent → new + INSERT memory_jobs。原子性保证：任务入队 当且仅当 提交已提交。CAS 失败（HEAD 已移动）→ 返回 ErrCASConflict → 调用方 reconcile 并重试。
  4. 后台慢慢消化 — 异步维护，不阻塞前台 维护 Worker 用 FOR UPDATE SKIP LOCKED 出队 memory_jobs，在独立 git worktree 里运行 Reflection（历史整合）、Defrag（拆分过大文件）、GC（标记清除不可达对象）、Reindex（增量更新搜索索引）。全程不干扰前台 Worker 正在读取的 SHA 快照。任务超时被 Reaper 回收，毒任务（反复崩溃 worker 的任务）在 maxAttempts 次后进入 failed 状态（L5d）。

写顺序的崩溃安全保证

architecture.md §6.2 明确要求：先写对象到对象存储（幂等），再在 Postgres 中执行 CAS。在 CAS 之前崩溃，只留下不可达的垃圾对象，GC 会在宽限窗口后清理它们；不会产生指向不完整提交的悬空 ref。反过来如果先 CAS 再写对象，崩溃会留下一个 HEAD 指向尚不存在的对象——这是严重的一致性缺陷。

5. 几个关键取舍

单写者 vs 乐观多写者

选择：每个 agent 单写者，Router 串行化

Router 通过粘性路由或逐 Agent 锁，保证同一时刻只有一个 Worker 对某个 agent 执行写操作。简单、正确，且 agent 的记忆访问模式本就是顺序的（一问一答）。

放弃：乐观多写者 + 自动 merge

对结构化数据（JSON、代码）三路 merge 可行；对 Markdown prose 不可行——任何自动 merge 都有损，可能把两段不相关的记忆拼错位置，或静默地覆盖内容。

提交即一次数据库事务，替代重型消息中间件

选择：单个 Postgres 事务 — CAS + 入队任务

CAS 和 INSERT memory_jobs 在同一事务里提交，原子性保证"任务入队 iff 提交已提交"。没有孤儿任务，没有提交/发送竞争（commit/emit race）。

放弃：Kafka outbox / 双写

先写数据库再写 Kafka：如果 Kafka 写失败，任务丢失；如果先写 Kafka，DB 写失败，任务已入队但提交没发生——两种顺序都有不一致窗口。需要复杂的 outbox 表来弥补。

搜索本地化：查询不碰对象存储

选择：Variant A — 索引本地存储文件全文内容

查询时零对象存储访问，延迟低、无依赖。记忆文件是小型 Markdown，内容冗余可忽略。L4 已在此基础上叠加嵌入向量，实现 trigram 词法 + 语义混合召回。

Variant B — 索引只存偏移量 + Range GET

查询时需对对象存储发起 Range GET，增加跨可用区网络往返，且在高并发下可能打穿对象存储限速配额。当记忆文件很大时才值得考虑，当前用例不适用。

为什么不用 Temporal 和 Kafka（architecture.md §12）

Guardrail — 禁止引入 Temporal 或 Kafka

Temporal 解决的是"长时运行、崩溃可恢复、多步骤工作流"的问题。我们的维护任务幂等、可从状态重推导、秒级完成，接受"从头重试"就不需要耐久的中途 resume——Temporal 的成本完全是浪费。

Kafka 解决的是"解耦写路径、扇出到多个消费者"的问题。我们只有一个消费者（维护 Worker），Postgres 同一事务入队提供了解耦且没有双写不一致。

仅当任务变得真正长时复杂（→ 才需要 Temporal）或消费者增多需要重放（→ 才需要 Kafka）时再重新评估。M5 计划升级到 River（同 Postgres，更丰富的重试语义），但不需要跨越到 Kafka。

6. 九层地图（L1–L5d）

项目按层（Layer）顺序构建，每层在前一层基础上叠加一组保证。截至 main 分支，L1–L5d 全部合并完成（CLAUDE.md Build/test/run 状态）。

层 加了什么 核心保证 深度报告
L1 MemStore 核心：本地 FS 对象后端 + Postgres refs + CAS；CommitWithCAS；单写者路由 agent 可读/编辑/提交记忆，历史版本化、可 diff 存储与一致性
L2 Agent 循环 + Session；工具集（recall / read / edit）；上下文组装；LLM fake/anthropic 后端 完整的推理-编辑-提交循环端到端可运行 请求路径
L3 SHA 键控 LRU 读缓存；assembleSystem 缓存支持；Router/cmd 注入共享缓存实例 system/ 内容跨轮次命中缓存，不重复读对象存储；缓存永远不需要失效 请求路径
L4 混合搜索（trigram 词法 + 嵌入向量语义）；Recall 工具返回行范围；Search Variant A（本地内容） 查询时零对象存储访问；混合信号优于纯词法（§15.2 验证 3-0） 搜索与索引
L4b 嵌入向量索引持久化（独立于 GC 的 embedding store）；增量 reindex（git diff 驱动） 嵌入向量在 worker 重启后幸存；reindex 只处理变更差量 搜索与索引
L5a GC Worker：标记清除不可达对象；grace 窗口防误删新写入对象 垃圾对象（CAS 前崩溃残留）被定期清除；对象存储不无限增长 维护管线
L5b memory_jobs 消费者；Reflection（历史整合）；reconcile loop + 轮询 Postgres 每轮维护：GC → 出队 memory_jobs（reflection + reindex）；任务按 agent 去重 维护管线
L5c 确定性 Defrag：按顶级标题拆分过大 Markdown 文件（ENGRAM_DEFRAG_MAX_BYTES） 收敛性：反复运行产生相同结果；system/ 文件不无限膨胀 维护管线
L5d 停滞任务 Reaper：claimed_at 字段迁移；ReapStaleJobs 将超时 running 任务回收为 pending；毒任务防护（maxAttempts） Worker 崩溃不会永久阻塞任务；超过重试上限的毒任务进入 failed 状态（§14 已解决） 维护管线

7. 技术栈

语言

Go

  • 所有 I/O 函数第一参数为 context.Context
  • 错误用 %w 包装，调用链完整可追溯
  • 无全局可变状态；包边界用小接口隔离
  • 测试用表驱动（table-driven），Postgres 依赖测试在 ENGRAM_TEST_DB 未设置时自动 t.Skip
数据库

Postgres + pgx

  • agent_refs：refs CAS，唯一强一致性协调点
  • memory_jobs：任务队列，FOR UPDATE SKIP LOCKED 出队，按 agent+kind 去重
  • maintenance_cursor：reconcile loop 游标
  • 迁移：golang-migrate，扩展-收缩策略，单步从不破坏性
对象存储

S3 / 本地 FS

  • 始终隐藏在 ObjStore 接口后方；上层代码无感知替换
  • 生产用 S3/OSS；开发/测试用本地 FS 后端（ENGRAM_OBJ）
  • 要求桶具备读后写强一致性（S3 自 2020 年起满足）
  • Embedding store 独立于 git 对象 store，GC 不扫描前者
Git 层

go-git + 自定义 Storer

  • go-git 的 Storer 接口由 ObjStore 实现支撑
  • 服务内部不调用 git 二进制——shell out 在服务中不可靠且难以测试
  • 工作树 materialize 用 osfs；测试可换 memfs，无需磁盘
  • 未来路径：全内存 billy Filesystem，让 worker 无本地磁盘依赖（§10.1，尚未实现）

核心接口速览（architecture.md §11）

type MemStore interface {
    // ResolveHead 读取 agent 当前 HEAD（廉价 ref 读，不触碰对象存储）
    ResolveHead(ctx context.Context, agentID string) (CommitHash, error)

    // Materialize 将 at 处的树检出到 dir（可丢弃的工作副本）
    Materialize(ctx context.Context, agentID string, at CommitHash, dir string) error

    // CommitWithCAS 写对象，构建提交，在同一 DB tx 中原子推进 HEAD 并入队任务
    // HEAD 已移动则返回 ErrCASConflict
    CommitWithCAS(ctx context.Context, agentID string, parent CommitHash,
        dir string, jobs []Job) (CommitHash, error)
}

type SystemCache interface {
    Get(ctx context.Context, agentID string, sha CommitHash) (*Resident, error)
}

type Search interface {
    Recall(ctx context.Context, agentID, query string, k int) ([]Hit, error)
    Reindex(ctx context.Context, agentID string, from, to CommitHash) error
}
