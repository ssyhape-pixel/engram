Build Journey

How Engram grew from a first scaffold commit to 10 packages, 32 source files, 29 test files, and 2 runnable binaries — and the engineering discipline that drove every layer.

Based on main branch L1–L5d · 113 commits · 2026-06

1 · Building layer by layer

The architecture doc targeted M5 (scale) from day one, but the implementation didn't get there in a single push. Each layer had to land on main with a clean build and a fully green test suite before the next one could start. This isn't just "discipline for its own sake" — every merge is a genuinely shippable increment: no long-lived feature branches, no "we'll test it once everything's integrated" debt.

1.2 Layer dependency graph

Layer dependency graph. L3 and L4/L4b are two independent optimization tracks (read acceleration vs. search) that converge at L5b's DrainJobs: the Reindex job requires both the persistent embedding cache (L4b) and the job-consumer infrastructure (L5b). L5c → L5d is a linear wrap-up.

Core meta-decision

After every layer merges to main, go build ./... and a full go test ./... (including DB-backed tests, requires ENGRAM_TEST_DB) must be completely green before the next layer begins. No long-lived feature branches. No "test it once it's all integrated."

2 · The process every layer goes through

Every layer follows the same five phases, in order, without skipping or shuffling steps. This isn't ceremony — each phase produces a concrete artifact (a file, a test, a commit), and the artifact has to exist before the phase counts as done.

  1. Brainstorming Clarify use-case boundaries, interface shapes, and how the new layer relates to existing packages. Artifact: a Spec document under docs/superpowers/specs/. Every section must include an SVG or Mermaid diagram — prose-plus-code-block alone is not acceptable. The spec must be "readable offline" without depending on any external rendering service.
  2. Writing a plan Once the spec is confirmed, break the implementation into bite-sized TDD tasks and store them in docs/superpowers/plans/. Each task's acceptance criterion must be "running that task's tests in isolation passes green" — "we'll test it together with the next task" is not allowed. Target task granularity: 30–90 minutes.
  3. Subagent-driven execution Each task is dispatched to a fresh subagent with no prior session context to bleed through. TDD order is strict: write the test first (watch it fail), then write the implementation, then watch the test go green. Each task maps to exactly one feat/fix/refactor/test commit. Independent tasks can be dispatched concurrently to separate subagents.
  4. Two-phase review (spec compliance + code quality) Phase 1: compliance check against the spec — does the implementation cover every design point, and do the interfaces match the diagrams? Phase 2: code-quality check — error propagation with %w, interface size, test coverage, edge cases. In practice these are usually combined into a single final review that produces a batch of fix/refactor commits before the merge.
  5. Local merge + branch deletion git merge --no-ff preserves the layered history (each layer boundary is visible on the mainline), then the feature branch is deleted — no dangling refs. GitHub PR merges are deliberately avoided to prevent auto-squash from erasing commit granularity.

2.1 Coding rules (five hard constraints)

These five rules are spelled out explicitly in CLAUDE.md and checked item-by-item during the review phase:

RuleWhat it requiresWhy
Tests first Write the test for a new interface, watch it fail, then write the implementation Writing tests first forces you to nail down the interface contract before you code; tests written after tend to cover only the paths that already work
%w error wrapping All fmt.Errorf calls must use %w, never %s Callers can use errors.Is/errors.As to match precisely; %s turns the error into a string and destroys reachability
context.Context as first argument Every I/O function takes ctx context.Context as its first parameter Timeout cancellation must be able to propagate; functions without a context almost always require a breaking interface change when they're wired up later
Small interfaces Package boundaries expose only the minimal interface; internal implementation details do not leak Small interfaces are easy to mock, easy to test, and easy to swap out; large interfaces introduce tight coupling across packages
Table-driven tests Multi-scenario coverage uses []struct{name, input, want} — no copy-pasted test functions Adding a new test case is one extra struct literal; you don't have to duplicate the entire test function body

3 · Bugs the review chain actually caught

The table below lists real defects captured during review or testing, each traceable to a fix commit. Every one of them was found and fixed before the merge — not in production.

Symptom Root cause Fix Commit
Connection pool lock leak WithGlobalLock's defer unlock used the caller-supplied ctx — if that context was already cancelled, the unlock SQL couldn't execute, leaving the advisory lock held permanently; when the connection was returned to the pool it still carried the lock, and the next borrower inherited it Changed unlock to use context.Background(), fully decoupled from the caller's context lifetime 6f8c410
Reflection self-triggering loop After Reflect committed a new commit, that commit enqueued a new reflect job in the same transaction; the new job triggered Reflect again, infinitely Reflect's commit call passes jobs=nil — maintenance commits must never produce new maintenance jobs 9db0762
Reflection writing back empty/partial content When reading a resident file failed, Reflect silently swallowed the error and continued, committing empty content as "consolidated"; or if the resident file was already empty, it still made an LLM call and committed the empty result, corrupting memory Read errors now return immediately (not skipped); if resident content is empty, return early without making an LLM call d601f50
System cache caching partial content buildSystemContent's original signature returned no error; if reading a system/ file failed mid-way, the already-read portion was written into the LRU cache as "complete content" Signature changed to (string, error); the cache is only written when error == nil 63ad4cd
Two env vars selecting different embedding models cmd/api used ENGRAM_PROVIDER to select the embedder; cmd/maintenance used ENGRAM_EMBEDDER — both processes shared one ObjCache embedding store but used different models, causing vector-dimension conflicts and wasted embeddings cmd/maintenance unified to use ENGRAM_EMBEDDER; convention: both ends must use the same model when sharing the ENGRAM_EMB_OBJ path 3a37820
Wrong convergence rationale in defrag comment A code comment said defrag decided whether to split based on "file size" (size-based); the actual isSplittable checks for multiple top-level headings — size-based logic cannot guarantee convergence (a small file with multiple headings would trigger repeatedly) Comment corrected to describe structural convergence; confirmed isSplittable only returns true when there are multiple top-level headings (after splitting, each file has one — no further trigger) 9d1688c
GC mistakenly sweeping the embedding store The global mark-and-sweep GC walks the objstore; the embedding cache's ObjCache and the main objstore can be configured with different root paths, but the GC lacked root-path discrimination and could identify embedding objects as unreachable orphans and delete them Added a distinct-root guard: shared storage paths are only allowed when filepath.Clean(ENGRAM_EMB_OBJ) != filepath.Clean(ENGRAM_OBJ); cmd/maintenance uses filepath.Clean for the comparison 25286b8
Concurrent commits silently overwriting each other Session's early CAS-failure path directly overwrote HEAD with the local view instead of returning ErrConflict — a concurrent writer's commit was silently discarded CAS failure now explicitly returns ErrConflict; Session catches it and retries up to a bounded number of times before surfacing the error 1d58581
Read tool off-by-one: 0-based vs 1-based line numbers The recall tool returns 1-based line numbers (consistent with git diff / grep), but the read tool used 0-based offsets internally — the model passed recall's line numbers directly to read, landing one line off Read tool changed to 1-based, consistent with recall and the ToolDef docs; test cases updated accordingly (start=5 rather than start=4) 90c8b1d
Pattern

Of the 9 bugs above, 3 of them (the advisory lock leak, the ENGRAM_EMBEDDER divergence, and the GC deleting embeddings) are cross-boundary integration defects — unit tests cover the behavior of a single function and cannot see the contract between two independent processes or two independent storage paths. This is direct evidence for why the two-phase review (spec compliance + code quality) cannot be skipped.

4 · How tests stay out of each other's way

Postgres-dependent tests run concurrently under go test ./.... In early versions this produced intermittent flakes. The root cause: multiple _test.go files shared the same refs table, so one test's DELETE or assertion would race with another test's writes.

The fix uses two strategies, each targeting a different class of operation:

4.1 Ordinary CRUD tests: per-test agent ID + scoped DELETE

For ordinary refs / MemStore tests (commit d3f4bba, merge 86f71dd):

  • Each test generates a unique agent ID (t.Name() hash + random suffix), so no two concurrent tests ever touch the same row.
  • Cleanup DELETEs only the rows the test itself inserted (WHERE agent_id = $1) — never a TRUNCATE of the whole table.
  • All global TRUNCATE calls removed: truncating globally under concurrent tests wipes out rows another test is actively using.

4.2 Global-scan operations: per-test schema isolation

For operations like ClaimJob and DrainJobs that scan globally without an agent_id filter, scoped DELETE isn't enough. The scan sees pending jobs inserted by other tests and breaks "exactly N jobs claimed" assertions. The fix is per-test schema isolation:

  • Each test runs SET search_path TO test_<unique_name> inside a transaction, so all DDL and DML execute in an isolated schema.
  • Global scans (SELECT ... FOR UPDATE SKIP LOCKED) can only see rows in the same schema — data from other tests is completely invisible.
  • After the test completes, DROP SCHEMA CASCADE cleans up with no residue.

Left: old approach — multiple tests write to the same table and global scans interfere with each other. Right: new approach — tests that perform global operations each get their own schema; scans only see that test's own data.

Result

After the isolation retrofit, ENGRAM_TEST_DB="..." go test -race ./... runs zero flakes across multiple consecutive runs. 29 test files across 10 packages execute concurrently — no -p 1 serialization needed.

5 · Hard constraints (guardrails)

CLAUDE.md lists four design constraints that must never be violated. Each has a clear engineering rationale — these are not matters of taste.

ConstraintEngineering rationaleCost of violation
No Temporal / Kafka Maintenance jobs are idempotent, short (second-scale), and re-dispatchable; Postgres transactional enqueue already eliminates the outbox dual-write problem; the reconcile loop is sufficient; River is a drop-in for M5 Temporal/Kafka require independent ops overhead and independent health monitoring; net benefit is negative at agent-memory scale
Objects are immutable Content-addressing means a hash always maps to the same content — the L3 LRU cache never needs invalidation; GC works by reachability; append-only writes are crash-safe Any mutation breaks cache correctness, introduces invalidation complexity, and makes mark-and-sweep GC impossible
Concurrency control lives only at the ref CAS A single serialization point keeps reasoning simple; all writers compete on one atomic operation, losers retry, no two-phase commit, no distributed lock protocol A second concurrency control point introduces distributed transactions; deadlock probability grows exponentially
system/ stays small; cache/index/working tree are derived views system/ is loaded into the LLM context every turn — bytes directly consume tokens; derived views can be rebuilt at any time, so a pod crash loses nothing system/ bloat causes token explosion every round; treating the cache as primary storage means data loss on pod restart

6.3 The one intentionally deferred feature

The embedding store's eviction logic was intentionally deferred — not a technical gap, but a priority call:

  • Embeddings are a content-addressed derived view and can be fully rebuilt from HEAD via maintenance.Reindex.
  • At agent-memory scale, each agent's embedding store is small (Markdown text, a bounded number of embeddings).
  • Implementing LRU eviction would add complexity to ObjCache (access-time tracking, eviction policy design) that isn't worth investing in before there's a concrete need.
§14 open-question tracking

Of the 5 open questions listed in architecture.md §14, the stale-running job reaper was fully resolved in L5d (commit 950afe2, documentation closed at aa6a1a5). The remaining 4 — index eventual-consistency window, cross-tenant dedup vs. isolation, reflection trigger policy, warm sidecar threshold — remain open, waiting for scale-driven requirements.

6.4 §15 research A/B candidates

Deep research cross-validation in 2026 confirmed most of the design decisions. Two real optimization goals have already been implemented in the build, aligned with the research recommendations:

  • Hybrid recall (§15.2): L4 already implements trigram + semantic RRF fusion.
  • Queue hygiene (§15.3): L5b's SKIP LOCKED + no ORDER BY, plus L5d's stale reaper, correspond to the "operational leverage" recommendations; River is kept as an M5 drop-in.

Storage layer alternatives (§15.4 Prolly Trees / lakeFS SSTable) and structured memory (§15.5 Zep/Graphiti) are preserved as A/B candidates; current scale does not require them.
