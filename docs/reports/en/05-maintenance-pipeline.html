Maintenance Pipeline

A background Go worker that runs entirely outside the request path: GC, job queue draining, reflection consolidation, defragmentation, and stale-job reaping. Everything is idempotent and re-derivable — no Temporal or Kafka needed.

1. What the background worker does

cmd/maintenance/main.go is the sole maintenance process. It never runs on the foreground request path — agent session commits, LLM calls, and tool executions do not wait on it. Everything the worker does satisfies the same precondition:

Idempotent, fully re-derivable from stored state, and short-lived per execution. A crash means restart-and-redo — no irreproducible intermediate state, no heavyweight workflow engine needed.

Four tasks run in a fixed sequence to form one "round":

  • GC (mark-sweep): compute the set of objects reachable from all agent HEADs; delete unreachable content-addressed objects that are past the grace period (L5a).
  • ReapStaleJobs: move jobs that were left permanently in running due to a worker crash back to pending (L5d).
  • EnqueueDefrag: scan all agents and idempotently enqueue defrag jobs for agents with splittable .md files (L5c).
  • DrainJobs: consume pending jobs one by one, dispatching them to the reflect / reindex / defrag handlers (L5b / L5c).

The entire round runs under a Postgres session-level advisory lock (gcLockKey = 1). If the lock cannot be acquired the round is skipped, so multiple worker instances automatically degrade to active/standby when scaled horizontally:

Invariant The maintenance worker is completely decoupled from the foreground request path and never blocks an agent session. All writes go through the same CommitWithCAS interface used by the foreground — the system has exactly one consistency control point.

2. A job's lifecycle (state machine)

memory_jobs is the Postgres job queue that drives the entire maintenance pipeline. Key columns:

ColumnTypeMeaning
idbigserial PKGlobally unique job ID
agent_idtextOwning agent
kindtextreflect / reindex / defrag
from_shatext nullableHEAD commit SHA at enqueue time (for future diff-since use in reflect)
statetextpending / running / failed
attemptsint default 0Number of attempts so far; transitions to failed at maxAttempts
claimed_attimestamptz nullableTimestamp of the most recent claim; used by the Reaper to detect stuck jobs

Jobs are enqueued from two sources:

  • In-transaction enqueue (commit path): CommitRef advances HEAD and inserts reflect/reindex jobs inside the same Postgres transaction — a job exists if and only if the commit succeeded (replaces the Kafka + outbox pattern).
  • Out-of-band enqueue (defrag scan): EnqueueJob uses a partial-unique index (agent_id, kind) WHERE state='pending' with ON CONFLICT DO NOTHING, so there is at most one pending row per (agent, kind) pair.

memory_jobs state machine. done means the row is deleted immediately — no tombstone; failed rows are retained for inspection.

Design highlight Completing a job (done) deletes the row outright — the queue only holds active work, so historical throughput has no effect on table size. failed rows are kept for audit and are never picked up again by Drain or Reaper.

3. Claiming jobs without races

ClaimJob (refs/refs.go:122) atomically "grabs" one pending row in a single transaction. The sequence is: SELECT + FOR UPDATE SKIP LOCKED to lock a row, immediately UPDATE it to running and record claimed_at=now(), then commit. 

func (r *Refs) ClaimJob(ctx context.Context) (*DequeuedJob, error) {
    tx, err := r.pool.Begin(ctx)
    ...
    err = tx.QueryRow(ctx,
        `SELECT id, agent_id, kind, from_sha, attempts FROM memory_jobs
         WHERE state='pending' FOR UPDATE SKIP LOCKED LIMIT 1`).
        Scan(&j.ID, &j.AgentID, &j.Kind, &fromSHA, &j.Attempts)
    ...
    tx.Exec(ctx,
        `UPDATE memory_jobs SET state='running', claimed_at=now() WHERE id=$1`, j.ID)
    tx.Commit(ctx)
    return &j, nil
}

internal/memstore/refs/refs.go:122–151

A few key design points:

  • SKIP LOCKED: skips rows already locked by another transaction, so multiple workers each grab a different job without waiting on each other. There is currently only one worker, but the design scales horizontally with no code changes.
  • No ORDER BY: pending jobs are not ordered ("queue hygiene"), which prevents a class of failing jobs from permanently blocking fair processing of others.
  • LIMIT 1: only one row per call, keeping progress granular and error boundaries tight.
  • claimed_at=now(): records the claim timestamp so the Reaper can later determine whether a job has timed out.

On success, CompleteJob simply DELETEs the row. On failure, RetryJob does attempts++ and sets state to either pending or failed depending on whether maxAttempts has been reached, and clears claimed_at: 

UPDATE memory_jobs
SET attempts = attempts + 1,
    state = CASE WHEN attempts + 1 >= $2 THEN 'failed' ELSE 'pending' END,
    claimed_at = NULL
WHERE id=$1

4. Dispatching by kind + busy-loop prevention

Deps: the dependency container

Deps bundles the dependencies each job handler needs, keeping handlers decoupled from one another: 

type Deps struct {
    Store          memstore.MemStore
    Completer      Completer       // for reflect
    Emb            search.Embedder // for reindex
    EmbCache       cache.Cache     // for reindex
    DefragMaxBytes int             // for defrag
}

internal/maintenance/drain.go:15–21

processJob: dispatch by kind, always resolve the job

func processJob(ctx context.Context, r *refs.Refs, deps Deps,
    job *refs.DequeuedJob, maxAttempts int) error {
    switch job.Kind {
    case "reflect":
        ran, err := r.WithGlobalLock(ctx, agentKey(job.AgentID), func(ctx context.Context) error {
            return Reflect(ctx, deps.Store, deps.Completer, job.AgentID, job.FromSHA)
        })
        if !ran || err != nil {
            return r.RetryJob(ctx, job.ID, maxAttempts)
        }
        return r.CompleteJob(ctx, job.ID)
    case "reindex":
        if err := Reindex(ctx, deps.Store, deps.Emb, deps.EmbCache, job.AgentID); err != nil {
            return r.RetryJob(ctx, job.ID, maxAttempts)
        }
        return r.CompleteJob(ctx, job.ID)
    case "defrag":
        ran, err := r.WithGlobalLock(ctx, agentKey(job.AgentID), func(ctx context.Context) error {
            return Defrag(ctx, deps.Store, job.AgentID, deps.DefragMaxBytes)
        })
        if !ran || err != nil {
            return r.RetryJob(ctx, job.ID, maxAttempts)
        }
        return r.CompleteJob(ctx, job.ID)
    default:
        return r.CompleteJob(ctx, job.ID) // unknown kind: discard rather than block the queue
    }
}

internal/maintenance/drain.go:34–61

Three core rules apply to every branch:

  1. Always resolve the job: regardless of success or failure, processJob always calls either CompleteJob or RetryJob — a job row never stays permanently in running.
  2. Unknown kinds are completed immediately: unknown jobs are not retried, to prevent zombie jobs from clogging the queue.
  3. Failing to acquire the per-agent lock (ran=false) counts as failure: falls back to RetryJob and tries again next round.

DrainJobs: seen-set for busy-loop prevention

func DrainJobs(ctx context.Context, r *refs.Refs, deps Deps, maxAttempts int) (int, error) {
    seen := map[int64]struct{}{}
    processed := 0
    for {
        job, err := r.ClaimJob(ctx)
        ...
        if job == nil { return processed, nil } // queue is empty
        if _, dup := seen[job.ID]; dup {
            // same job seen again within one round — it was requeued
            // RetryJob once more (second attempts bump) then end this round
            r.RetryJob(ctx, job.ID, maxAttempts)
            return processed, nil
        }
        seen[job.ID] = struct{}{}
        processJob(ctx, r, deps, job, maxAttempts)
        processed++
    }
}

internal/maintenance/drain.go:68–91

The seen set tracks the IDs of jobs already processed during this round. If a job is returned to pending by RetryJob and is then claimed again in the same round, DrainJobs immediately calls RetryJob a second time (bumping attempts again) and ends the round. This way a repeatedly failing job eventually reaches failed after enough rounds rather than spinning forever.

Watch out The seen-set is only valid within a single DrainJobs call (one round); it is not persisted across rounds. GC and Reap run before each drain, so stale running jobs are already recovered by the time drain starts — the seen set is always clean at round entry.

5. Per-agent singleton lock + no self-triggering

agentKey: FNV hash to advisory lock key

func agentKey(agentID string) int64 {
    h := fnv.New64a()
    _, _ = h.Write([]byte(agentID))
    return int64(h.Sum64())
}

internal/maintenance/drain.go:25–30

Both reflect and defrag jobs run under r.WithGlobalLock(ctx, agentKey(agentID), ...). This is the maintenance-side per-agent singleton lock, preventing concurrent maintenance writes to the same agent's memory. It uses a different advisory lock key from the foreground writer (foreground uses the global key=1; maintenance uses the agent-hash key) — the two are independent.

Hash collisions only cause two agents to occasionally serialize, which does not affect correctness — real correctness is guaranteed by the CAS in CommitWithCAS; the advisory lock just reduces unnecessary CAS conflicts.

jobs=nil: breaking the self-trigger loop

When reflect and defrag write back, they pass nil as the jobs argument to CommitWithCAS:

// jobs=nil: reflection never enqueues a reflect job (no self-trigger loop).
if _, err := store.CommitWithCAS(ctx, agentID, head, dir, nil); err != nil { ... }

internal/maintenance/reflect.go:72  |  internal/maintenance/defrag.go:159

The foreground agent passes []refs.Job{{Kind:"reflect"},{Kind:"reindex"}} on commit; maintenance passes nil — a maintenance commit never enqueues new maintenance jobs, structurally severing the "maintenance triggers maintenance" feedback loop rather than relying on runtime checks.

Invariant Maintenance commits use jobs=nil. Commits from reflect and defrag never enqueue any new jobs, so the maintenance pipeline cannot grow into an infinite loop. This is a structural guarantee, not a flag or conditional check.

CAS conflict → ErrConflict → retry

If an agent advances HEAD while a maintenance write is in flight (concurrent foreground write), CommitWithCAS's CAS fails. reflect.go and defrag.go surface this as ErrConflict, and processJob calls RetryJob to return the job to pending. On the next drain Reflect re-resolves HEAD, sees the latest state, and re-consolidates — it never force-overwrites.

Reflect execution steps

  1. ResolveHead Fetch the current HEAD (ignores fromSHA — always reads the latest state; fromSHA is preserved for a future diff-since optimization).
  2. Materialize Materialize the HEAD tree into a temporary directory (os.MkdirTemp, defer RemoveAll).
  3. readSystemDir Walk the system/ subdirectory and concatenate all files into a single string with path headers.
  4. Skip if no residents if resident == "" { return nil } — when system/ is empty, no commit is produced, avoiding pointless commit churn.
  5. Complete Call the Completer with reflectSystemPrompt plus the resident content to obtain a consolidated summary.
  6. Write system/reflection.md Write the LLM output into the materialized directory (MkdirAll ensures the path exists).
  7. CommitWithCAS(jobs=nil) CAS commit; conflicts surface as ErrConflict and the caller calls RetryJob.

readSystemDir error propagation

walkErr := filepath.WalkDir(systemDir, func(path string, d fs.DirEntry, err error) error {

        }

fs.ErrNotExist (system/ directory not yet created) is silently handled via SkipAll — readSystemDir returns "" and the caller skips this consolidation pass. Other read errors (permission denied, IO faults) propagate up and ultimately trigger RetryJob.

Design highlight Reflect always works from the latest HEAD, ensuring the consolidated result is never based on a stale resident set. The fromSHA field is already in place as a hook for a future diff-since-last-reflect optimization — at that point only the delta would be sent to the LLM instead of a full re-read.

7. Splitting large files (Defrag)

Splittability check

A file qualifies for splitting (isSplittable) when it satisfies all three conditions:

  1. Has a .md extension;
  2. Byte size exceeds maxBytes (default ENGRAM_DEFRAG_MAX_BYTES=16384);
  3. splitTopLevel produces at least 2 parts (i.e., at least 2 # top-level headings).
func isSplittable(path string, content []byte, maxBytes int) bool {
    return strings.HasSuffix(path, ".md") &&
        len(content) > maxBytes &&
        len(splitTopLevel(content)) >= 2
}

internal/maintenance/defrag.go:52–54

splitTopLevel: splitting on top-level headings

func splitTopLevel(content []byte) [][]byte {
    var parts [][]byte
    var cur []byte
    flush := func() {
        if len(bytes.TrimSpace(cur)) > 0 { parts = append(parts, cur) }
        cur = nil
    }
    for _, ln := range bytes.SplitAfter(content, []byte("\n")) {
        if len(ln) == 0 { continue }
        if isTopHeading(ln) { flush() }
        cur = append(cur, ln...)
    }
    flush()
    return parts
}

internal/maintenance/defrag.go:24–44

Only lines starting with # (a single # followed by a space) trigger a split; ## and deeper headings are not split points. Any preamble before the first top-level heading becomes part 0 if it contains non-whitespace characters.

Why splitting does not recurse forever: structural convergence

Defrag convergence does not rely on "each part being smaller than maxBytes after splitting" — it relies on a structural constraint:

Each base.NN.md part produced by a split contains exactly one top-level heading (the # line that triggered the split). Calling splitTopLevel on that part returns exactly 1 element, so isSplittable is false. The next scan sees it as non-splittable and does not enqueue it; the next Defrag run ignores it too.

Side effect: a single oversized section (larger than maxBytes but with only one top-level heading) is never split. This is intentional — slicing markdown at a byte boundary destroys semantic structure; splits should be driven by document structure, not byte counts.

Defrag convergence flow. Each part produced by a split contains exactly 1 top-level heading, so isSplittable returns false for it permanently.

Invariant Convergence is structurally guaranteed: after each split, every part has exactly 1 top-level heading, making isSplittable permanently false for it — regardless of file size (a single oversized section is left intact).

Naming scheme and the empty-targets optimization

The original file notes.md is deleted and each part is named notes.01.md, notes.02.md, and so on. If the traversal finds no splittable files (targets is empty), Defrag returns nil immediately with no commit produced.

8. Reaping stuck jobs

If a worker crashes while processing a job, that job's state is permanently stuck in running with claimed_at frozen. Each round's ReapStaleJobs moves it back to pending: 

func (r *Refs) ReapStaleJobs(ctx context.Context, before time.Time, maxAttempts int) (int, error) {
    tag, err := r.pool.Exec(ctx,
        `UPDATE memory_jobs
         SET attempts = attempts + 1,
             state = CASE WHEN attempts + 1 >= $2 THEN 'failed' ELSE 'pending' END,
             claimed_at = NULL
         WHERE state='running' AND (claimed_at IS NULL OR claimed_at < $1)`,
        before, maxAttempts)
    ...
    return int(tag.RowsAffected()), nil
}

internal/memstore/refs/refs.go:181–193

before = time.Now().Add(-reapAfter), where reapAfter defaults to 10 minutes (ENGRAM_JOB_REAP_AFTER). claimed_at IS NULL handles the edge case where a row is in running state but claimed_at was never written (defensive handling).

Poison-job protection: every Reap also increments attempts. A job that crashes the worker every time will cycle through "crash → Reap → pending → crash → Reap…" and automatically transition to failed after maxAttempts iterations, never to be picked up by ClaimJob again.

Scenarioclaimed_at stateReap action

claimed_at is NULL (anomaly)NULLattempts++ → pending or failed
Poison job, Nth reapbefore the cutoffattempts reaches limit → failed, loop terminates
Watch out The Reaper is designed assuming a single worker: reapAfter must be significantly longer than the expected execution time of any individual job. If multiple workers run concurrently in the future, reapAfter must be set more conservatively, or a heartbeat-renewal mechanism must be introduced, to avoid misidentifying a healthy running job as stale.

9. Round order and configuration

Logical dependencies in execution order

The execution order within a single round has inherent logical dependencies and cannot be shuffled:

StepWhy it must be hereFailure policy
① GC Runs before drain so this round cleans the most up-to-date reachability set; must never delete objects based on an incomplete reachable set. Entire round returns error, logged, no crash
② ReapStaleJobs Runs before drain so stale running jobs are back in pending and can be picked up in this round's drain. Log only, round continues
③ EnqueueDefrag Runs before drain so defrag jobs found in this scan are processed in this same round's drain (fastest response path). Log only, round continues
④ DrainJobs Runs last; completed GC and Reap are not rolled back by a drain failure. Round returns error, logged
