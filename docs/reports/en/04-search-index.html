Search & Index

Two complementary retrieval strategies — trigram inverted index and paragraph-level semantic vectors — are fused into a single ranked result via RRF. Vectors are computed once and persisted; every subsequent call transparently reuses them across session restarts, pod migrations, and background pre-warming, all sharing the same vector store.

1. Philosophy: memory is fetched on demand, not front-loaded

Engram's recall is a mid-loop tool call — the model issues it on demand during reasoning, not a pre-inference "RAG dump" that retrieves every relevant chunk up front.

Only two things are loaded at the start of inference:

  • Resident system/ files — injected directly into the system prompt, accelerated by the L3 SHA read cache (see report ② Request Path);
  • Tree index (filenames + description fields) — tells the model that long-tail memory exists, without pulling its content in.

Only when the model judges that some long-tail memory is relevant to the current task does it explicitly call recall(query, k), getting back precise line ranges and snippets. This is "fetch on demand" rather than "bulk preload."

Search locality (Variant A, default): The trigram index and semantic index are rebuilt in memory from the materialized working directory when a Session opens, with zero object-storage access at query time. Indexes are discarded when the Session closes — they are disposable derived views, not authoritative storage.

Invariant

Indexes (trigram + semantic) are derived, disposable views. The source of truth lives entirely in the content-addressed objects in the object store plus the Postgres ref. Cache crashes and index loss can all be rebuilt from objects with no impact on correctness (core invariant ③).

2. Keyword retrieval: trigram inverted index

Plain-English explanation of "trigram inverted index": every line of text is broken into sliding windows of three characters (shingles), building a reverse lookup table — given any three-character shingle, you can instantly find which lines contain it. When searching for "project goals," the index first uses pro, roj, oje… to narrow the candidate lines, then exactly verifies which lines actually contain the full string.

This index is pure CPU, zero I/O. It is rebuilt synchronously with the materialized tree when a Session opens, lightweight enough to be discarded and rebuilt at any time.

Trigram extraction

// trigram.go:21–37
func trigrams(s string) []string {
    r := []rune(strings.ToLower(s))
    if len(r) < 3 { return nil }
    seen := make(map[string]struct{}, len(r))
    out := make([]string, 0, len(r)-2)
    for i := 0; i+3 <= len(r); i++ {
        t := string(r[i : i+3])
        if _, ok := seen[t]; ok { continue }   // deduplicate: count each trigram once per line
        seen[t] = struct{}{}
        out = append(out, t)
    }
    return out
}

Deduplication is essential: a trigram that appears more than once in a line is counted only once, preventing posting-list bloat. Both indexing and querying are lowercased; returned snippets preserve original casing.

Candidate intersection → exact verification

candidates(query) (trigram.go:65–102) works in two steps:

  1. Slice into trigrams, look up posting lists.Each three-character shingle of the query fetches the set of lines that contain it; if any trigram is absent from the entire index, return empty immediately (early exit).
  2. Intersect starting from the smallest set.Sort posting sets by size ascending and filter from the smallest one, minimizing wasted iteration.
  3. Exact substring verification.strings.Contains(lowercase(lineText), lowercase(query)) is the final gate — the trigram intersection is a pre-filter (may have false positives); the substring check removes them. No false negatives, no false positives.
  4. Short queries fall back to full scan.Queries shorter than 3 runes have no trigrams to use, so a linear scan over all lines is performed.

3. Semantic retrieval: paragraph vector index

Plain-English explanation: a passage of text is "turned into a sequence of numbers" such that passages with similar meaning produce similar number sequences (high cosine similarity). Keyword retrieval requires an exact token match between query and document; semantic retrieval captures cross-lingual and near-synonym associations like "项目目标" and "project goals," as well as meaning distributed across multiple lines.

Sectionization logic

sectionize (semantic.go:40–52) splits a file into sections at any Markdown heading line (lines starting with #). Content before the first heading forms its own section; a file with no headings is treated as a single section. Each section carries a 1-based line range used for Hit location.

Embedding key design — content addressing

// semantic.go:63–66
func embKey(model, text string) string {
    sum := sha256.Sum256([]byte(text))
    return "emb:" + model + ":" + base64.RawStdEncoding.EncodeToString(sum[:])
}

Two pieces of information make up the key:

  • model prefix — different embedding models have different vector dimensions and semantic spaces; mixing them makes similarity scores meaningless;
  • sha256(text) — identical content maps to the same key across any time and any worker; the key changes naturally when content changes, providing inherent content addressing.

BuildSemantic: cache hit → reuse, miss → batch Embed → Put

Vectors are stored as base64-encoded little-endian float32 byte strings (semantic.go:68–86), directly compatible with cache.Cache's string interface — no extra serialization layer needed.

4. Merging both strategies: RRF fusion

Plain-English explanation of RRF (Reciprocal Rank Fusion): score each candidate result by its rank in each list — higher rank means higher score; results that rank high in both lists accumulate the highest total score. The formula is 1 / (60 + rank + 1); the constant 60 smooths the gap between top ranks (a standard hyperparameter), so the score difference between rank 1 and rank 2 is not dramatic. When the same result appears in multiple lists, its scores are summed.

RRF implementation

// hybrid.go:56–85
func rrf(lists [][]Hit, k int) []Hit {
    const rrfK = 60
    scores := map[string]*agg{}
    var order []string   // insertion order for stable tie-breaking
    for _, list := range lists {
        for rank, hit := range list {
            id := fmt.Sprintf("%s:%d-%d", hit.Path, hit.LineStart, hit.LineEnd)
            a, ok := scores[id]
            if !ok {
                a = &agg{hit: hit}
                scores[id] = a
                order = append(order, id)
            }
            a.score += 1.0 / float64(rrfK+rank+1)   // accumulate contribution
        }
    }
    sort.SliceStable(order, func(i, j int) bool {
        return scores[order[i]].score > scores[order[j]].score
    })
    // take top-k
}

Hit IDs are identified as path:lineStart-lineEnd; line-level and paragraph-level Hit IDs use the same format, so when a single line is also an entire section, scores from both indexes are merged.

Degradation strategy

NewHybrid (hybrid.go:22–28) builds the semantic index on a best-effort basis at construction time — if BuildSemantic returns an error (e.g., the embedding service is unavailable), h.sem stays nil, and Recall uses only the trigram list without returning an error to the caller. Query-time semantic failure degrades silently in the same way: 

// hybrid.go:41–47
if h.sem != nil {
    if semHits, err := h.sem.Search(ctx, query, n); err == nil {
        lists = append(lists, semHits)
    }
    // query-time embedding failure: drop semantic, keep trigram (degrade gracefully)
}
Invariant

Recall never returns an error due to embedding service failure. In the worst case it degrades to pure trigram retrieval, which covers exact-vocabulary-match scenarios. See test TestHybridDegradesWhenEmbeddingFails (hybrid_test.go).

5. Content-fingerprint read cache

cache.Cache (cache.go:13–16) is a minimal string KV interface: 

type Cache interface {
    Get(key string) (string, bool)
    Put(key, val string)
}

All cache keys in Engram are content hashes, so a cache hit is equivalent to recomputing the value — there is no stale-value problem. The only form of "invalidation" is LRU eviction under memory pressure, which affects hit rate but not correctness.

LRU (cache.go:23–69):

  • A single sync.Mutex guards all operations — Get needs to update the recency list (MoveToFront), so an RWMutex cannot be used;
  • Capacity is bounded by entry count; the oldest entry is evicted when the limit is exceeded;
  • A repeated Put for the same key refreshes recency without adding an entry.

The Session layer has two separate cache instances (see the comment at router.go:40: "they MUST be different instances"):

  • sysCache — cache.NewLRU(1024), caches the assembled system/ content; see report ②;
  • embCache — cache.NewTiered(LRU, ObjCache), caches embedding vectors; see §6.
Invariant

sysCache and embCache are distinct instances pointing to different backing ObjStore directories. System content is never written into the embedding store, and embedding vectors never pollute the system cache.

6. Persisting expensive vectors (L4b)

Voyage API calls carry both latency and cost, yet a section's content is immutable once written (content-addressed) — the vector computed for the same section is identical at any time, on any worker. L4b persists embedding vectors to a separate content-addressed ObjStore, turning "expensive network embedding calls" into "cheap disk/S3 reads."

The key elegance of the design: the code that computes vectors (BuildSemantic) is unchanged by a single line; persistence is achieved purely by swapping in a different cache implementation — dependency injection, zero intrusion.

ObjCache: the persistence layer

// objcache.go:21–36
func safeKey(key string) string {
    sum := sha256.Sum256([]byte(key))
    return hex.EncodeToString(sum[:])   // pure hex, safe for file paths and S3 keys
}

func (o *ObjCache) Get(key string) (string, bool) {
    data, err := o.objs.Get(context.Background(), safeKey(key))
    if err != nil { return "", false }   // error = miss, best-effort
    return string(data), true
}

func (o *ObjCache) Put(key, val string) {
    _ = o.objs.Put(context.Background(), safeKey(key), []byte(val))   // errors ignored
}

embKey is already a base64 string (containing :), which is unsafe to use directly as a file path; safeKey applies another sha256 + hex encoding. The key is "the hash of the key," not "the hash of the value," but uniqueness is guaranteed by embKey's internal sha256(model+text).

Tiered: memory layer in front, persistence layer behind

// tiered.go:11–25
func (t *Tiered) Get(key string) (string, bool) {
    if v, ok := t.front.Get(key); ok { return v, true }   // check LRU first
    if v, ok := t.back.Get(key); ok {
        t.front.Put(key, v)   // promote from disk to memory
        return v, true
    }
    return "", false
}

func (t *Tiered) Put(key, val string) {
    t.front.Put(key, val)   // synchronous dual write
    t.back.Put(key, val)
}

On read, the LRU is checked first (fast); on LRU miss the ObjCache (disk/S3) is queried, and a hit is promoted to LRU for the next warm read. Writes go synchronously to both layers, keeping persistence and in-memory cache consistent.

BuildSemantic accepts a cache.Cache interface and has no knowledge of whether the backing store is an LRU, ObjCache, or Tiered. semantic.go is unchanged; persistence is delivered entirely through dependency injection.

Win

When a Session opens for the first time, if the reindex worker has already pre-warmed embeddings, every section hits the ObjCache and Voyage API calls are zero. Across Session restarts, pod migrations, and process restarts, embedding persistence is transparently effective — the behavior of BuildSemantic does not change at all.

7. Incremental updates: background pre-warming, zero network at session open

maintenance.Reindex (reindex.go:23–46) is the maintenance-side component of L4b: it materializes the agent HEAD's complete tree (not just system/ — search indexes all files), calls BuildSemantic, and its primary purpose is to trigger the Put side-effect — writing every section's embedding into the persistent embCache. The returned *SemanticIndex is discarded. 

// reindex.go:41–44
// BuildSemantic persists missing embeddings into embCache as a side-effect.
if _, err := search.BuildSemantic(ctx, emb, embCache, files); err != nil {
    return fmt.Errorf("maintenance: reindex build semantic: %w", err)
}

Why no diff is needed

Content addressing makes incremental updates naturally free — no diff computation required:

  1. Compute embKey for each section.embKey(model, text) = sha256(model+text); identical content produces identical key.
  2. Check embCache.Already-persisted sections hit immediately, skipping the Embed call.
  3. Only new or modified sections touch the network.The Voyage API is called only when the key does not exist (the section is new or its content has changed).
  4. A second reindex makes zero network calls.For the same HEAD, all sections are already persisted; the Embed call count is zero.

This behavior is precisely verified with a countingEmbedder: 

// reindex_test.go:46–52
// Second reindex: every chunk's embedding is already persisted → 0 new embeds.
if err := Reindex(ctx, store, ce, embCache, agentID); err != nil { ... }
if ce.calls != first {
    t.Fatalf("second reindex embedded %d more texts, want 0 (incremental persistence broken)",
        ce.calls-first)
}

Frontend Session and background worker share the same vector store

Both sides (cmd/api and cmd/maintenance) each build an ObjStore pointing to the same directory (ENGRAM_EMB_OBJ), then each wraps it in a Tiered(LRU + ObjCache). The LRU is in-process and not shared; the ObjStore is disk/S3 and is shared naturally.

Gotcha

Both sides must use the same embedding model (ENGRAM_EMBEDDER environment variable). embKey carries the model prefix; different models produce different keys, making the vector store useless — embeddings pre-warmed by maintenance are completely invisible to the Session, which falls back to calling Voyage on every open.

ENGRAM_PROVIDER (LLM inference provider) and ENGRAM_EMBEDDER (vector model) are two independent variables; do not conflate them when configuring. In production both sides should be hardcoded to voyage.

Background pre-warm → session zero-network sequence

The background worker computes and persists each section's vector via a network call on the first pass; subsequent Session opens find all vectors on disk, making zero Voyage API calls.

8. Isolation from GC

Engram's GC (L5a, maintenance/gc.go) sweeps unreferenced git objects via reachability analysis. GC operates on the git object store (ENGRAM_OBJ); embedding vectors are stored in a separate directory (ENGRAM_EMB_OBJ, default ./engram-embeddings). These are different objstore.NewLocal instances; the GC reachability computation never touches the embedding store.

Both commands enforce directory separation at startup: 

// cmd/api/main.go:81–83
if filepath.Clean(embObjRoot) == filepath.Clean(objRoot) {
    log.Fatalf("ENGRAM_EMB_OBJ and ENGRAM_OBJ must be different directories (got %q)", embObjRoot)
}

// cmd/maintenance/main.go:133–135
if filepath.Clean(embObjRoot) == filepath.Clean(objRoot) {
    log.Fatal("ENGRAM_EMB_OBJ must differ from ENGRAM_OBJ (GC must never sweep the embedding store)")
}
Invariant

GC never sweeps embedding vectors. The embedding store is an append-only content-addressed store: vectors for old sections are not referenced by the current HEAD, but they are also not in the git object store, so GC cannot reach them. The hard path-equality check at startup is a second layer of defense.

Expiry management for embeddings (if ever needed) is a problem independent of git GC, and is not currently implemented (acceptable: memory footprint is small, vector accumulation rate is slow).

A full analysis of git GC is in report ③ Storage & Consistency.

Hybrid = Trigram + Semantic

Lexical precision + semantic fuzziness, fused by RRF(k=60); silently degrades to pure trigram on embedding failure — Recall never errors.

Tiered(LRU + ObjCache) injected into BuildSemantic; semantic.go unchanged, Voyage calls drastically reduced.

Incremental free

Content-addressed Keys
