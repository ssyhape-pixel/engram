Request Path: Router · Session · Agent Loop · Tools

From a user message to a durable commit — a two-layer single-writer design, context assembly, tool-driven memory retrieval and writes, and the CAS commit protocol.

1. Overview

Engram's request path (architecture zone ①) is the only part of the system that talks directly to an LLM provider. Its philosophy fits in one sentence: compute is stateless; state lives in the memory repository.

Every conversation maps to a Session. A Session holds two transient things — chat history and a materialised working directory (workdir) — but neither constitutes persistent state. The real persistent state is the agent's memory repository: a git tree in content-addressed object storage, plus a HEAD ref in Postgres. When the Session ends the workdir is destroyed. The next time it opens it is re-materialised from HEAD — no state leaks.

Core invariant At most one active Session per agentID at any time. The Router uses sync.Mutex + active map[string]bool to quickly block same-process concurrency; the Postgres ref CAS (ErrCASConflict) is the final cross-pod backstop. No lossy automatic Markdown merge is ever performed.

2. Single-writer: why + two-layer protection

Why must writes be serialised?

Memory files are structured prose Markdown, not database rows. When two writers concurrently modify the same file, any automatic three-way merge (git-merge style) produces semantic errors — context is severed, facts are scrambled, and contradictory sentences can be conjured from thin air. This kind of corruption is silent and hard to trace back.

Engram's answer is: abandon multi-writer entirely and use single-writer serialisation. The cost is reduced concurrency (only one Session per agent at a time), but memory correctness is guaranteed absolutely. The two protection layers each guard a different failure surface:

In-process lock — fast path

Guards against: concurrent requests within the same pod (multiple threads / goroutines on the same machine).

Implementation: sync.Mutex + active map[string]bool (router.go:32-33). Non-blocking admission: if already occupied, immediately returns ErrAgentBusy without queuing.

Value: kills the conflict before the expensive object writes and Postgres transaction, adding zero extra latency.

Database CAS — cross-machine backstop

Guards against: sticky routing failures in multi-pod deployments, or out-of-process tools writing directly to Postgres.

Implementation: refs.CommitRef performs an atomic WHERE head=$parent update in a single Postgres transaction; if the parent hash doesn't match it returns ErrCASConflict (memstore/memstore.go:65).

Value: even if the process lock is bypassed, the database layer ensures the ref can never be silently overwritten. See ③ Storage & Consistency for details.

2.1 Process lock: claim / free

The Router struct (internal/agent/router.go:24-34): 

type Router struct {
    store    memstore.MemStore
    prov     LLMProvider
    scratch  string
    cache    cache.Cache   // sysCache — L3 system/ read cache
    embCache cache.Cache   // embCache — L4 vector embedding cache (must be separate)
    emb      search.Embedder

    mu     sync.Mutex
    active map[string]bool
}

Both claim and free operate under mu (router.go:45-60): 

func (r *Router) claim(agentID string) bool {
    r.mu.Lock()
    defer r.mu.Unlock()
    if r.active[agentID] {
        return false          // already occupied — reject immediately
    }
    r.active[agentID] = true
    return true
}

func (r *Router) free(agentID string) {
    r.mu.Lock()
    delete(r.active, agentID)
    r.mu.Unlock()
}
Design choice The lock semantics are admission, not queuing. ErrAgentBusy means "not now" — the caller decides the retry strategy (return 429 immediately, sleep briefly and retry, or enqueue at a higher layer). The Router itself holds no queuing state, perfectly consistent with the "stateless worker" philosophy.

2.2 sync.Once prevents double-free

Open wraps release in a sync.Once before handing it to the Session (router.go:125-131): 

var once sync.Once
release := func() {
    once.Do(func() {
        os.RemoveAll(workdir)
        r.free(agentID)
    })
}

Scenario: the caller executes both defer sess.Close() and an explicit sess.Close(), or both the error path and the happy path trigger a close. Without sync.Once, a second free call would delete the active map entry — which might already have been set again by a new Session, causing that Session to be incorrectly "freed." sync.Once guarantees RemoveAll + free execute exactly once as an atomic unit, no matter how many times Close is called.

2.3 What each layer guards — side-by-side

3. The lifecycle of a Session

A Session is the complete lifecycle container for a single conversation (internal/agent/session.go:22-34): 

type Session struct {
    agentID  string
    store    memstore.MemStore
    prov     LLMProvider
    tools    *Toolset
    head     memstore.CommitHash  // currently committed HEAD (advances on each commit)
    workdir  string               // materialised temporary working directory
    history  []Message            // this Session's chat history (in-memory only)
    dirty    bool                 // whether this turn had a successful edit
    maxSteps int                  // guards against runaway loops, default 16
    release  func()               // called on Close: RemoveAll + free
    cache    cache.Cache          // L3 SHA-keyed cache; nil means recompute each turn
}

history lives only for the duration of the Session. dirty is the write marker for the current turn (a Send call); it is cleared after a successful commit. head advances after each successful commit and always corresponds to a real commit in object storage.

3.1 Open: full sequence of steps

  1. claim(agentID) Checks the active map under mu; returns ErrAgentBusy immediately if already occupied.
  2. ResolveHead(agentID) Reads the current HEAD commit hash from Postgres (router.go:96). On failure: free immediately and return the error.
  3. MkdirTemp Creates a temporary working directory under the scratch directory. Any / in the agentID is replaced with _ to be compatible with the MkdirTemp pattern (router.go:104-106).
  4. Materialize(HEAD, workdir) Calls gitfs.Materialize to write the git tree referenced by the HEAD commit into the working directory (router.go:111).
  5. readWorkdirFiles + NewHybrid Reads all workdir files into a map[string][]byte and constructs a search.HybridSearch snapshot (router.go:116-122).
  6. sync.Once release closure + NewSession Assembles the release function (wrapped in Once) and returns the *Session (router.go:125-132).

3.2 Send: full timing of one turn

Send (session.go:73-117) is the core loop of the request path. It records base := len(s.history) on entry and rolls back to base on failure, ensuring history never holds dangling entries.

  1. history += user message Appends Message{Role: RoleUser, Text: userMessage}.
  2. assembleSystem(ctx) Builds the system prompt: resident system/ content + tree index. May hit the L3 cache (see §4).
  3. Generate(System, history, Tools) Calls LLMProvider. Returns ToolCalls (turn not finished) or a final Text.
  4. Dispatch each ToolCall Toolset.Dispatch routes to list / read / recall / edit. A successful edit sets s.dirty = true (session.go:98-99).
  5. history += assistant-toolcalls + tool-results Appends two messages, then loops back to step 2. At most maxSteps=16 iterations.
  6. Model returns final text, exits loop When len(resp.ToolCalls) == 0, break; history += assistant text.
  7. commit (only when dirty=true) Calls store.CommitWithCAS: writes objects first, then advances the ref (see §7). On success: head = newHead, dirty = false.

Failure rollback: if steps 2/3 error, or the loop exhausts maxSteps, execute s.history = s.history[:base]. If the commit step (step 7) fails, history is not rolled back and dirty stays true — the next Send call can re-attempt the commit.

4. Context assembly

Before each Generate call, assembleSystem constructs the complete system prompt string. It is made up of two concatenated parts:

PartSourceCache key (L3)Purpose
Resident system/ content All files under the system/ directory in workdir sys:<systemSubtree-SHA> Core memory the model sees every turn (persona, rules, key facts)
Tree index walkDir over the full tree, reading each file's frontmatter description: idx:<rootTree-SHA> "path: description" index — lets the model know long-tail files exist so it can recall them on demand

4.1 Cache branch vs. direct recompute

Full logic (session.go:141-170):

func (s *Session) assembleSystem(ctx context.Context) (string, error) {
    if s.cache == nil || s.dirty {
        return s.buildResident()  // no cache or uncommitted edits: recompute from workdir
    }
    rootTree, systemSubtree, err := s.store.TreeKeys(ctx, s.head)
    if err != nil {
        return s.buildResident() // TreeKeys read failure is non-fatal; degrade to recompute, don't abort session
    }
    sysKey := "sys:" + string(systemSubtree)
    sys, ok := s.cache.Get(sysKey)
    if !ok {
        built, berr := s.buildSystemContent()
        if berr != nil { return "", berr }
        sys = built
        s.cache.Put(sysKey, sys)
    }
    idxKey := "idx:" + string(rootTree)
    idx, ok := s.cache.Get(idxKey)
    if !ok {
        built, berr := s.buildTreeIndex()
        if berr != nil { return "", berr }
        idx = built
        s.cache.Put(idxKey, idx)
    }
    return sys + idx, nil
}
Cache correctness Cache keys are immutable git tree hashes, read directly from object storage by TreeKeys — not routed through Postgres. As long as HEAD hasn't advanced, the system/ subtree content is unchanged, and sys:SHA remains valid forever. When dirty=true (workdir has uncommitted edits), the cache is bypassed entirely and recomputed directly from workdir — ensuring the model sees the latest edits rather than a stale cached version. A TreeKeys read failure also silently degrades to recompute, never aborting the session. The underlying L3 cache implementation is described in ④ Search & Index.

4.2 buildSystemContent — resident content

(session.go:187-213) Walks workdir/system/; each file is appended to the builder in ## path\ncontent\n format. If the system/ directory does not exist, WalkDir triggers fs.ErrNotExist, which is converted to fs.SkipAll (silent — returns an empty string). Real I/O errors (permission issues, disk failure) are propagated in full — preventing partial content from being cached.

4.3 buildTreeIndex — the cheap directory

(session.go:217-238) Walks the full workdir tree; for each file it calls frontmatterDescription(path) (tools.go:172-192) — a bufio.Scanner that scans line by line, finds the YAML frontmatter --- delimiters, and extracts the description: field value. This is read-only frontmatter: the file body is never read, so token overhead is minimal. All lines are sorted then concatenated in path: description format.

Design highlight Memory is agentic, not RAG. The tree index gives the model only "clues" (path + description) rather than stuffing the full long-tail content into context. The model reasons over the index during inference to decide which files are relevant, then fetches them precisely via the recall / read tools — saving tokens while keeping information semantically accurate.

5. Tools: how the model "goes and looks things up"

Toolset (internal/agent/tools.go:18-25) is bound to a single Session's workdir and is itself stateless. All tools are routed through Dispatch(ctx, ToolCall) ToolResult. The Session tracks dirty; the Toolset holds no write marker of its own.

5.1 Tool definitions (the schema sent to the model)

(tools.go:30-35) Four tools:

// list — list memory files (includes descriptions, i.e. the tree index)
{Name: "list",
 Description: "List memory files with their descriptions (the tree index).",
 InputSchema: {"type":"object","properties":{}}}

// read — read a file, with an optional 1-based line range
{Name: "read",
 Description: "Read a memory file, optionally a 1-based inclusive line range.",
 InputSchema: {"type":"object",
   "properties": {"path":{"type":"string"},
                  "start":{"type":"integer"},   // 1-based, inclusive
                  "end":  {"type":"integer"}},  // 1-based, inclusive
   "required": ["path"]}}

// recall — search memory, returns matching line ranges
{Name: "recall",
 Description: "Search memory for a query; returns matching line ranges.",
 InputSchema: {"type":"object",
   "properties": {"query":{"type":"string"}},
   "required": ["query"]}}

// edit — write to a memory file (create or overwrite)
{Name: "edit",
 Description: "Write content to a memory file (creates or overwrites).",
 InputSchema: {"type":"object",
   "properties": {"path":{"type":"string"}, "content":{"type":"string"}},
   "required": ["path","content"]}}

5.2 recall — search the long tail, return line ranges

(tools.go:140-153) doRecall calls t.search.Recall(ctx, t.agentID, query, 8), returning up to 8 Hit{Path, LineStart, LineEnd, Snippet} values serialised as path:start-end: snippet format back to the model: 

for _, h := range hits {
    fmt.Fprintf(&b, "%s:%d-%d: %s\n", h.Path, h.LineStart, h.LineEnd, h.Snippet)
}

L2's backend is GrepSearch (substring scan over workdir); L4 replaces it with HybridSearch (trigram + vector hybrid). The Search interface is unchanged; the tool code is unaware of the swap. See ④ Search & Index for details.

5.3 read — fetch by line range

(tools.go:110-138)

start, end := intInput(in, "start"), intInput(in, "end")
if start <= 0 && end <= 0 {
    return okResult(callID, string(data))   // no range: return full content
}
// after clamping boundaries:
return okResult(callID, strings.Join(all[start-1:end], "\n"))
Note: 1-based line numbers The start/end parameters of read are a 1-based closed interval, perfectly aligned with the LineStart/LineEnd values returned by recall. When the model receives notes.md:5-12 from recall, it can pass start=5, end=12 directly to read and get back exactly those lines — zero conversion needed. The internal implementation is all[start-1 : end] (Go slice, 0-based half-open interval). A 0-based API would require the model to subtract one every time, making off-by-one errors very easy to introduce. start<=0 && end<=0 explicitly means "no range restriction," which is unambiguous with 1-based (valid line numbers are >= 1).

5.4 edit — write to workdir, set dirty

(tools.go:155-168) doEdit first calls safePath (tools.go:47-54) to run filepath.Clean and a prefix check, rejecting any .. escape paths. Then os.MkdirAll creates the parent directories, and os.WriteFile overwrites (or creates) the file. Session's Send checks call.Name == "edit" && !res.IsError after Dispatch returns and sets s.dirty = true (session.go:98-99).

6. The abstraction for talking to models

6.1 Interface

(internal/agent/provider.go:62-64)

type LLMProvider interface {
    Generate(ctx context.Context, req Request) (Response, error)
}
Boundary value This interface is the sole boundary between the agent loop and any LLM provider. Session never imports the Anthropic SDK or sends HTTP requests directly. FakeProvider drives all automated tests through this same interface — zero network calls. Swapping providers requires only implementing this interface; no Session or Toolset code changes.

6.3 FakeProvider — scripted deterministic driver

(internal/agent/fake.go)

type FakeProvider struct {
    Steps []func(req Request) Response
    calls int
}
func (f *FakeProvider) Generate(ctx context.Context, req Request) (Response, error) {
    if f.calls >= len(f.Steps) {
        return Response{}, fmt.Errorf("fake: no scripted response for call %d (have %d)", f.calls, len(f.Steps))
    }
    step := f.Steps[f.calls]
    f.calls++
    return step(req), nil
}

Each Step receives the full Request (including the system prompt and history) and can make assertions within it — verifying that context assembly is correct. A typical test script: Step 0 returns a recall ToolCall, Step 1 returns an edit ToolCall, Step 2 returns the final text. When the call count exceeds the script length an error is returned, preventing a test with missing steps from silently passing.

6.4 AnthropicProvider — real API integration

(internal/agent/anthropic.go) Maps the provider-agnostic protocol to the Anthropic Messages API (POST /v1/messages), default model claude-sonnet-4-6, max_tokens=4096, version header anthropic-version: 2023-06-01. The HTTP client is injectable (anthropic.go:33-36) for substituting a mock RoundTripper in tests.

Message format mapping (anthropic.go:88-121):

Internal RoleAnthropic representationNotes
RoleUser user + text block
RoleAssistant (with text) assistant + text block + optional tool_use blocks No text block added when text is empty
RoleAssistant (tool calls only) assistant + tool_use blocks Entire message skipped when blocks are empty (anthropic.go:108-110)
RoleTool user + tool_result blocks Anthropic requires tool results to be sent back as the user role
Two key guards for the Anthropic API

(1) The input field of a tool_use block must not be null — a nil map is replaced with an empty map (anthropic.go:103-105).

(2) An assistant message must have at least one content block, otherwise the API returns 400 — an empty blocks slice causes a continue that skips the entire message (anthropic.go:108-110).

The system and tools fields of apiRequest are tagged omitempty, so they are not serialised to JSON when empty, avoiding validation errors from the Anthropic API on empty fields.

7. Commit and conflict

7.1 Two-phase write order: objects before ref

CommitWithCAS (memstore/memstore.go:60-68) strictly follows "write objects first, then advance the ref":

gitfs.Commit Write blob + tree + commit objects to ObjStore
Content-addressed, idempotent
on success
refs.CommitRef Single Postgres tx:
CAS HEAD parent→new
+ enqueue reindex + reflect

If objects write successfully but the Postgres tx fails (or the process crashes at that point), only unreachable garbage objects remain — GC (L5a) handles cleanup, no torn ref is produced. The reverse situation (ref advances before objects are flushed) is impossible under this write order.

7.2 Commit enqueues two maintenance jobs

// session.go:121
jobs := []memstore.Job{{Kind: "reindex"}, {Kind: "reflect"}}

Both jobs are enqueued in the same Postgres transaction as the ref CAS (see ⑤ Maintenance Pipeline). This guarantees "a job is enqueued if and only if the commit succeeds" — no Kafka needed, no outbox table: the Postgres transaction itself is the atomic unit.

7.3 ErrCASConflict — surface it, don't clobber

(session.go:123-128)

if errors.Is(err, memstore.ErrCASConflict) {
    // Impossible under single-writer-per-agent (Router lock). If it occurs,
    // the invariant was violated; surface it rather than silently clobbering
    // the other writer's tree (no lossy merge). Multi-writer reconciliation is L5.
    return fmt.Errorf("agent: commit conflict — single-writer invariant violated for %s: %w",
        s.agentID, err)
}
Design decision: no retry, no clobber

Under correct Router single-writer protection, ErrCASConflict should never occur. If it does, it means there was a side-channel write — an out-of-process tool modified Postgres directly, or sticky routing failed in a multi-pod setup.

An approach that was considered but rejected: bounded retry + clobber (re-materialise the new HEAD, replay this turn's edits, then commit). The fundamental reason for rejection:

  • "Replaying edits" requires merging this turn's changes onto a file tree that has already been modified by the other writer — for prose Markdown this is a lossy operation that can introduce semantic errors.
  • Discarding the other writer's changes equally violates the "no lossy merge" principle.
  • Multi-writer coordination (reconcile + reflection) is L5's responsibility and should not be silently fixed at the Session layer.

Current behaviour: surface the error transparently so the caller knows the invariant was violated, rather than silently "fixing" a problem that should not be fixed at this layer.

7.4 Commit flow summary

StepLocationFailure consequence
1. gitfs.Commit — write all objects to ObjStore memstore.go:61 Only garbage objects; ref unchanged; GC cleans up; Session dirty stays true
2. refs.CommitRef — CAS HEAD + enqueue jobs (single Postgres tx) memstore.go:65 ErrCASConflict (single-writer violated) or generic DB error; surfaced to caller
3. Session updates head, dirty=false session.go:132-133 On commit failure dirty stays true; next Send can re-attempt the commit

8. cmd/api wiring

cmd/api/main.go is the interactive development harness that wires the entire request path together: 

// storage layer
store := memstore.New(objstore.NewLocal(objRoot), refs.New(pool))

// provider selection
switch providerName {
case "anthropic":
    prov = agent.NewAnthropic(key)
default:
    prov = &agent.FakeProvider{Steps: []func(agent.Request) agent.Response{
        func(r agent.Request) agent.Response {
            return agent.Response{Text: "(fake) received your message"}
        },
    }}
}

// embedder selection
switch env("ENGRAM_EMBEDDER", "fake") {
case "voyage":
    emb = search.NewVoyage(vkey)
default:
    emb = search.NewFakeEmbedder(256)
}

// embedding cache (tiered: in-memory LRU + persistent ObjCache)
embCache := cache.NewTiered(cache.NewLRU(4096), cache.NewObjCache(objstore.NewLocal(embObjRoot)))

// defensive check: sysCache and embCache must use different root directories
if filepath.Clean(embObjRoot) == filepath.Clean(objRoot) {
    log.Fatalf("ENGRAM_EMB_OBJ and ENGRAM_OBJ must be different directories")
}

// Router wiring
router := agent.NewRouter(store, prov, os.TempDir(), cache.NewLRU(1024), embCache, emb)

The last two lines are critical: sysCache (cache.NewLRU(1024)) and embCache (cache.NewTiered) are different instances with different underlying object root directories. Sharing a single instance would cause plain-text system/ content to be written into the vector embedding store, leading to type confusion and backend pollution. When embObjRoot == objRoot the program fatals on startup — an active defence of this invariant.
