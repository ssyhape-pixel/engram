Storage & Consistency

Immutable objects · gitfs · refs CAS · job queue · advisory lock · GC · migrations — Engram's persistence layer from top to bottom

Layers covered: L1 MemStore core · L2 agent loop · L3 SHA read cache · L5a GC · L5b maintenance jobs

Engram's persistence layer is made up of three collaborating pieces: a hash-keyed immutable object store (ObjStore), a custom git Storer (gitfs) that uses git purely as a data format, and a single mutable pointer stored in Postgres (refs). Together they guarantee that objects are immutable, history is replayable, and concurrent writes never produce a torn state.

The central interface is MemStore (internal/memstore/memstore.go:33–38) — four methods: ResolveHead, Materialize, CommitWithCAS, TreeKeys. Each section below dissects the storage machinery behind one of those methods.

1 · Immutable objects, keyed by fingerprint

Every unit of storage — blob (file contents), tree (directory snapshot), commit (version node) — is keyed by the SHA-1 hash of its own content. "Same content" and "same key" are the same thing, which means:

  • Cache by SHA, no invalidation logic needed. The same hash always maps to the same bytes; a cache hit never needs a freshness check.
  • Writes deduplicate naturally. If two agents have an identical system/persona.md, only one blob is stored.
  • GC can use reachability. Walk the object graph from all agent HEADs; anything not reached is garbage and safe to collect.

ObjStore interface

The interface is defined in internal/memstore/objstore/objstore.go:16–26 with six methods: 

type ObjStore interface {
    Has(ctx context.Context, key string) (bool, error)
    Get(ctx context.Context, key string) ([]byte, error)
    Put(ctx context.Context, key string, data []byte) error // idempotent
    Iter(ctx context.Context, fn func(key string) error) error
    Stat(ctx context.Context, key string) (time.Time, error) // returns object creation time, used by GC
    Delete(ctx context.Context, key string) error             // missing key is not an error (idempotent)
}

key is a hex hash string. The comment on Put says idempotent explicitly: the same key implies identical bytes, so writing it twice is harmless. Stat returns the last-modified time — which for an immutable object equals its creation time — and GC uses this value to enforce the grace period.

Local backend: atomic writes

The development and test environment uses a local filesystem backend (internal/memstore/objstore/local.go). Objects are stored under <root>/objects/<key[:2]>/<key> — a two-character prefix shard, identical to the .git/objects/ layout, to avoid oversized single directories.

Put uses the write-to-temp-then-os.Rename pattern for atomic visibility (local.go:53–79): 

func (l *Local) Put(ctx context.Context, key string, data []byte) error {
    p := l.path(key)
    if _, err := os.Stat(p); err == nil {
        return nil  // already exists, idempotent return
    }
    os.MkdirAll(filepath.Dir(p), 0o755)
    tmp, _ := os.CreateTemp(filepath.Dir(p), ".tmp-*")
    tmpName := tmp.Name()
    defer os.Remove(tmpName)
    tmp.Write(data)
    tmp.Close()
    os.Rename(tmpName, p)  // atomic visibility: readers see either nothing or complete bytes
}

os.Rename within the same filesystem is atomic: any reader either does not see the file at all or sees the complete bytes. Because objects are immutable, if two goroutines race to write the same key, the last rename wins — the result is always correct.

Object graph: commit → tree → blob + parent chain

Fig 1 · Object graph: commit→tree→blob + parent chain. Changing one file = new blob + new affected trees + new commit; C0 and all its objects are left intact.

Invariant Objects are never mutated. Changing a file = new blob + new tree (and all ancestor trees affected) + new commit. Old objects are retained until GC determines they are unreachable and past the grace period. Nothing is ever overwritten in place.

2 · gitfs: using git as a working format

Engram uses git's blob/tree/commit data structures to represent memory versions, but never shells out to the git binary and has no dependency on a .git/ directory. All git I/O goes through go-git, with a custom Storage that replaces its default filesystem backend.

Storage: a custom go-git Storer

internal/memstore/gitfs/storage.go implements go-git's storage.Storer interface, bridging the object layer to ObjStore. It embeds a memory.Storage and delegates non-object data (refs, config, index) to it — those are ephemeral per-session state; the authoritative ref always lives in Postgres.

The storage format uses git loose-object frame encoding (storage.go:42–59): 

// frame encoding: "<type> <size>\x00<content>"
// sha1(frame(o)) == o.Hash() — this is exactly git's hash definition
func frame(o plumbing.EncodedObject) ([]byte, error) {
    content, _ := io.ReadAll(r)
    buf.WriteString(o.Type().String())  // e.g. "blob"
    buf.WriteByte(' ')
    buf.WriteString(strconv.FormatInt(o.Size(), 10))
    buf.WriteByte(0)   // \x00 separator
    buf.Write(content)
    return buf.Bytes(), nil
}

Because it uses git's native loose-object encoding, Engram object keys are identical to the hashes in a real git repository — the same content produces the same SHA in any git implementation. This means standard git tooling can verify or export objects directly in the future.

SetEncodedObject (storage.go:77–93): compute hash → frame → objs.Put(h.String(), data). EncodedObject reverses it: objs.Get → unframe → return MemoryObject.

Materialize: check out a commit into a directory

internal/memstore/gitfs/repo.go:37–59: 

func Materialize(ctx context.Context, objs objstore.ObjStore, at string, dir string) error {
    st := NewStorage(ctx, objs)
    seedRefs(st, at)                     // fake refs/heads/main and HEAD in the in-memory Storage
    repo, _ := git.Open(st, osfs.New(dir))
    wt, _ := repo.Worktree()
    wt.Checkout(&git.CheckoutOptions{Hash: plumbing.NewHash(at), Force: true})
}

seedRefs sets a temporary refs/heads/main on the in-memory Storage pointing at the target commit, so go-git can open the repository normally — without touching any on-disk .git directory.

Commit: snapshot a directory into ObjStore

repo.go:63–96: 

func Commit(ctx context.Context, objs objstore.ObjStore, parent string, dir string, msg string) (string, error) {
    st := NewStorage(ctx, objs)
    // parent == "" → git.Init (first commit); otherwise seedRefs + git.Open
    wt.AddWithOptions(&git.AddOptions{All: true})
    h, _ := wt.Commit(msg, &git.CommitOptions{Author: author(), AllowEmptyCommits: true})
    return h.String(), nil  // hex commit SHA
}

Inside wt.Commit, go-git calls st.SetEncodedObject for every new blob, tree, and commit object, which ultimately lands each one via objs.Put. The return value is the new commit's SHA, which CommitWithCAS then passes to the Postgres CAS.

TreeKeys: extracting immutable keys for the L3 cache

repo.go:98–123 implements TreeKeys: it parses the commit object directly from ObjStore, extracting the rootTree hash and the system/ subtree hash, with no Postgres query required. Both SHAs are purely immutable — any file change produces a new rootTree; the system/ subtree SHA only changes when content inside system/ changes — making them ideal never-expire cache keys for the L3 cache (see ② Request Path).

3 · The single mutable pointer + compare-and-swap

The only pointer in the entire system that is permitted to change is the agent_id → HEAD row in Postgres. All concurrency control converges on a single atomic CAS against this pointer.

agent_refs table

-- 000001_init.up.sql
CREATE TABLE agent_refs (
  agent_id   text PRIMARY KEY,
  head       text NOT NULL,
  updated_at timestamptz NOT NULL DEFAULT now()
);

CommitRef: what the CAS SQL looks like

internal/memstore/refs/refs.go:259–265: 

ct, err := tx.Exec(ctx,
    `UPDATE agent_refs SET head=$1, updated_at=now()
     WHERE agent_id=$2 AND head=$3`,
    next, agentID, parent)
if ct.RowsAffected() == 0 {
    return ErrCASConflict
}

The WHERE head=$3 clause is the compare step: the UPDATE only matches a row (giving RowsAffected() == 1) when the HEAD stored in the database matches the parent the caller holds. If another writer has already committed and advanced HEAD, RowsAffected() == 0 and the function returns ErrCASConflict; the caller must re-read the latest HEAD and retry.

Invariant All concurrency control converges on this one CAS. There is no optimistic multi-writer merge, no distributed lock, no CRDT. The Session Router ensures at most one writer per agent at a time upstream; the CAS is the last line of defence against any unexpected race. Calling CommitRef for an agent that does not yet exist also returns ErrCASConflict (0 rows updated), so callers must Bootstrap first.

ResolveHead and Bootstrap

ResolveHead (refs.go:36–46): SELECT head FROM agent_refs WHERE agent_id=$1; a missing row returns ErrAgentNotFound.

Bootstrap (refs.go:49–57): INSERT … ON CONFLICT (agent_id) DO NOTHING — idempotently sets the initial HEAD.

4 · A commit is a single Postgres transaction

CommitWithCAS (memstore.go:60–69) splits the write into two strictly ordered phases:

  1. Objects first (write ObjStore) Call gitfs.Commit, writing all new blob/tree/commit objects into ObjStore. Writes are idempotent and safe to retry. A crash here leaves only unreachable orphan objects; GC cleans them up after the grace period.
  2. Atomically advance the ref (write Postgres) Inside a single Postgres transaction: CAS UPDATE agent_refs and INSERT INTO memory_jobs (reindex, reflect). Either both succeed or both roll back — there is no half-committed state where the ref has advanced but jobs were not enqueued.
Design win Same-tx enqueue replaces the Kafka + outbox pattern. Job enqueue and ref advance happen in the same Postgres transaction, which naturally guarantees "job is enqueued if and only if the commit was committed" — no separate message broker required, no outbox polling.

Crash-safety matrix

Crash timingConsequenceRecovery
Crash during object writes Some objects orphaned in ObjStore; ref not updated; no jobs enqueued GC reclaims orphans after the grace period; agent state is unaffected
Crash after BEGIN, before COMMIT Postgres auto-rolls back; ref and job queue unchanged Caller receives an error and retries the whole commit
Crash after COMMIT Ref advanced, jobs enqueued, new objects reachable Nothing to recover; maintenance worker drains jobs normally

5 · The memory_jobs queue table

memory_jobs is a Postgres table acting as Engram's maintenance job queue — eliminating the need for a separate message broker.

Schema and key design decisions

-- 000001_init.up.sql
CREATE TABLE memory_jobs (
  id         bigserial PRIMARY KEY,
  agent_id   text NOT NULL,
  kind       text NOT NULL,     -- "reindex" | "reflect" | "defrag" | "gc"
  from_sha   text,              -- commit SHA that triggered this job (for incremental processing)
  state      text NOT NULL DEFAULT 'pending',  -- pending | running | failed
  attempts   int  NOT NULL DEFAULT 0,
  created_at timestamptz NOT NULL DEFAULT now()
);

-- per-agent singleton: at most one pending row per (agent_id, kind)
CREATE UNIQUE INDEX memory_jobs_pending_uniq
  ON memory_jobs (agent_id, kind) WHERE state = 'pending';

-- 000002_claimed_at.up.sql  (second migration)
ALTER TABLE memory_jobs ADD COLUMN claimed_at timestamptz;

Three key design choices:

  • Partial unique index (WHERE state = 'pending'): at most one pending row per agent per kind. Duplicate INSERTs from a commit retry become idempotent no-ops.
  • attempts: retry counter. RetryJob increments it on each failure; once it reaches maxAttempts the state is set to 'failed', preventing a poison job from crash-looping the worker.
  • claimed_at (added in the second migration): timestamp recorded when a worker claims a job. ReapStaleJobs uses it to detect a crashed worker — any row with state='running' and claimed_at < before is safely returned to the queue.

CommitRef enqueue logic

refs.go:268–275, inside the same transaction that succeeded the CAS: 

for _, j := range jobs {
    tx.Exec(ctx,
        `INSERT INTO memory_jobs (agent_id, kind, from_sha) VALUES ($1,$2,$3)
         ON CONFLICT DO NOTHING`,
        agentID, j.Kind, next)
}

A typical jobs slice contains two entries: {Kind:"reindex"} and {Kind:"reflect"}.

6 · WithGlobalLock: Postgres advisory lock

Some maintenance tasks (global GC, per-agent singleton guarantee) need to ensure only one worker is executing at a time. Engram implements this via a Postgres session-level advisory lock, with no separate distributed-lock service required.

Implementation

internal/memstore/refs/refs.go:84–109: 

func (r *Refs) WithGlobalLock(ctx context.Context, key int64, fn func(context.Context) error) (bool, error) {
    conn, err := r.pool.Acquire(ctx)  // acquire a connection from the pool and hold it
    defer conn.Release()

    var got bool
    conn.QueryRow(ctx, `SELECT pg_try_advisory_lock($1)`, key).Scan(&got)
    if !got {
        return false, nil  // lock held by another session → skip this round
    }

    // unlock must use context.Background(), not the request ctx
    defer func() {
        _, _ = conn.Exec(context.Background(), `SELECT pg_advisory_unlock($1)`, key)
    }()

    return true, fn(ctx)
}
Gotcha (real bug) Unlock must use context.Background(), not the incoming request ctx.

Advisory locks are session-scoped: they are bound to the connection, which is returned to the pool after conn.Release() and may be reused by the next request. If the unlock statement runs with an already-cancelled request ctx, pg_advisory_unlock never executes, and the lock leaks onto the connection. Every subsequent pg_try_advisory_lock call with the same key returns false permanently — until that connection is recycled by the pool. This was a real bug caught during code review and has been fixed.

Current call sites:

  • Global GC: ensures only one worker runs the mark-sweep at a time (fixed key).
  • Per-agent maintenance (reflection, reindex): called with a different key per agent, providing a per-agent singleton.

7 · GC: reachability mark-sweep + grace period

GC is a pure function (internal/maintenance/gc.go) with the signature: 

func GC(ctx context.Context, objs objstore.ObjStore,
        reachable map[string]struct{}, grace time.Duration, now time.Time) (Stats, error)

It does not query Postgres or look at refs — the caller is responsible for pre-computing the reachable set and passing it in. This design makes GC fully unit-testable with an in-memory ObjStore.

Step 1: compute the reachable closure (ReachableObjects)

internal/memstore/gitfs/reach.go:18–53 starts an iterative DFS from all agent HEADs, using three dedup maps to avoid revisiting nodes: 

func ReachableObjects(ctx context.Context, objs objstore.ObjStore, heads []string) (map[string]struct{}, error) {
    seenCommits := map[string]struct{}{}
    seenTrees   := map[string]struct{}{}
    stack := append([]string{}, heads...)

    for len(stack) > 0 {
        h := stack[len(stack)-1]; stack = stack[:len(stack)-1]
        if _, ok := seenCommits[h]; ok { continue }
        seenCommits[h] = struct{}{}

        c, err := object.GetCommit(st, plumbing.NewHash(h))
        if err != nil {
            return nil, fmt.Errorf("gitfs: reach commit %s: %w", h, err) // failure → nil
        }
        reachable[h] = struct{}{}
        addTree(reachable, seenTrees, st, tree)    // expand the whole tree
        for _, p := range c.ParentHashes {
            stack = append(stack, p.String())       // preserve full history
        }
    }
    return reachable, nil
}

seenTrees matters because multiple commits frequently share the same subtree (common for small changes); without it, every shared subtree would be expanded once per commit that references it. Any error returns nil, err — the caller (GC runner) must detect this and abort; it must never pass an incomplete nil map into sweep.

Step 2: sweep (evaluate each key)

gc.go:32–68. Delete condition: unreachable AND past the grace period: 

// snapshot keys first, then iterate (cannot Delete while ranging over Iter)
objs.Iter(ctx, func(k string) error { keys = append(keys, k); return nil })

for _, k := range keys {
    if _, ok := reachable[k]; ok { s.Kept++; continue }  // reachable → skip

    mt, err := objs.Stat(ctx, k)
    if errors.Is(err, objstore.ErrNotFound) { continue }  // concurrent delete, skip
    if err != nil { s.StatErrors++; continue }             // Stat failure → conservatively keep

    if now.Sub(mt) > grace {
        if err := objs.Delete(ctx, k); err != nil {
            s.DelErrors++; continue   // Delete failure → best-effort, do not abort
        }
        s.Swept++
    } else {
        s.Kept++  // unreachable but within grace period → keep (in-flight write protection)
    }
}

Why the grace period matters

Recall the two-phase commit: objects are written to ObjStore first, and the Postgres ref is advanced only afterward. There is a brief window where an object exists in ObjStore but has not yet been made reachable by a ref update, so ReachableObjects (walking from all HEADs) cannot see these new objects yet. The grace period (default 1 hour) ensures that "young" objects are never prematurely deleted — they will be referenced by the ref shortly.

GC flow diagram

Fig 4 · GC flow. If the reachable-set computation fails, GC aborts immediately without sweeping; unreachable objects within the grace period are conservatively retained.

Invariant Never sweep with an incomplete reachable set. ReachableObjects returns nil, err on any error. The GC runner must detect a non-nil err and abort — it must not call GC(). This prevents the disaster of "traversal interrupted → live objects mistaken for garbage → deleted."
Scope GC operates only on the git object store and never touches the search index (embedding store). The search index has its own independent lifecycle management (see ④ Search & Index). Keeping them decoupled avoids cross-store reachability complexity.

Stats struct

type Stats struct {
    Scanned    int  // total keys snapshotted by Iter
    Swept      int  // unreachable+expired objects successfully deleted
    Kept       int  // reachable or within grace period (conservatively retained)
    StatErrors int  // Stat failures → conservatively kept, not counted in Kept
    DelErrors  int  // Delete failures → best-effort keep, does not abort GC
}

A Delete failure on a single object does not abort the whole GC run (best-effort); it is counted in DelErrors for worker logs and alerting. GC only returns a top-level error if Iter itself fails.

8 · Migrations: expand-contract + compile-time embed

The database schema is managed by golang-migrate, with migration files embedded into the binary at compile time via //go:embed (refs/migrate.go:14): 

//go:embed all:migrations
var migrationsFS embed.FS

Migrate(dsn string) is idempotent — safe to call multiple times, suitable for running unconditionally at production startup. The expand-contract principle ensures each migration step is forward-compatible (add new columns/indexes before dropping old structures).
