Architecture Overview

Engram is a "cloud-side notebook" for agents — it stores memory using the git object model, guarantees consistency with a single small lock in Postgres, and scales horizontally with stateless workers. This page is the global map; subsystem details live in Request Path, Storage & Consistency, Search & Index, Maintenance Pipeline.

1. What We Are Building

Every LLM conversation starts from a blank slate — the model has no idea what you said last time, no memory of what it wrote before. That is exactly what Engram is here to fix: give each agent a memory repository that it can read and write across sessions, with full version history, multi-tenant isolation, deployed in the cloud, and served on demand by stateless workers.

The starting point is MemFS (the git-backed local memory filesystem from the Letta project), but we've moved it from "single-machine local" to "multi-tenant cloud." Each agent owns its own git repository — files are Markdown memory fragments, history is git commits, and content addressing is the SHA of git objects. The cloud-scale challenges are: many agents writing concurrently, pods crashing at any moment, stateless workers, and no single-point bottleneck.

Core Philosophy — Memory Is Fetched on Demand, Not Stuffed Into the Model

Before each inference turn, only system/ (resident key facts) and a lightweight tree index (filenames + description fields) are loaded into context. The long tail — memories that exist but are not frequently needed — is pulled on demand by the model mid-loop via the recall / read tools. There is no automatic top-k dump, no pre-loading of "potentially useful" content into the prompt. This principle dramatically reduces per-turn token cost while preserving full memory reachability.

The overall design is deliberately lightweight: stateless compute + content-addressed object storage + a single Postgres as the coordination hub, with no Temporal or Kafka (see §5 for why). Minimal infrastructure: one object-storage bucket + Postgres + pods with no local persistence.

2. The Four Zones

The architecture is logically divided into four zones (architecture.md §3). Read paths flow top-down through the acceleration layers; the write path (commit) goes directly to canonical storage; maintenance is fully async and never on the critical path.

Four-zone topology. Solid red lines = write path (objects first, then CAS); dashed green lines = read path (cache/index first); dashed orange lines = async maintenance.

① Request Path

Session Router → Agent Worker → LLM

The Router serializes writes per agent_id (sticky routing or a per-agent lock). The Worker is a stateless pod: it mounts a working tree, assembles context, runs the tool loop, and commits. The LLM is pure inference, stateless.

→ Deep dive: Request Path.

② Read Accelerators

SHA Cache + Hybrid Search

The Memory Cache is keyed by the system/ subtree hash — no invalidation needed (the same SHA always maps to the same content). The Search Service holds full file contents locally (Variant A), so queries require zero object-storage access.

→ Deep dive: Search & Index.

③ Canonical Store

Object Storage + Postgres

Object storage holds content-addressed immutable objects (objects/<hash>). Postgres is the system's single strong-consistency coordination point: it owns the agent_id→HEAD CAS ref and the memory_jobs job queue.

→ Deep dive: Storage & Consistency.

④ Maintenance

Single Go Maintenance Worker

A separate process that runs periodically: GC (mark-and-sweep unreachable objects), Reflection (history consolidation), Defrag (split oversized files), Reindex (incremental index updates), Reaper (reclaim stalled jobs). Fully async — never blocks the foreground.

→ Deep dive: Maintenance Pipeline.

3. Three Rules That Run Through Everything

The three invariants below are the spine of the entire architecture (architecture.md §2, CLAUDE.md core invariants). They derive from one another: the first makes caching and GC possible; the second gives the whole system a single arbiter; the third lets you throw away and rebuild any derived view at any time.

Invariant ① — Every piece of content is immutable, stored by its fingerprint

File contents, tree objects, commit objects — all are keyed by the SHA of their own content and stored in object storage. Objects are never modified; only new objects are appended. Changing memory = producing new objects + a new commit.

Three things follow from this rule:

  • Cache entries never need invalidation. sha → materialized content is an immutable mapping. LRU eviction is about saving space, not freshness. New content = new SHA = new key = cache miss = re-populate — that's "natural invalidation."
  • Deduplication across versions. If the same content is written twice, only one object is stored; the object store deduplicates for free.
  • Mark-and-sweep GC is safe. The same hash can never point to two different pieces of content, so "objects not reachable from HEAD" are garbage and can be swept periodically without any risk of deleting live data.
Invariant ② — The only mutable thing is the HEAD pointer, protected by a single "check before you change" lock

The only mutable pointer in the system is agent_id → HEAD, stored in the agent_refs table in Postgres. All concurrency control, ordering guarantees, and consistency are funneled through this single CAS operation — the sole arbiter for the entire system.

The SQL primitive (architecture.md §9):

UPDATE agent_refs SET head=$new, updated_at=now()
WHERE agent_id=$id AND head=$expected_parent;

Zero rows returned → HEAD has already moved → caller reconciles and retries. This single serialization point replaces any external lock service. Object storage needs no conditional writes because it stores only immutable objects.

This maps exactly to lakeFS's design (architecture.md §15.1, validation 3-0): lakeFS stores committed metadata in object storage and protects branch pointers with a KV-store SetIf() CAS, for precisely the same reason — object stores have no conditional-write primitive.

Invariant ③ — Everything else is a disposable, recomputable derived view

Caches, the trigram index, the embedding vector index, working-tree copies — all are views derived from objects + refs, and therefore stateless and disposable.

  • Cache: Pod crashes → on restart, repopulate from object storage. Zero data loss.
  • Search index: Eventually consistent; memory written moments ago may not be searchable for a few seconds — acceptable, because the current turn's content is already in context.
  • Working tree: A temporary directory mounted on worker scratch / tmpfs, re-materializable from any SHA at any time.
Common Pitfall

Do not treat the cache, search index, or working tree as a source of truth. They are derived products — disposable. Objects + refs are authoritative (CLAUDE.md guardrails). Once you start treating derived views as the source of truth, you'll start worrying about "what if the cache is out of sync with storage" — and the correct answer is: when a derived view is inconsistent, throw it away and rebuild.

4. From One Request to One Commit

From a user message arriving, to an agent editing memory, committing, and triggering background maintenance — a full lifecycle walkthrough (architecture.md §8).

  1. Load resident facts + directory The Router serializes by agent_id and hands the request to a Worker. The Worker reads the current HEAD from Postgres and queries the Memory Cache with Get(agentID, systemTreeSHA). Hit → immediately returns the materialized system/ content + tree index (filenames, descriptions); miss → fetches objects from object storage and populates the cache. Context assembly: resident facts + tree index + user message, sent to the LLM.
  2. Agent loop — the model decides what to fetch The LLM receives its context and enters a tool-call loop. It may call recall(query) (sends a trigram+vector hybrid query to the Search Service, returns matching line-number ranges — not entire files), read(path) (reads a file from the working tree or object storage), or edit(path, content) (writes a modification into the working tree). All of this happens during inference; the model autonomously decides which memories to retrieve — no pre-stuffing of context.
  3. Commit — write objects first, then advance HEAD, enqueue jobs in the same transaction The Worker calls CommitWithCAS: ① write all new content from the working tree as content-addressed objects, idempotent PUT to object storage (this step can crash, leaving only garbage objects that GC will clean); ② in a single Postgres transaction, execute the CAS HEAD: parent → new + INSERT memory_jobs. Atomicity guarantees: job is enqueued if and only if the commit committed. CAS failure (HEAD already moved) → returns ErrCASConflict → caller reconciles and retries.
  4. Background digestion — async maintenance, never blocks the foreground The Maintenance Worker dequeues memory_jobs with FOR UPDATE SKIP LOCKED and runs Reflection (history consolidation), Defrag (split oversized files), GC (mark-and-sweep unreachable objects), and Reindex (incremental search-index update) in an isolated git worktree. This never interferes with SHA snapshots currently being read by foreground Workers. Timed-out jobs are reclaimed by the Reaper; poison jobs (those that repeatedly crash workers) enter a failed state after maxAttempts attempts (L5d).

Crash Safety of the Write Order

architecture.md §6.2 is explicit: write objects to object storage first (idempotent), then CAS in Postgres. Crashing before the CAS leaves only unreachable garbage objects, which GC cleans up after a grace window — it never produces a dangling ref pointing to an incomplete commit. The reverse order — CAS first, then write objects — would leave a HEAD pointing to objects that do not yet exist, which is a serious consistency flaw.

5. Key Trade-offs

Single-writer vs. optimistic multi-writer

Chosen: single writer per agent, Router serializes

The Router uses sticky routing or a per-agent lock to ensure only one Worker is writing for a given agent at any moment. Simple, correct, and well-matched to how agents access memory — sequentially, one turn at a time.

Rejected: optimistic multi-writer + automatic merge

Three-way merge works fine for structured data (JSON, code); it does not work for Markdown prose — any automatic merge is lossy, potentially stitching unrelated memory fragments in the wrong place or silently overwriting content.

Commit as a single database transaction, replacing heavy message middleware

Chosen: single Postgres transaction — CAS + enqueue

CAS and INSERT memory_jobs commit in the same transaction, guaranteeing atomically that "job is enqueued iff commit committed." No orphaned jobs, no commit/emit race.

Rejected: Kafka outbox / dual write

Write DB then write Kafka: if the Kafka write fails, the job is lost. Write Kafka first: if the DB write fails, the job is already enqueued but the commit never happened. Either order has an inconsistency window. Closing that window requires a complex outbox table.

Local search: queries never touch object storage

Chosen: Variant A — index stores full file content locally

Zero object-storage access at query time; low latency, no dependencies. Memory files are small Markdown; the content duplication is negligible. L4 layers embedding vectors on top of this, enabling trigram lexical + semantic hybrid recall.

Variant B — index stores only offsets + Range GET

Queries require Range GETs against object storage, adding cross-AZ network round-trips and potentially hitting object-storage rate limits under high concurrency. Only worth considering when memory files are large — not the case here.

Why not Temporal and Kafka (architecture.md §12)

Guardrail — Do not introduce Temporal or Kafka

Temporal solves "long-running, crash-recoverable, multi-step workflows." Our maintenance jobs are idempotent, re-derivable from state, and complete in seconds — "retry from scratch" is acceptable, so the overhead of durable mid-flight resume is pure waste.

Kafka solves "decoupling the write path, fanning out to multiple consumers." We have exactly one consumer (the Maintenance Worker), and the same-transaction Postgres enqueue provides that decoupling with zero dual-write risk.

Revisit only if jobs become genuinely long and complex (→ then Temporal) or consumers multiply and need replay (→ then Kafka). M5 plans to upgrade to River (same Postgres, richer retry semantics), but no need to cross over to Kafka.

7. Tech Stack

Language

Go

  • All I/O functions take context.Context as their first argument
  • Errors are wrapped with %w for full, traceable call chains
  • No global mutable state; small interfaces at package boundaries
  • Table-driven tests; Postgres-dependent tests auto-t.Skip when ENGRAM_TEST_DB is unset
Database

Postgres + pgx

  • agent_refs: refs CAS — the sole strong-consistency coordination point
  • memory_jobs: job queue, dequeued with FOR UPDATE SKIP LOCKED, deduplicated per agent+kind
  • maintenance_cursor: reconcile loop cursor
  • Migrations via golang-migrate, expand-contract strategy, never destructive in a single step
Object Storage

S3 / Local FS

  • Always hidden behind the ObjStore interface; upper layers swap backends transparently
  • Production: S3/OSS; development/tests: local FS backend (ENGRAM_OBJ)
  • Requires read-after-write strong consistency from the bucket (S3 satisfies this since 2020)
  • Embedding store is independent of the git object store; GC does not scan the former
Git Layer

go-git + custom Storer

  • go-git's Storer interface is backed by an ObjStore implementation
  • Services never shell out to the git binary — shelling out is unreliable and hard to test in a service
  • Working-tree materialization uses osfs; tests can swap in memfs, no disk required
  • Future path: fully in-memory billy Filesystem, making workers disk-free (§10.1, not yet implemented)

Core Interface Summary (architecture.md §11)

type MemStore interface {
    // ResolveHead reads the agent's current HEAD (cheap ref read, no object-storage access)
    ResolveHead(ctx context.Context, agentID string) (CommitHash, error)

    // Materialize checks out the tree at `at` into dir (a disposable working copy)
    Materialize(ctx context.Context, agentID string, at CommitHash, dir string) error

    // CommitWithCAS writes objects, builds a commit, atomically advances HEAD and
    // enqueues jobs in the same DB tx. Returns ErrCASConflict if HEAD has moved.
    CommitWithCAS(ctx context.Context, agentID string, parent CommitHash,
        dir string, jobs []Job) (CommitHash, error)
}

type SystemCache interface {
    Get(ctx context.Context, agentID string, sha CommitHash) (*Resident, error)
}

type Search interface {
    Recall(ctx context.Context, agentID, query string, k int) ([]Hit, error)
    Reindex(ctx context.Context, agentID string, from, to CommitHash) error
}
