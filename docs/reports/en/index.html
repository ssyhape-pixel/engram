Engram, in depth

How a "cloud-side memory system for AI agents" grows from nothing — from immutable blocks of storage all the way to the worker that quietly tidies the memory in the background.

Based on the main branch · all nine layers built and merged · six reports + this index

Picture every AI agent owning a notebook that's all its own. It can flip back to a few resident pages of essentials, fetch long-tail details only when it needs them, and durably write new things back so it still remembers next time it wakes up. Engram is the system that gives thousands of agents that notebook, on the server side.

Its approach is deliberately simple: every block you write is immutable (to change something, you write a new block), and the one pointer that does change — "which page is current" — lives in a database, guarded by a small "check-before-you-change" lock. Everything else — caches, the search index, working copies — is disposable and rebuildable at any time. That's how the system stays stateless and horizontally scalable while still being strongly consistent.

The soul of it in one line: memory is something the agent "goes and fetches", not something dumped on the model. Before it starts reasoning, it loads only the resident essentials and a thin index of what exists; the real details, the model decides for itself whether to go pull.

Status

The whole planned roadmap is done: 10 code modules, 2 runnable processes (one foreground, one background), nine layered merges. The full test suite passes (serialized) and the race detector is clean on the concurrent modules. The one piece deliberately deferred is vector-store eviction — vectors are derived, recomputable, and small, so there's no pressure yet.
