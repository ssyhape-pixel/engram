③ 存储与一致性

内容寻址对象 · gitfs · refs CAS · GC — Engram 持久层的核心机制

覆盖层级：L1 MemStore core · L2 agent loop · L3 SHA read cache · L5a GC · L5b maintenance jobs

Engram 的持久层由三个协作部分构成：一个纯粹不可变的内容寻址对象存储（objstore）、 一个将 git 作为工作格式的自定义 Storer（gitfs），以及存储在 Postgres 中的唯一可变指针（refs）。 三者共同保证了：对象不可变、历史可重放、并发写入不会产生裂变。

1 · 内容寻址对象模型

每一个存储单元——blob（文件内容）、tree（目录快照）、commit（版本快照）——都以其自身内容的 SHA-1 哈希作为键。 这个性质带来三个直接收益：

  • 按 SHA 缓存：相同的哈希永远对应相同的字节。缓存命中无需失效逻辑。
  • 写入天然去重：两个 agent 的 system/persona.md 若内容相同，存储中只保留一份 blob。
  • GC 可推导：从所有 agent HEAD 出发做可达性遍历，未被引用的对象即为垃圾，可安全回收。

ObjStore 接口

接口定义于 internal/memstore/objstore/objstore.go:16，只有六个方法： 

type ObjStore interface {
    Has(ctx context.Context, key string) (bool, error)
    Get(ctx context.Context, key string) ([]byte, error)
    Put(ctx context.Context, key string, data []byte) error // idempotent
    Iter(ctx context.Context, fn func(key string) error) error
    Stat(ctx context.Context, key string) (time.Time, error)
    Delete(ctx context.Context, key string) error
}

key 是对象的十六进制哈希字符串。Put 被显式标记为幂等：相同的 key 意味着完全相同的字节， 重复写入是无害的空操作。

Local 后端

开发/测试环境使用本地文件系统后端 internal/memstore/objstore/local.go。 对象按 <root>/objects/<key[:2]>/<key> 路径组织（两字符前缀分片，避免单目录过大， 与 .git/objects/ 布局一致）。

Put 使用先写临时文件再 os.Rename 的原子可见性模式 （local.go:53–80）： 

tmp, err := os.CreateTemp(filepath.Dir(p), ".tmp-*")
// ... write data ...
if err := os.Rename(tmpName, p); err != nil { ... }

这保证了并发写同一对象时，观察者只会看到完整的字节，永远不会读到半写状态。 由于对象不可变，最后一次 rename 胜出，结果始终正确。

2 · gitfs：go-git Storer over ObjStore

Engram 使用 git 作为工作格式——利用 git 的 blob/tree/commit 数据结构来表达内存快照—— 但从不 shell out 到 git 二进制，也不依赖 .git/ 目录文件。 所有 git I/O 都通过 go-git 完成， 并以自定义的 Storage 替换其默认的文件系统后端。

Storage：go-git 的自定义 Storer

internal/memstore/gitfs/storage.go 实现了 go-git 的 storage.Storer 接口，将对象层桥接到 ObjStore。

存储格式采用 git 的 loose-object 帧格式（storage.go:42–59）： 

// frame encodes: "<type> <size>\x00<content>"
// sha1(frame(o)) == o.Hash()  ← 这正是 git 的哈希定义
func frame(o plumbing.EncodedObject) ([]byte, error) { ... }

SetEncodedObject 计算哈希、帧化对象，然后调用 objs.Put （storage.go:77–93）； EncodedObject 则反向执行：objs.Get → unframe → 返回 MemoryObject。 引用、配置、index 等非对象数据仍委托给内嵌的 memory.Storage，它们是每 session 临时的， 权威引用始终在 Postgres refs 表中。

Materialize：将提交检出到目录

internal/memstore/gitfs/repo.go:39–58： 

func Materialize(ctx context.Context, objs objstore.ObjStore, at string, dir string) error {
    st := NewStorage(ctx, objs)
    seedRefs(st, at)                                    // 在内存中伪造 branch ref
    repo, _ := git.Open(st, osfs.New(dir))
    wt, _ := repo.Worktree()
    wt.Checkout(&git.CheckoutOptions{Hash: plumbing.NewHash(at), Force: true})
}

seedRefs 在 in-memory Storage 上设置一个临时 refs/heads/main 指向目标 commit， 使 go-git 能够正常打开仓库。整个过程不接触磁盘上的 .git 目录。

Commit：将目录快照写入 ObjStore

internal/memstore/gitfs/repo.go:63–96： 

func Commit(ctx context.Context, objs objstore.ObjStore, parent string, dir string, msg string) (string, error) {
    st := NewStorage(ctx, objs)
    // init (首次) 或 open (有 parent)
    wt.AddWithOptions(&git.AddOptions{All: true})
    h, _ := wt.Commit(msg, &git.CommitOptions{Author: author(), AllowEmptyCommits: true})
    return h.String(), nil
}

go-git 在 wt.Commit 内部调用 st.SetEncodedObject， 对每个新 blob、tree、commit 对象执行一次 objs.Put。 返回的十六进制 SHA 就是新 commit 的哈希，随后由 CommitWithCAS 用于 Postgres CAS。

TreeKeys：从对象中读取缓存键

repo.go:98–123 实现 TreeKeys，直接解析 ObjStore 中的 commit 对象，提取 rootTree 哈希与 system/ 子树哈希，无需查询 Postgres。 这两个值被 L3 缓存层用作不可变缓存键（详见报告 ②）。

3 · refs：唯一可变指针 + CAS

在整个系统中，唯一被允许变化的指针是 Postgres 中的 agent_id → HEAD 映射。 所有并发控制都收敛于对这个指针的一次原子 CAS。

agent_refs 表

CREATE TABLE agent_refs (
  agent_id   text PRIMARY KEY,
  head       text NOT NULL,
  updated_at timestamptz NOT NULL DEFAULT now()
);

CommitRef 的 CAS SQL

internal/memstore/refs/refs.go:259–267： 

ct, err := tx.Exec(ctx,
    `UPDATE agent_refs SET head=$1, updated_at=now()
     WHERE agent_id=$2 AND head=$3`,
    next, agentID, parent)
if ct.RowsAffected() == 0 {
    return ErrCASConflict
}

WHERE head=$3 是 compare 步骤：只有当 Postgres 中存储的 HEAD 与调用方持有的 parent 一致时， UPDATE 才会命中一行（RowsAffected() == 1）。如果另一个写者已先行提交， HEAD 已变更，RowsAffected() == 0，返回 ErrCASConflict， 调用方需重新读取最新 HEAD 后重试。

不变量 所有并发控制收敛于这一个 CAS。 系统无乐观多写者合并、无分布式锁、无 CRDT。单写者路由（Session Router）在上游确保同一个 agent 同时只有一个写者，CAS 作为最后一道防线捕获任何意外的竞争。

ResolveHead 与 Bootstrap

ResolveHead（refs.go:36–46）直接 SELECT head FROM agent_refs WHERE agent_id=$1； Bootstrap（refs.go:49–57）使用 INSERT ... ON CONFLICT (agent_id) DO NOTHING，幂等地设置初始 HEAD。 一个不存在的 agent 传给 CommitRef 时同样会因 0 rows affected 而返回 ErrCASConflict——调用方必须先 Bootstrap。

4 · 提交即一个 Postgres 事务

CommitWithCAS（memstore.go:60–69）将写操作分为严格有序的两阶段：

  1. 对象优先：调用 gitfs.Commit，将所有新 blob/tree/commit 对象写入 ObjStore。 这一步可重试、幂等，即便崩溃也只留下不可达的孤立对象（GC 会清理）。
  2. 原子推进 ref：在一个 Postgres 事务中执行 CAS 并入队维护任务。 要么整体成功，要么整体回滚——不存在"ref 已更新但任务未入队"的中间态。

写入序列（时序图）

sequenceDiagram
    participant AW as AgentWorker

    participant PG as Postgres

    AW->>OBJ: Put blob 1
    AW->>OBJ: Put blob 2
    AW->>OBJ: Put tree root

    Note over AW,OBJ: 对象写入完成，幂等可重试

    AW->>PG: BEGIN
    AW->>PG: UPDATE agent_refs SET head=C_new WHERE head=C_old
    AW->>PG: INSERT INTO memory_jobs (reindex)
    AW->>PG: INSERT INTO memory_jobs (reflect)
    AW->>PG: COMMIT
    Note over AW,PG: 原子提交，任务入队 iff ref 推进成功
设计亮点 同-tx 入队替代了 Kafka + outbox 模式。 任务入队与 ref 推进在同一个 Postgres 事务中，天然保证了"任务已入队 当且仅当 commit 已提交"， 无需额外的事务性消息队列，也无需 outbox 轮询。

崩溃安全性

崩溃时机后果修复
写对象期间崩溃 部分对象孤悬于 ObjStore，ref 未更新 GC 宽限期后自动回收（不影响任何 agent 状态）
BEGIN 后、COMMIT 前崩溃 Postgres 自动回滚；ref 未变更，任务未入队 调用方收到错误，重试整个 commit
COMMIT 后崩溃 ref 已推进，任务已入队，新对象可达 无需修复；维护任务将正常被消费

5 · memory_jobs 队列表 + 同-tx 入队

表结构

-- 000001_init.up.sql
CREATE TABLE memory_jobs (
  id         bigserial PRIMARY KEY,
  agent_id   text NOT NULL,
  kind       text NOT NULL,         -- "reindex" | "reflect" | "defrag" | "gc"
  from_sha   text,                  -- 触发该任务的 commit SHA
  state      text NOT NULL DEFAULT 'pending',  -- pending | running | failed
  attempts   int  NOT NULL DEFAULT 0,

-- per-agent singleton：同一 (agent_id, kind) 最多一条 pending 行
CREATE UNIQUE INDEX memory_jobs_pending_uniq
  ON memory_jobs (agent_id, kind) WHERE state = 'pending';

-- 000002_claimed_at.up.sql
ALTER TABLE memory_jobs ADD COLUMN claimed_at timestamptz;

CommitRef 的入队逻辑

refs.go:268–275，在 CAS 成功后的同一事务中： 

for _, j := range jobs {
    tx.Exec(ctx,
        `INSERT INTO memory_jobs (agent_id, kind, from_sha) VALUES ($1,$2,$3)
         ON CONFLICT DO NOTHING`,
        agentID, j.Kind, next)
}

ON CONFLICT DO NOTHING 与 partial unique index 配合，使重复入队（如 commit 重试）成为幂等空操作。 典型的 jobs 切片包含 {Kind: "reindex"} 与 {Kind: "reflect"}。

任务的消费逻辑（ClaimJob、CompleteJob、RetryJob） 以及 drain 主循环详见⑤ 维护管线。

6 · WithGlobalLock：advisory lock

部分维护任务（尤其是全局 GC）需要在全集群范围内只有一个 worker 同时执行。 Engram 通过 Postgres advisory lock 实现这一点， 无需独立的分布式锁服务。

实现

refs.go:84–109： 

func (r *Refs) WithGlobalLock(ctx context.Context, key int64, fn func(context.Context) error) (bool, error) {
    conn, err := r.pool.Acquire(ctx)   // 从连接池取出一个连接并持有
    defer conn.Release()

    var got bool
    conn.QueryRow(ctx, `SELECT pg_try_advisory_lock($1)`, key).Scan(&got)
    if !got {
        return false, nil              // 锁已被其他 session 持有，跳过
    }

    // 解锁必须用 context.Background()，不能用请求 ctx
    defer func() {
        _, _ = conn.Exec(context.Background(), `SELECT pg_advisory_unlock($1)`, key)
    }()

    return true, fn(ctx)
}
坑点（真实 Bug） 解锁必须使用 context.Background()，而不是传入的请求 ctx。

Advisory lock 是 session 级的：它绑定在连接上，而连接在 conn.Release() 后 归还连接池，可被下一个请求复用。若解锁语句使用了已取消的请求 ctx， pg_advisory_unlock 将不会执行，锁泄漏到连接池的复用连接上， 导致此后所有尝试获取同一 key 的请求永远返回 ran=false——直到该连接被池回收。 这是代码审查阶段捕获的真实 Bug。

WithGlobalLock 当前被两处调用：

  • 全局 GC 扫描：保证同一时刻只有一个 worker 在执行 mark-sweep。
  • per-agent 维护（reflection、reindex）也可借助此机制实现 singleton 保证（传入不同的 key）。

7 · GC：可达性 mark-sweep + 龄期宽限

GC 是一个纯函数（internal/maintenance/gc.go）， 接受 ObjStore、一个预计算的可达集合 map[string]struct{}、宽限期 grace、 以及当前时间 now，不依赖任何全局状态，因此完全可用 in-memory store 单元测试。

可达对象计算（ReachableObjects）

internal/memstore/gitfs/reach.go:18–53 实现从所有 agent HEAD 出发的迭代式 BFS/DFS 遍历： 

func ReachableObjects(ctx context.Context, objs objstore.ObjStore, heads []string) (map[string]struct{}, error) {
    // seenCommits 防止重复遍历历史分支中的公共祖先
    // seenTrees   防止跨 commit 重复展开同一棵子树
    stack := append([]string{}, heads...)
    for len(stack) > 0 {
        h := stack[len(stack)-1]; stack = stack[:len(stack)-1]
        // ... 取出 commit → 加入 reachable
        // ... 展开 tree (addTree，递归子树，seenTrees 去重)
        // ... 将 parentHashes 推入 stack (保留完整历史)
    }
    return reachable, nil  // 错误时返回 nil — 调用方 MUST NOT sweep
}

addTree（reach.go:57–77）递归展开 tree 的所有 entries， 对 filemode.Dir 类型的 entry 继续递归。seenTrees 去重确保多个 commit 共享同一子树时 只遍历一次，这在大量小文件变更（只有少数 blob 不同）的场景下节省大量遍历开销。

sweep 阶段

gc.go:32–68，删除条件为 不可达 AND 超过宽限期： 

func GC(..., reachable map[string]struct{}, grace time.Duration, now time.Time) (Stats, error) {
    // 快照 keys（删除期间不能边遍历边修改）
    objs.Iter(ctx, func(k string) error { keys = append(keys, k); return nil })

    for _, k := range keys {
        if _, ok := reachable[k]; ok { s.Kept++; continue }  // 可达，保留
        mt, err := objs.Stat(ctx, k)
        if now.Sub(mt) > grace {
            objs.Delete(ctx, k)   // 不可达 且 超龄 → 删除
            s.Swept++
        } else {
            s.Kept++              // 不可达 但 在宽限期内 → 保留（可能是 in-flight 写入）
        }
    }
}

宽限期的作用

回顾提交的两阶段：对象先写入 ObjStore，然后才在 Postgres 中推进 ref。 存在一个短暂的时间窗口：对象已存在于 ObjStore，但 ref 尚未更新， 因此从所有 HEAD 出发的可达性遍历看不到这些新对象。 若 GC 在此窗口内运行，宽限期（默认 1 小时）确保这些"年轻"对象不会被误删。

不变量 永远不要对部分计算的可达集合执行 sweep。 ReachableObjects 遇到任何错误（包括对象读取失败）都返回 nil, err， 调用方（GC runner）检测到非 nil error 后必须中止，不得调用 GC()。 这防止了因遍历中断导致的"误判为垃圾"而删除存活对象的灾难性错误。
设计说明 GC 只作用于 git 对象存储，从不触及 embedding 存储。 搜索索引（L4）有独立的生命周期管理（详见④ 搜索与索引）。 两者解耦，避免了跨存储的可达性判断复杂度。

Stats 结构

type Stats struct {
    Scanned    int
    Swept      int
    Kept       int      // 可达 或 在宽限期内
    StatErrors int      // Stat 失败 → 保守保留
    DelErrors  int      // Delete 失败 → 最佳努力保留
}

单个对象的 Delete 失败不会中止整个 GC（best-effort），但会记入 DelErrors 供 worker 日志/告警使用。GC 仅在 Iter 失败时才返回顶层 error。

8 · 迁移：golang-migrate + embed

数据库 schema 通过 golang-migrate 管理，迁移文件内嵌进二进制（refs/migrate.go:14）： 

//go:embed all:migrations
var migrationsFS embed.FS

Migrate(dsn string) 幂等——多次调用安全，生产启动时可直接调用。 expand-contract 原则确保每一步迁移都是向前兼容的（先加新列/索引，再删旧结构）。

迁移版本变更内容文件
000001_init 创建 agent_refs、memory_jobs（含 partial unique index）、 maintenance_cursor migrations/000001_init.up.sql
000002_claimed_at 向 memory_jobs 添加 claimed_at timestamptz， 支持 ReapStaleJobs 的超时判断 migrations/000002_claimed_at.up.sql

claimed_at 字段是 ReapStaleJobs（refs.go:181–193） 的依据：凡 state='running' 且 claimed_at < before（或 IS NULL）的行， 说明其 worker 已经崩溃，安全地将其 attempts 加一、state 归 pending（或 failed），以便重新消费。
