⑤ 维护管线

一个后台 Go worker，在前台请求路径之外运行——GC、反射整合、碎片整理、失效 job 回收—— 全部幂等、可重推导，无需 Temporal 或 Kafka。

1. 维护 worker 概览

Engram 的维护管线是一个单进程 Go worker（cmd/maintenance/main.go）， 与请求路径完全分离。它的职责是：

  • GC：mark-sweep 清除不可达且超出宽限期的 content-addressed 对象（L5a）；
  • Reaper：将因 worker 崩溃而停留在 running 状态的 job 回收到 pending（L5d）；
  • Defrag 扫描：遍历所有 agent，为包含可拆分 .md 文件的 agent 幂等入队 defrag job（L5c）；
  • DrainJobs：依次消费 memory_jobs 中的 pending 条目，派发 reflect / reindex / defrag（L5b / L5c）。

每一项操作都是幂等的、可重推导的：GC 只删除已确认不可达的对象； reflect / defrag 通过 CAS 写入，冲突则重试；reindex 纯粹填充 embedding 缓存，无副作用。 这就是为什么不需要 Temporal 或 Kafka——job 不携带不可再生的中间状态，崩溃重启即可重做。

单 round 流程

flowchart TD
    TIMER([定时器到期]) --> LOCK[WithGlobalLock gcLockKey=1]
    LOCK -->|未获锁| SKIP[跳过本轮]
    LOCK -->|获锁| GC[GC mark-sweep\n清除不可达对象]
    GC --> REAP[ReapStaleJobs\n回收 stale running job]
    REAP --> SCAN[EnqueueDefrag\n扫描全部 agent 幂等入队]
    SCAN --> DRAIN[DrainJobs\n逐条消费 pending job]
    DRAIN --> UNLOCK[释放全局锁]
    UNLOCK --> SLEEP[Sleep ENGRAM_GC_INTERVAL]
    SLEEP --> TIMER

cmd/maintenance/main.go:148–185

整个 round 包裹在一个 Postgres session 级 advisory lock（gcLockKey=1）之内， 保证在任何时刻最多只有一个 worker 实例在执行维护，水平扩展时自动降级为主备。

不变量 维护 worker 运行在请求路径之外，永不阻塞前台 agent 会话。 所有写入均通过与前台相同的 CommitWithCAS 接口进行，确保单一一致性控制点。

2. memory_jobs 状态机

表结构与关键列

memory_jobs 是 Postgres 中驱动整个维护管线的任务队列。其核心列为：

列类型含义
idbigserial PK全局唯一 job ID
agent_idtext归属 agent
kindtextreflect / reindex / defrag
from_shatext nullable入队时的 HEAD commit SHA（供 diff-since 用）
statetextpending / running / failed
attemptsint default 0已尝试次数；达到 maxAttempts 转为 failed
claimed_attimestamptz nullable最近一次被 claim 的时间；Reaper 据此判断是否 stale

入队来源有两类：

  • 同事务入队（commit 路径）：CommitRef 在同一 Postgres tx 里推进 HEAD 并插入 reflect/reindex job， 保证 job 当且仅当 commit 成功时存在（替代 Kafka + outbox 模式）。
  • 带外入队（defrag 扫描）：EnqueueJob 通过 partial-unique index (agent_id, kind) WHERE state='pending' 做 ON CONFLICT DO NOTHING，同一 agent 的同类 pending job 永远只有一行。

状态转换

stateDiagram-v2
    [*] --> pending : CommitRef 或 EnqueueJob 入队
    pending --> running : ClaimJob FOR UPDATE SKIP LOCKED
    running --> done : CompleteJob 删除行
    done --> [*]
    running --> pending : RetryJob attempts 未达上限
    running --> failed : RetryJob attempts 达到 maxAttempts
    running --> pending : ReapStaleJobs claimed_at 超时
    running --> failed : ReapStaleJobs attempts 达到 maxAttempts
    failed --> [*] : 不再重试 留存审计
设计亮点 完成（done）状态即行删除，不留墓碑——队列只包含活跃工作，历史完成量对表大小无影响。 failed 行留存供审计排查（不会被 Drain 或 Reaper 二次处理）。

3. SKIP LOCKED 消费

ClaimJob（refs/refs.go:122）用一个两步事务原子地"抢占"一行 pending job： 

// ClaimJob atomically claims one pending job (FOR UPDATE SKIP LOCKED, no ORDER
// BY — queue hygiene) and marks it 'running'. Returns nil if none pending.
func (r *Refs) ClaimJob(ctx context.Context) (*DequeuedJob, error) {
    tx, err := r.pool.Begin(ctx)
    ...
    err = tx.QueryRow(ctx,
        `SELECT id, agent_id, kind, from_sha, attempts FROM memory_jobs
         WHERE state='pending' FOR UPDATE SKIP LOCKED LIMIT 1`).
        Scan(&j.ID, &j.AgentID, &j.Kind, &fromSHA, &j.Attempts)
    ...
    tx.Exec(ctx,
        `UPDATE memory_jobs SET state='running', claimed_at=now() WHERE id=$1`, j.ID)
    tx.Commit(ctx)
    return &j, nil
}

internal/memstore/refs/refs.go:122–151

几个关键设计点：

  • FOR UPDATE SKIP LOCKED：跳过被其他事务锁定的行，天然支持多 worker（尽管当前单 worker）， 避免锁等待导致 claim 挂起。
  • 无 ORDER BY：队列公平性（所谓 queue hygiene）——不对 pending job 排序， 防止因某类 job 长期失败而饥饿其他 job。
  • LIMIT 1：每次只取一行，保持细粒度进度与错误边界。
  • claimed_at=now()：标记声领时间，供 Reaper 后续判断是否超时。

CompleteJob 直接删除行（DELETE FROM memory_jobs WHERE id=$1）； RetryJob 执行 attempts++ 并根据是否达到 maxAttempts 将 state 设为 pending 或 failed， 同时清空 claimed_at： 

UPDATE memory_jobs
SET attempts = attempts + 1,
    state = CASE WHEN attempts + 1 >= $2 THEN 'failed' ELSE 'pending' END,
    claimed_at = NULL
WHERE id=$1

internal/memstore/refs/refs.go:164–174

4. Deps 派发与 DrainJobs

Deps 结构体

Deps 是 DrainJobs 的依赖容器，按 job kind 分离关注点： 

type Deps struct {

    Completer      Completer       // reflect
    Emb            search.Embedder // reindex
    EmbCache       cache.Cache     // reindex
    DefragMaxBytes int             // defrag
}

internal/maintenance/drain.go:15–21

这样 processJob 的每个分支只取用自己需要的字段，且各 handler 依赖独立可 mock。

processJob 派发

func processJob(ctx context.Context, r *refs.Refs, deps Deps,
    job *refs.DequeuedJob, maxAttempts int) error {
    switch job.Kind {
    case "reflect":
        ran, err := r.WithGlobalLock(ctx, agentKey(job.AgentID), func(ctx context.Context) error {
            return Reflect(ctx, deps.Store, deps.Completer, job.AgentID, job.FromSHA)
        })
        if !ran || err != nil {
            return r.RetryJob(ctx, job.ID, maxAttempts)
        }
        return r.CompleteJob(ctx, job.ID)
    case "reindex":
        if err := Reindex(ctx, deps.Store, deps.Emb, deps.EmbCache, job.AgentID); err != nil {
            return r.RetryJob(ctx, job.ID, maxAttempts)
        }
        return r.CompleteJob(ctx, job.ID)
    case "defrag":
        ran, err := r.WithGlobalLock(ctx, agentKey(job.AgentID), func(ctx context.Context) error {
            return Defrag(ctx, deps.Store, job.AgentID, deps.DefragMaxBytes)
        })
        if !ran || err != nil {
            return r.RetryJob(ctx, job.ID, maxAttempts)
        }
        return r.CompleteJob(ctx, job.ID)
    default:
        // Unknown kind: discard so it can't clog the queue.
        return r.CompleteJob(ctx, job.ID)
    }
}

internal/maintenance/drain.go:34–61

三条核心规则贯穿所有分支：

  1. 总是解决 job：无论成功或失败，processJob 一定调用 CompleteJob 或 RetryJob， job 行绝不会永久停留在 running。
  2. 未知 kind 直接 Complete：unknown job 不重试，避免僵尸 job 堵塞队列。
  3. 未获到 per-agent lock（ran=false）视为失败：退回 RetryJob，等下一轮再试。

DrainJobs 与 seen-set 防忙循环

func DrainJobs(ctx context.Context, r *refs.Refs, deps Deps, maxAttempts int) (int, error) {
    seen := map[int64]struct{}{}
    processed := 0
    for {
        job, err := r.ClaimJob(ctx)
        ...
        if job == nil {
            return processed, nil   // 队列已空
        }
        if _, dup := seen[job.ID]; dup {
            // 同一 round 内再次见到同一 job —— 它被 requeue 了
            // 再 Retry 一次（bumps attempts）然后结束本轮
            if err := r.RetryJob(ctx, job.ID, maxAttempts); err != nil { ... }
            return processed, nil
        }
        seen[job.ID] = struct{}{}
        if err := processJob(ctx, r, deps, job, maxAttempts); err != nil { ... }
        processed++
    }
}

internal/maintenance/drain.go:68–91

seen 集合记录本 round 内已处理过的 job ID。如果一个 job 被 RetryJob 回到 pending 并被同 round 的下一次 ClaimJob 再次取到， 则立刻再 RetryJob（第二次 bump attempts）并终止本轮循环—— 防止单个反复失败的 job 导致本 round 无限消耗资源，并借助 maxAttempts 在若干 round 内最终使其进入 failed。

注意 seen-set 只在一个 DrainJobs 调用（一轮）内有效，不跨 round 持久化。 每轮 drain 前 GC 和 Reap 已先行执行，stale running job 已被回收，所以 drain 开始时 seen 集合总是干净的。

5. per-agent advisory lock 与防自触发循环

agentKey：FNV 哈希到 advisory lock key

func agentKey(agentID string) int64 {
    h := fnv.New64a()
    _, _ = h.Write([]byte(agentID))
    return int64(h.Sum64())
}

internal/maintenance/drain.go:25–30

每个 agent 的 reflect 和 defrag job 都在 WithGlobalLock(ctx, agentKey(agentID), ...) 下运行。 这是维护侧的单例锁，防止同 agent 的多个 job（例如同 round 内有多个 reflect job 时）并发写入。

哈希碰撞只会让两个 agent 偶尔串行化，不影响正确性——advisory lock 是"尽力而为"的串行化， 真正的正确性由 CommitWithCAS 的 CAS 保障。

jobs=nil：切断自触发回路

reflect 和 defrag 在写回时均调用： 

// jobs=nil: reflection never enqueues a reflect job (no self-trigger loop).
if _, err := store.CommitWithCAS(ctx, agentID, head, dir, nil); err != nil { ... }

internal/maintenance/reflect.go:72  |  internal/maintenance/defrag.go:159

CommitWithCAS 接受 jobs []refs.Job 参数，传入 nil 表示本次提交不入队任何维护 job。 这与前台 agent 提交（传入 reflect/reindex）形成对比： 维护提交永远不会触发新的维护 job，从根本上断开反馈回路。

不变量 maintenance commit 使用 jobs=nil，因此不会自我触发 reflect 或 defrag job。 这是防止维护管线进入无限自增长循环的结构性保障，而非依赖运行时检查。

CAS 冲突 → ErrConflict → 重试

如果 agent 在维护写入期间推进了 HEAD（前台并发写），CommitWithCAS 的 CAS 会失败， reflect.go 和 defrag.go 均将其转化为 ErrConflict， processJob 随即调用 RetryJob 将 job 还回 pending。 下一轮 Drain 时 Reflect 会重新 ResolveHead，看到最新状态再做整合—— 绝不做有损的强制覆盖。

Reflect 执行流程

Reflect（reflect.go:38）的完整路径：

  1. ResolveHead：取当前 HEAD（忽略 fromSHA，总是看最新状态；fromSHA 保留供未来 diff-since 使用）。
  2. Materialize：将 HEAD tree 物化到临时目录。
  3. readSystemDir：遍历 system/ 子目录，将所有文件拼接为带路径标题的字符串。 若 system/ 不存在，返回 ("", nil)。
  4. 空常驻集跳过：if resident == "" { return nil }——当 system/ 为空时不提交任何内容，避免产生无意义的 commit churn。
  5. Complete：调用 Completer，传入 reflectSystemPrompt + 居民内容，获得整合摘要。
  6. 写入 system/reflection.md：将 LLM 输出写到物化目录中。
  7. CommitWithCAS(jobs=nil)：CAS 提交，冲突返回 ErrConflict。

readSystemDir 错误传播

fs.ErrNotExist（system/ 目录尚未创建）通过 SkipAll 静默处理， WalkDir 返回 nil，readSystemDir 返回 ""，上层跳过本次整合。 其他读取错误（权限问题、IO 故障）则向上传播，最终触发 RetryJob。

设计亮点 Reflect 总是基于最新 HEAD（不是 fromSHA 快照），确保整合结果不会基于过时的居民集。 fromSHA 字段已保留，为未来 diff-since-last-reflect 优化提供钩子， 届时可以只把变化部分送给 LLM 而非全量重读。

7. Defrag（碎片整理）

可拆分性判断

一个文件符合拆分条件（isSplittable）需同时满足三个条件：

  1. 以 .md 结尾；
  2. 字节大小超过 maxBytes（默认 ENGRAM_DEFRAG_MAX_BYTES=16384）；
  3. splitTopLevel 解析后至少产生 2 个部分（即至少有 2 个 # 顶级标题）。
func isSplittable(path string, content []byte, maxBytes int) bool {
    return strings.HasSuffix(path, ".md") &&
        len(content) > maxBytes &&
        len(splitTopLevel(content)) >= 2
}

internal/maintenance/defrag.go:52–54

splitTopLevel：按顶级标题切分

func splitTopLevel(content []byte) [][]byte {
    var parts [][]byte
    var cur []byte
    flush := func() {
        if len(bytes.TrimSpace(cur)) > 0 {
            parts = append(parts, cur)
        }
        cur = nil
    }
    for _, ln := range bytes.SplitAfter(content, []byte("\n")) {
        if len(ln) == 0 { continue }
        if isTopHeading(ln) { flush() }
        cur = append(cur, ln...)
    }
    flush()
    return parts
}

internal/maintenance/defrag.go:24–44

只有 # （单个 # 加空格）开头的行触发切分，## 及以上不是切分点。 首个顶级标题之前的前导内容（preamble）若含非空白字符则成为第 0 部分。

收敛性：结构性而非尺寸性

Defrag 的收敛不依赖"切分后每部分都小于 maxBytes"，而是依赖结构性约束：

每个拆分产生的 base.NN.md 部分恰好包含一个顶级标题（就是那个触发切分的 # 行）。 因此对该部分调用 splitTopLevel 只返回 1 个元素，isSplittable 为 false。 下一轮扫描不会为它再入队 defrag job，下一次 Defrag 也不会对它再拆分。

副作用：单个超大节（大于 maxBytes 但只有一个顶级标题）永远不会被拆分， 因为 splitTopLevel 只返回 1 部分。这是故意为之—— 强行按字节截断 markdown 会产生损坏的语义结构；拆分应由文档结构决定，不由字节数决定。

8. Reaper：stale running job 回收

如果 worker 进程在处理 job 的过程中崩溃，该 job 的 state 停留在 running， claimed_at 不再更新。下一轮的 ReapStaleJobs 将其回收： 

func (r *Refs) ReapStaleJobs(ctx context.Context, before time.Time, maxAttempts int) (int, error) {
    tag, err := r.pool.Exec(ctx,
        `UPDATE memory_jobs
         SET attempts = attempts + 1,
             state = CASE WHEN attempts + 1 >= $2 THEN 'failed' ELSE 'pending' END,
             claimed_at = NULL
         WHERE state='running' AND (claimed_at IS NULL OR claimed_at < $1)`,
        before, maxAttempts)
    ...
    return int(tag.RowsAffected()), nil
}

internal/memstore/refs/refs.go:181–193

before = time.Now().Add(-reapAfter)，其中 reapAfter 默认 10 分钟（ENGRAM_JOB_REAP_AFTER）。 claimed_at IS NULL 处理了极端情况：行处于 running 但 claimed_at 从未被写入（不应发生，但防御性处理）。

毒药 job 保护：每次 Reap 也执行 attempts++， 因此一个每次都让 worker 崩溃的 job 在 maxAttempts 轮（崩溃→Reap→pending→崩溃……）后 自动进入 failed，不再被 ClaimJob 取到。

注意 Reap 的设计前提是单 worker 假设：reapAfter 必须远大于任何单个 job 的预期执行时间。 若多 worker 并发运行（未来扩展），需将 reapAfter 设得更保守，或改用心跳续约机制， 避免将正常运行中的 job 误判为 stale。
场景claimed_at 状态Reap 动作
worker 正常处理中在 reapAfter 窗口内不触碰
worker 崩溃，job 超时早于 beforeattempts++ → pending 或 failed
claimed_at 为 NULL（异常）NULLattempts++ → pending 或 failed
毒药 job，第 N 次 Reap早于 beforeattempts 达上限 → failed，终止循环

9. Round 顺序与配置

执行顺序的必要性

单 round 内的执行顺序并非随意，有逻辑依赖关系：

步骤原因必须在此位置
① GC 先于 drain，确保本轮清理的是最新可达集；GC 失败则整个 round 中止（不能用不完整的可达集删对象）
② ReapStaleJobs 先于 drain，确保 stale running job 已回到 pending，本轮 drain 能取到它们；scan/reap 错误仅记日志，不中止
③ EnqueueDefrag 先于 drain，确保本轮扫描出的 defrag job 在本轮 drain 中就能被处理（最快响应路径）；扫描错误仅记日志
④ DrainJobs 最后执行；drain 错误中止 round 但不影响已完成的 GC 和 reap

全部在 WithGlobalLock(gcLockKey) 内执行——整个 round 在同一个 Postgres session 上持有全局锁， 同时只有一个 worker 实例执行任何维护操作。

最佳努力 vs. 硬失败

round 内各步骤的错误策略是有意分层的：

  • GC 失败：整个 round 返回错误，记日志，但不崩溃——下一轮重试。 原因：不能基于不完整的可达集删对象，宁可跳过一轮。
  • ReapStaleJobs 失败：仅记日志（log.Printf("reap error: %v", rerr)），round 继续。 已 reap 的 job 等下轮处理；未能 reap 的 stale job 下轮再 reap。
  • EnqueueDefrag 失败：仅记日志，round 继续。 本轮未入队的 defrag job 在下轮扫描时会再次入队（幂等）。
  • DrainJobs 失败：round 返回错误，记日志。已处理的 job 不回滚（已 Complete/Retry）。
