维护管线

一个后台 Go worker，完全在请求路径之外运行：GC、任务队列消化、反思整合、碎片整理、失效任务回收。 全部幂等、可重推导，不需要 Temporal 或 Kafka。

1. 后台 worker 是干嘛的

cmd/maintenance/main.go 是唯一的维护进程。它永远不在前台请求路径上—— agent 会话提交、LLM 调用、工具执行都不等它。 worker 做的四件事都满足同一个前提：

幂等、可从存储状态完全重新推导、单次执行时间短。 因此崩溃重启即可重做，不携带不可再生的中间状态，不需要重型工作流引擎。

四件事按固定顺序依次执行，共同构成一"轮"（round）：

  • GC（mark-sweep）：计算全部 agent HEAD 可达的对象集，删除不可达且已过宽限期的内容地址对象（L5a）。
  • ReapStaleJobs：将因 worker 崩溃而永久停在 running 状态的任务退回 pending（L5d）。
  • EnqueueDefrag：遍历所有 agent，为含有可拆分 .md 文件的 agent 幂等入队 defrag 任务（L5c）。
  • DrainJobs：逐条消费队列里的 pending 任务，派发到 reflect / reindex / defrag 处理器（L5b / L5c）。

整个 round 包裹在一个 Postgres session 级 advisory lock（gcLockKey = 1）内。 未获锁则跳过本轮，这样水平扩展时多个 worker 实例自动降级为主备：

一轮的完整流程（全部在全局锁内） ·  cmd/maintenance/main.go:148–185

不变量 维护 worker 与前台请求路径完全分离，永不阻塞 agent 会话。 所有写入都走与前台相同的 CommitWithCAS 接口，系统只有一个一致性控制点。

2. 任务的一生（状态机）

memory_jobs 是 Postgres 中驱动整个维护管线的任务队列。核心列：

列类型含义
idbigserial PK全局唯一任务 ID
agent_idtext归属 agent
kindtextreflect / reindex / defrag
from_shatext nullable入队时的 HEAD commit SHA（供 reflect 未来 diff-since 用）
statetextpending / running / failed
attemptsint default 0已尝试次数；达到 maxAttempts 转为 failed
claimed_attimestamptz nullable最近被 claim 的时间；Reaper 据此判断是否卡死

任务有两种入队来源：

  • 同事务入队（commit 路径）：CommitRef 在同一 Postgres tx 里推进 HEAD， 同时插入 reflect/reindex 任务—— 任务当且仅当 commit 成功时才存在（替代 Kafka + outbox 模式）。
  • 带外入队（defrag 扫描）：EnqueueJob 通过 partial-unique index (agent_id, kind) WHERE state='pending' 做 ON CONFLICT DO NOTHING， 同一 agent 的同类 pending 任务永远只有一行。

memory_jobs 状态机。done 即行删除，不留墓碑；failed 行留存供排查。

设计亮点 完成（done）状态即行删除——队列只包含活跃工作，历史完成量对表大小无影响。 failed 行留存供审计，不会被 Drain 或 Reaper 二次处理。

3. 抢任务不打架

ClaimJob（refs/refs.go:122）用一个事务原子地"抢占"一行 pending 任务。 整个过程是：SELECT + FOR UPDATE SKIP LOCKED 锁住一行， 立刻 UPDATE 把它标为 running 并记下 claimed_at=now()，提交。 

func (r *Refs) ClaimJob(ctx context.Context) (*DequeuedJob, error) {
    tx, err := r.pool.Begin(ctx)
    ...
    err = tx.QueryRow(ctx,
        `SELECT id, agent_id, kind, from_sha, attempts FROM memory_jobs
         WHERE state='pending' FOR UPDATE SKIP LOCKED LIMIT 1`).
        Scan(&j.ID, &j.AgentID, &j.Kind, &fromSHA, &j.Attempts)
    ...
    tx.Exec(ctx,
        `UPDATE memory_jobs SET state='running', claimed_at=now() WHERE id=$1`, j.ID)
    tx.Commit(ctx)
    return &j, nil
}

internal/memstore/refs/refs.go:122–151

几个关键设计点：

  • SKIP LOCKED：跳过已被其他事务锁住的行，多 worker 各抢各的，不会抢到同一条，也不会因锁等待卡住。 当前是单 worker，但这个设计使未来水平扩展无需修改。
  • 无 ORDER BY：不对 pending 任务排序（"queue hygiene"），防止某类失败任务长期阻塞其他任务的公平处理。
  • LIMIT 1：每次只取一行，保持细粒度进度与错误边界。
  • claimed_at=now()：记录认领时间，供 Reaper 后续判断是否超时卡死。

完成时，CompleteJob 直接 DELETE 行。 失败时，RetryJob 执行 attempts++ 并根据是否达到 maxAttempts 将 state 设为 pending 或 failed， 同时把 claimed_at 清空： 

UPDATE memory_jobs
SET attempts = attempts + 1,
    state = CASE WHEN attempts + 1 >= $2 THEN 'failed' ELSE 'pending' END,
    claimed_at = NULL
WHERE id=$1

4. 按种类派发 + 防忙转

Deps：依赖容器

Deps 把每种任务处理器所需的依赖打包在一起，互不耦合： 

type Deps struct {

}

processJob：按 kind 分发，永远解决任务

func processJob(ctx context.Context, r *refs.Refs, deps Deps,
    job *refs.DequeuedJob, maxAttempts int) error {
    switch job.Kind {
    case "reflect":
        ran, err := r.WithGlobalLock(ctx, agentKey(job.AgentID), func(ctx context.Context) error {
            return Reflect(ctx, deps.Store, deps.Completer, job.AgentID, job.FromSHA)
        })
        if !ran || err != nil {
            return r.RetryJob(ctx, job.ID, maxAttempts)
        }
        return r.CompleteJob(ctx, job.ID)
    case "reindex":
        if err := Reindex(ctx, deps.Store, deps.Emb, deps.EmbCache, job.AgentID); err != nil {
            return r.RetryJob(ctx, job.ID, maxAttempts)
        }
        return r.CompleteJob(ctx, job.ID)
    case "defrag":
        ran, err := r.WithGlobalLock(ctx, agentKey(job.AgentID), func(ctx context.Context) error {
            return Defrag(ctx, deps.Store, job.AgentID, deps.DefragMaxBytes)
        })
        if !ran || err != nil {
            return r.RetryJob(ctx, job.ID, maxAttempts)
        }
        return r.CompleteJob(ctx, job.ID)
    default:
        return r.CompleteJob(ctx, job.ID) // 未知 kind：丢弃，不堵队列
    }
}

internal/maintenance/drain.go:34–61

三条核心规则贯穿所有分支：

  1. 总是解决任务：无论成功还是失败，processJob 一定调用 CompleteJob 或 RetryJob，任务行绝不永久停留在 running。
  2. 未知 kind 直接 Complete：unknown 任务不重试，避免僵尸任务堵塞队列。
  3. 未获 per-agent 锁（ran=false）视为失败：退回 RetryJob，等下一轮再试。

DrainJobs：seen-set 防忙转

func DrainJobs(ctx context.Context, r *refs.Refs, deps Deps, maxAttempts int) (int, error) {
    seen := map[int64]struct{}{}
    processed := 0
    for {
        job, err := r.ClaimJob(ctx)
        ...
        if job == nil { return processed, nil } // 队列已空
        if _, dup := seen[job.ID]; dup {
            // 同一 round 内再次见到同一任务——它被 requeue 了
            // 再 RetryJob 一次（第二次 bump attempts）然后结束本轮
            r.RetryJob(ctx, job.ID, maxAttempts)
            return processed, nil
        }
        seen[job.ID] = struct{}{}
        processJob(ctx, r, deps, job, maxAttempts)
        processed++
    }
}

internal/maintenance/drain.go:68–91

seen 集合记录本 round 内已处理的任务 ID。 如果一个任务被 RetryJob 退回 pending，又被同 round 的下一次 ClaimJob 再次取到， DrainJobs 立刻第二次 RetryJob（再 bump 一次 attempts）并结束本轮。 这样一个反复失败的任务在若干 round 后会因 maxAttempts 进入 failed，不会忙转到天荒地老。

注意 seen-set 只在一次 DrainJobs 调用（一轮）内有效，不跨 round 持久化。 每轮 drain 前 GC 和 Reap 已先行执行，stale running 任务已被回收，所以 drain 开始时 seen 集合总是干净的。

5. per-agent 单例锁 + 不自我触发

agentKey：FNV 哈希到 advisory lock key

func agentKey(agentID string) int64 {
    h := fnv.New64a()
    _, _ = h.Write([]byte(agentID))
    return int64(h.Sum64())
}

internal/maintenance/drain.go:25–30

reflect 和 defrag 任务都在 r.WithGlobalLock(ctx, agentKey(agentID), ...) 下执行。 这是维护侧的 per-agent 单例锁，防止同一 agent 的多个维护任务并发写入内存。 它和前台写者持有的是不同的 advisory lock key（前台用全局锁 key=1，维护任务用 agent 哈希 key），两者独立。

哈希碰撞只让两个 agent 偶尔串行化，不影响正确性—— 真正的正确性由 CommitWithCAS 的 CAS 保障，advisory lock 只是减少无谓的 CAS 冲突。

jobs=nil：切断自触发回路

reflect 和 defrag 在写回时，传入 CommitWithCAS 的 jobs 参数是 nil：

// jobs=nil: reflection never enqueues a reflect job (no self-trigger loop).
if _, err := store.CommitWithCAS(ctx, agentID, head, dir, nil); err != nil { ... }

internal/maintenance/reflect.go:72  |  internal/maintenance/defrag.go:159

前台 agent 提交时传入 []refs.Job{{Kind:"reflect"},{Kind:"reindex"}}， 维护提交传入 nil——维护提交永远不会入队新的维护任务， 从结构上断开"维护触发维护"的反馈回路，而非依赖运行时检查。

不变量 maintenance commit 使用 jobs=nil。reflect 和 defrag 的提交不会入队任何新任务， 因此维护管线不会自我增长成无限循环。这是结构性保证，不是靠 flag 或条件判断。

CAS 冲突 → ErrConflict → 重试

如果 agent 在维护写入期间推进了 HEAD（前台并发写），CommitWithCAS 的 CAS 失败， reflect.go 和 defrag.go 将其转为 ErrConflict， processJob 调用 RetryJob 把任务退回 pending。 下一轮 Drain 时 Reflect 重新 ResolveHead，看到最新状态再做整合——绝不强制覆盖。

Reflect 执行步骤

  1. ResolveHead 取当前 HEAD（忽略 fromSHA，总是看最新状态；fromSHA 保留供未来 diff-since 优化）。
  2. Materialize 将 HEAD tree 物化到临时目录（os.MkdirTemp，defer RemoveAll）。
  3. readSystemDir 遍历 system/ 子目录，将所有文件拼接为带路径标题的字符串。
  4. 空居民集跳过 if resident == "" { return nil }——system/ 为空时不提交任何内容，避免产生无意义的 commit churn。
  5. Complete 调用 Completer，传入 reflectSystemPrompt + 居民内容，获得整合摘要。
  6. 写入 system/reflection.md 将 LLM 输出写到物化目录中（MkdirAll 确保路径存在）。
  7. CommitWithCAS(jobs=nil) CAS 提交，冲突返回 ErrConflict，调用方 RetryJob。

readSystemDir 错误传播

walkErr := filepath.WalkDir(systemDir, func(path string, d fs.DirEntry, err error) error {

internal/maintenance/reflect.go:86–107

fs.ErrNotExist（system/ 目录尚未创建）通过 SkipAll 静默处理， readSystemDir 返回 ""，上层跳过本次整合。 其他读取错误（权限、IO 故障）向上传播，最终触发 RetryJob。

设计亮点 Reflect 总是基于最新 HEAD，确保整合结果不会基于过时的居民集。 fromSHA 字段已保留，为未来 diff-since-last-reflect 优化提供钩子—— 届时可只把变化部分送给 LLM，而非全量重读。

7. 拆大文件（Defrag）

可拆分性判断

一个文件符合拆分条件（isSplittable）须同时满足三个条件：

  1. 以 .md 结尾；
  2. 字节大小超过 maxBytes（默认 ENGRAM_DEFRAG_MAX_BYTES=16384）；
  3. splitTopLevel 解析后至少产生 2 个部分（即至少有 2 个 # 顶级标题）。
func isSplittable(path string, content []byte, maxBytes int) bool {
    return strings.HasSuffix(path, ".md") &&
        len(content) > maxBytes &&
        len(splitTopLevel(content)) >= 2
}

internal/maintenance/defrag.go:52–54

splitTopLevel：按顶级标题切分

func splitTopLevel(content []byte) [][]byte {
    var parts [][]byte
    var cur []byte
    flush := func() {
        if len(bytes.TrimSpace(cur)) > 0 { parts = append(parts, cur) }
        cur = nil
    }
    for _, ln := range bytes.SplitAfter(content, []byte("\n")) {
        if len(ln) == 0 { continue }
        if isTopHeading(ln) { flush() }
        cur = append(cur, ln...)
    }
    flush()
    return parts
}

internal/maintenance/defrag.go:24–44

只有 # （单个 # 加空格）开头的行触发切分，## 及以上不是切分点。 首个顶级标题之前的前导内容（preamble）若含非空白字符则成为第 0 部分。

为什么不会越拆越多：结构性收敛

Defrag 的收敛不依赖"切分后每部分都小于 maxBytes"，而是依赖结构性约束：

每次拆分产生的 base.NN.md 部分恰好包含一个顶级标题（就是那个触发切分的 # 行）。 对该部分调用 splitTopLevel 只返回 1 个元素，isSplittable 为 false。 下一轮扫描发现它不可拆，不入队；下一次 Defrag 也不会对它再拆。

副作用：单个超大节（大于 maxBytes 但只有一个顶级标题）永远不会被拆分。 这是故意为之——强行按字节截断 markdown 会破坏语义结构；拆分应由文档结构决定，不由字节数决定。

Defrag 收敛流程。拆出的每部分恰好含 1 个顶级标题，isSplittable 对其永久返回 false。

不变量 收敛性由结构保证：每次拆分后各部分恰好有 1 个顶级标题，使 isSplittable 对其永久返回 false， 与文件大小无关（单节超大文件保持完整）。

命名方案与空优化

原文件 notes.md 被删除，各部分按 notes.01.md、notes.02.md…… 命名。 若遍历后 targets 为空（没有任何可拆分文件），Defrag 直接返回 nil，不产生空 commit。

8. 回收卡死的任务（Reaper）

如果 worker 在处理任务途中崩溃，该任务的 state 永久停在 running，claimed_at 不再更新。 每轮的 ReapStaleJobs 将其退回 pending： 

func (r *Refs) ReapStaleJobs(ctx context.Context, before time.Time, maxAttempts int) (int, error) {
    tag, err := r.pool.Exec(ctx,
        `UPDATE memory_jobs
         SET attempts = attempts + 1,
             state = CASE WHEN attempts + 1 >= $2 THEN 'failed' ELSE 'pending' END,
             claimed_at = NULL
         WHERE state='running' AND (claimed_at IS NULL OR claimed_at < $1)`,
        before, maxAttempts)
    ...
    return int(tag.RowsAffected()), nil
}

internal/memstore/refs/refs.go:181–193

before = time.Now().Add(-reapAfter)，其中 reapAfter 默认 10 分钟（ENGRAM_JOB_REAP_AFTER）。 claimed_at IS NULL 处理极端情况：行处于 running 但 claimed_at 从未被写入（防御性处理）。

毒药任务保护：每次 Reap 也执行 attempts++。 一个每次都让 worker 崩溃的任务，经过"崩溃→Reap→pending→崩溃→Reap……"循环， 在 maxAttempts 轮后自动进入 failed，不再被 ClaimJob 取到。

场景claimed_at 状态Reap 动作
worker 正常处理中在 reapAfter 窗口内不触碰
worker 崩溃，任务超时早于 beforeattempts++ → pending 或 failed
claimed_at 为 NULL（异常）NULLattempts++ → pending 或 failed
毒药任务，第 N 次 Reap早于 beforeattempts 达上限 → failed，终止循环
注意 Reap 的设计前提是单 worker：reapAfter 必须远大于任何单个任务的预期执行时间。 若未来多 worker 并发运行，需将 reapAfter 设得更保守，或改用心跳续约机制， 避免将正常运行中的任务误判为 stale。

9. 一轮的顺序与配置

执行顺序的逻辑依赖

单 round 内的执行顺序有内在逻辑依赖，不能随意调换：

步骤为什么必须在此位置失败策略
① GC 先于 drain，确保本轮清理的是最新可达集；绝不能基于不完整的可达集删对象。 整个 round 返回错误，记日志，不崩溃
② ReapStaleJobs 先于 drain，确保 stale running 任务已回到 pending，本轮 drain 能取到它们。 仅记日志，round 继续
③ EnqueueDefrag 先于 drain，确保本轮扫描出的 defrag 任务在本轮 drain 中就能被处理（最快响应路径）。 仅记日志，round 继续
④ DrainJobs 最后执行；已完成的 GC 和 Reap 不会被 drain 的失败回滚。 round 返回错误，记日志
