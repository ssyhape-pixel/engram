搜索与索引

两种互补的找法（三字片段倒排 + 段落语义向量）经 RRF 融合成一条结果； 向量算一次就持久存下来，后续调用透明复用——无论会话重启、Pod 换节点还是后台预热，都共享同一份向量库。

1. 立场：记忆是主动翻找的，不是开口前倒进去的

Engram 的 recall 是一个中循环工具调用——模型在推理过程中按需发起， 不是预推理阶段把所有相关片段一股脑检索进来的"RAG 前置"。

推理开始时只加载两类内容：

  • system/ 常驻文件——直接进入系统提示，由 L3 SHA 读缓存加速（详见报告 ② 请求路径）；
  • 树索引（文件名 + description 字段）——让模型知道长尾记忆的存在，但不把内容拉进来。

只有当模型判断某段长尾记忆与当前任务相关时，才显式调用 recall(query, k)， 拿回精确的行范围和片段。这类似"按需分页"而不是"提前整批倒入"。

搜索本地化（Variant A，默认）： Trigram 索引和语义索引在 Session 开启时于内存中从已物化的工作目录重建， 查询时零对象存储访问。索引随 Session 关闭而丢弃——它是可再生的派生视图，不是权威存储。

不变量

索引（trigram + semantic）是派生的、可丢弃的视图。真相永远在 object store 的 content-addressed 对象 + Postgres ref 里。缓存崩溃、索引丢失均可从对象重建，不影响正确性（核心不变量 ③）。

2. 关键词找法：三字片段倒排索引

白话解释什么是"三字片段倒排"： 把每一行文字拆成三个字符一组的滑动窗口（shingle），建一张反查表—— 给定任意三字片段，能立刻查到"哪些行含有它"。 找"project goals"这个词时，先用 pro、roj、oje…把候选行圈出来， 再精确验证哪些行真正包含完整字串。

这套索引是纯 CPU、零 I/O 的。Session 开启时与物化树同步重建，轻量到可以随时丢弃再来。

Trigram 提取

// trigram.go:21–37
func trigrams(s string) []string {
    r := []rune(strings.ToLower(s))
    if len(r) < 3 { return nil }
    seen := make(map[string]struct{}, len(r))
    out := make([]string, 0, len(r)-2)
    for i := 0; i+3 <= len(r); i++ {
        t := string(r[i : i+3])
        if _, ok := seen[t]; ok { continue }   // 同行重复 trigram 只记一次
        seen[t] = struct{}{}
        out = append(out, t)
    }
    return out
}

去重是关键：同一个 trigram 在一行中出现多次只记一次，避免 posting 列表膨胀。 索引和查询均 lowercase，返回的 snippet 保留原始大小写。

候选集求交 → 精确验证

candidates(query)（trigram.go:65–102）的逻辑分两步：

  1. 切 trigram，查 posting 列表。查询串的每个三字片段各自取出"含有它的行"集合；若某 trigram 在整个索引里找不到，立即返回空集（早退）。
  2. 从最小集合开始做交集。按 posting 集合大小升序排列，从最小的那个开始逐一过滤，最小化无效遍历。
  3. 精确子串验证。strings.Contains(lowercase(lineText), lowercase(query)) 是最终判据——trigram 交集是预过滤器（可能有假阳性），substring 检查移除它们。不漏报也不错报。
  4. 短查询退化为全量扫描。长度小于 3 个 rune 时无 trigram 可用，对所有行做一次 grep（退化为线性扫描）。
// trigram.go:107–135 (Search)
func (t *TrigramIndex) Search(query string, k int) []Hit {
    q := strings.ToLower(query)
    var matches []posting
    for p := range t.candidates(q) {

            matches = append(matches, p)
        }
    }
    sort.Slice(matches, func(i, j int) bool { /* path asc, line asc */ })
    // 截取 top-k，每条命中 LineStart == LineEnd（单行）
    ...
}

3. 语义找法：段落向量索引

白话解释：把一段文字"变成一串数字"，意思相近的文字对应的数字串也相近（余弦相似度高）。 关键词找法要求查询词和文档里的词一模一样；语义找法则能捕捉"项目目标"和"project goals" 这类跨语言/近义的关联，以及跨行分布的含义。

分段逻辑

sectionize（semantic.go:40–52）以 Markdown 任意级标题行（# 开头）为边界， 把文件切成若干段落。第一个标题前的内容自成一段；无标题则整个文件为一段。 每段携带 1-based 行范围，用于 Hit 定位。

Embedding key 设计——内容寻址

// semantic.go:63–66
func embKey(model, text string) string {
    sum := sha256.Sum256([]byte(text))
    return "emb:" + model + ":" + base64.RawStdEncoding.EncodeToString(sum[:])
}

两段信息共同构成 key：

  • model 前缀——不同嵌入模型的向量维度/语义空间不同，混用会让相似度计算失效；
  • sha256(text)——相同内容在任何时间、任何 worker 上对应同一个 key；内容一旦变化则 key 随之变化，天然内容寻址。

BuildSemantic：缓存命中 → 复用，缺失 → 批量 Embed → Put

// semantic.go:107–156（核心逻辑）

4. 两种找法揉一起：RRF 融合

白话解释 RRF（Reciprocal Rank Fusion）： 给每条候选结果在各自列表里的名次打分——名次越靠前，分越高； 两边都靠前的，总分最高。公式是 1 / (60 + rank + 1)， 常数 60 平滑了头部排名差异（标准超参数），使得第 1 名和第 2 名的分差不那么悬殊。 同一条结果出现在多个列表时，得分累加。

RRF 实现

// hybrid.go:56–85
func rrf(lists [][]Hit, k int) []Hit {
    const rrfK = 60
    scores := map[string]*agg{}
    var order []string   // 首次出现顺序，保证同分时稳定 tie-break
    for _, list := range lists {
        for rank, hit := range list {
            id := fmt.Sprintf("%s:%d-%d", hit.Path, hit.LineStart, hit.LineEnd)
            a, ok := scores[id]
            if !ok {
                a = &agg{hit: hit}
                scores[id] = a
                order = append(order, id)
            }
            a.score += 1.0 / float64(rrfK+rank+1)   // 累加贡献
        }
    }
    sort.SliceStable(order, func(i, j int) bool {
        return scores[order[i]].score > scores[order[j]].score
    })
    // 取 top-k
}

命中 ID 以 path:lineStart-lineEnd 标识；行级和段落级 Hit 的 ID 格式相同， 因此同一行恰好也是一个段落时，两边的分数会合并。

降级策略

NewHybrid（hybrid.go:22–28）在构建时 best-effort 构建语义索引—— 若 BuildSemantic 报错（如 embedding 服务不可用），h.sem 保持 nil， Recall 仅使用 trigram 列表，不向调用方报错。查询时 semantic 失败也同样静默降级：

    if semHits, err := h.sem.Search(ctx, query, n); err == nil {
        lists = append(lists, semHits)
    }
    // 查询时 embedding 失败：丢弃语义，保留 trigram（降级）
}
不变量

Recall 永远不因 embedding 服务故障而报错。最差情况退化为纯 trigram 检索， 覆盖精确词汇匹配场景。见测试 TestHybridDegradesWhenEmbeddingFails（hybrid_test.go）。

两种找法的互补盲区

Trigram 强在哪

  • 精确 token（如 xq7z9、API key 片段）
  • 无 embedding 服务时仍工作
  • 短文件、精确拼写场景
  • 零额外延迟（纯内存）

Semantic 强在哪

  • 跨行分布的短语（词在三行里各一个）
  • 同义词 / 近义描述（"目标" ≈ "goals"）
  • 跨语言语义相关
  • 模糊提问，不记得原文措辞

5. 按内容指纹的读缓存

cache.Cache（cache.go:13–16）是一个极简的字符串 KV 接口： 

type Cache interface {
    Get(key string) (string, bool)
    Put(key, val string)
}

Engram 的所有缓存 key 都是内容哈希，因此命中 = 等价于重新计算， 不存在"旧值失效"的问题。"失效"的唯一形式是 LRU 淘汰（空间压力），不影响正确性，只影响缓存率。

LRU（cache.go:23–69）：

  • 单 sync.Mutex 保护所有操作——Get 需要修改 recency list（MoveToFront），无法用 RWMutex；
  • 容量上限为 entry 数量；超出时淘汰最旧项；
  • 同一 key 重复 Put 刷新 recency，不增加 entry 数。

Session 层有两个独立的缓存实例（见 router.go:40 注释："they MUST be different instances"）：

  • sysCache——cache.NewLRU(1024)，缓存 system/ 内容的组装结果，详见报告 ②；
  • embCache——cache.NewTiered(LRU, ObjCache)，缓存 embedding 向量，详见 §6。
不变量

sysCache 与 embCache 是不同实例，且指向不同的后端 ObjStore 目录。 system 内容不会写入 embedding 存储，embedding 向量不会污染 system 缓存。

6. 把昂贵的向量存下来（L4b）

Voyage API 调用有延迟也有费用，而段落内容一旦写入就不可变（content-addressed）—— 同一段落在任何时间、任何 worker 上算出的向量完全相同。 L4b 把 embedding 向量持久化到独立的 content-addressed ObjStore， 让"昂贵的联网算向量"退化为"廉价的磁盘/S3 读"。

设计的关键妙处在于：算向量那段代码（BuildSemantic）一行没改， 持久化纯靠"换个缓存注进去"——依赖注入，零侵入。

ObjCache：持久化层

// objcache.go:21–36
func safeKey(key string) string {
    sum := sha256.Sum256([]byte(key))
    return hex.EncodeToString(sum[:])   // 纯十六进制，文件路径/S3 key 均安全
}

func (o *ObjCache) Get(key string) (string, bool) {
    data, err := o.objs.Get(context.Background(), safeKey(key))
    if err != nil { return "", false }   // 错误 = miss，best-effort
    return string(data), true
}

func (o *ObjCache) Put(key, val string) {
    _ = o.objs.Put(context.Background(), safeKey(key), []byte(val))   // 忽略错误
}

embKey 已经是 base64 字串（含 :），直接用于文件路径不安全， safeKey 再次 sha256 + hex 编码。key 是"key 的哈希"而非"值的哈希"， 但唯一性由 embKey 内部的 sha256(model+text) 保证。

Tiered：内存层在前，持久层在后

// tiered.go:11–25
func (t *Tiered) Get(key string) (string, bool) {
    if v, ok := t.front.Get(key); ok { return v, true }   // 先查 LRU
    if v, ok := t.back.Get(key); ok {
        t.front.Put(key, v)   // 从磁盘提升到内存
        return v, true
    }
    return "", false
}

func (t *Tiered) Put(key, val string) {
    t.front.Put(key, val)   // 同步双写
    t.back.Put(key, val)
}

读取时先查 LRU（快），LRU miss 再查 ObjCache（磁盘/S3），命中则提升到 LRU 以备下次热读。 写入同步双写，保证持久化与内存缓存一致。

7. 增量补算：后台预热，前台零联网

maintenance.Reindex（reindex.go:23–46）是 L4b 的维护端： 它物化 agent HEAD 的完整树（不只是 system/，search 要索引所有文件）， 调用 BuildSemantic，主要目的是触发 Put 副作用—— 把所有段落的 embedding 写入持久化 embCache。 返回的 *SemanticIndex 被丢弃。 

// reindex.go:41–44
// BuildSemantic persists missing embeddings into embCache as a side-effect.
if _, err := search.BuildSemantic(ctx, emb, embCache, files); err != nil {
    return fmt.Errorf("maintenance: reindex build semantic: %w", err)
}

为什么不需要 diff？

内容寻址使增量天然免费，不需要任何 diff 计算：

  1. 计算每段落的 embKey。embKey(model, text) = sha256(model+text)；内容相同则 key 相同。
  2. 查 embCache。已持久化的段落直接命中，跳过 Embed 调用。
  3. 只有新增或修改的段落才联网。key 不存在（段落是新的或内容有变）才会调用 Voyage API。
  4. 第二遍 reindex 零联网。对同一 HEAD 所有段落已持久化，Embed 调用次数为零。

这一行为通过 countingEmbedder 精确验证： 

// reindex_test.go:46–52
// Second reindex: every chunk's embedding is already persisted → 0 new embeds.
if err := Reindex(ctx, store, ce, embCache, agentID); err != nil { ... }
if ce.calls != first {
    t.Fatalf("second reindex embedded %d more texts, want 0 (incremental persistence broken)",
        ce.calls-first)
}

前台 Session 和后台 worker 共享同一份向量库

两端（cmd/api 和 cmd/maintenance）各自构建一个指向同一目录（ENGRAM_EMB_OBJ）的 ObjStore， 再各自包一层 Tiered(LRU + ObjCache)。 LRU 是进程内的，不共享；ObjStore 是磁盘/S3，天然共享。

潜在问题

两端必须使用同一个 embedding 模型（ENGRAM_EMBEDDER 环境变量）。 embKey 携带 model 前缀，模型不同则 key 不同，向量库白存—— maintenance 预热的 embedding 对 Session 完全无效，退化为每次 Session 都重新调用 Voyage。

ENGRAM_PROVIDER（LLM 推理提供商）和 ENGRAM_EMBEDDER（向量模型）是两个独立变量， 配置时不要混淆。生产部署应在两端固定设为 voyage。

后台预热 → 会话零联网时序

后台 worker 首次为每个段落联网算向量并持久化；之后的 Session 开启时所有向量均从磁盘命中，Voyage API 调用次数为零。

8. 和 GC 隔离

Engram 的 GC（L5a，maintenance/gc.go）通过可达性分析清除无引用的 git 对象。 GC 操作的是 git object store（ENGRAM_OBJ）； embedding 向量存储在独立的目录（ENGRAM_EMB_OBJ，默认 ./engram-embeddings）。 两者是不同的 objstore.NewLocal 实例，GC 的可达性集合计算完全不涉及 embedding store。

两个 cmd 均在启动时强制检查目录分离： 

// cmd/api/main.go:81–83
if filepath.Clean(embObjRoot) == filepath.Clean(objRoot) {
    log.Fatalf("ENGRAM_EMB_OBJ and ENGRAM_OBJ must be different directories (got %q)", embObjRoot)
}

// cmd/maintenance/main.go:133–135
if filepath.Clean(embObjRoot) == filepath.Clean(objRoot) {
    log.Fatal("ENGRAM_EMB_OBJ must differ from ENGRAM_OBJ (GC must never sweep the embedding store)")
}
不变量

GC 永远不会清除 embedding 向量。embedding store 是只增长的内容寻址存储： 旧段落的向量不被当前 HEAD 引用，但也不在 git object store 里，所以 GC 扫不到它们。 启动时路径相等的硬校验是双重防呆。

embedding 的"过期"管理（若需要）是独立于 git GC 的问题，目前未实现（可接受：记忆体量小，向量累积速度慢）。

git GC 的完整分析见报告 ③ 存储与一致性。

设计总结

L4 核心

Hybrid = Trigram + Semantic

词汇精确性 + 语义模糊性，RRF(k=60) 融合；embedding 失败时静默降级为纯 trigram，Recall 永不报错。

L4b 收益

Embedding 持久化

Tiered(LRU + ObjCache) 注入 BuildSemantic；semantic.go 零改动，Voyage 调用大幅减少。

增量免费

Content-addressed Keys

embKey = sha256(model + text)；已持久化的段落天然命中缓存，无需 diff，第二遍 reindex 零联网（有计数测试为证）。

安全隔离

GC 不碰 Embedding Store

独立 ObjStore 根目录 + 启动时路径相等硬校验，双重保障。
