1 · 一层层盖起来

架构文档从第一天就瞄准 M5（规模化），但实现不是一次推进的。每一层进 main 后，构建和全量测试必须全绿，下一层才能开工。这样做的好处不只是"纪律感"——每次 merge 都是一个真实可发布的增量，没有大功能分支，没有"等整合完再测"的债。

2 · 每层都走的规程

每一层都经历相同的五个阶段。顺序从不跳过，阶段之间不随意切换。这不是仪式感——每个阶段有具体的产物（文件、测试、提交），产物存在才算完成。

  1. 构思（brainstorming） 探清用例边界、接口形状、与已有包的关系。产物：docs/superpowers/specs/ 下的 Spec 文档，每个章节都必须有 SVG 或 Mermaid 图——不允许只有文字加代码块。Spec 必须"离线可读"，不依赖任何外部渲染服务。
  2. 写计划（writing-plans） Spec 确认后，把实现拆成一个个 bite-sized TDD 任务，存入 docs/superpowers/plans/。每个任务的验收条件必须是"单独运行该任务的测试可以绿"——不允许"等后面任务一起测"。任务粒度以 30–90 分钟为目标。
  3. 子 agent 执行（subagent-driven execution） 每个任务派发给一个新鲜 subagent，没有前置会话上下文的干扰。TDD 顺序严格：先写测试（看它失败），再写实现，再看测试变绿。每个任务对应一次独立的 feat/fix/refactor/test 提交。多个独立任务可以并发派发给不同 subagent。
  4. 双重审查（spec-review + code-review） 第一阶段：对照 Spec 做合规检查（实现是否覆盖了所有设计点，接口是否与图表一致）。第二阶段：代码质量检查（错误传播 %w、接口大小、测试覆盖、边界条件）。通常合并为一次最终审查，产出一批 fix/refactor 提交，之后才 merge。
  5. 本地 merge + 删分支 git merge --no-ff 保留分层历史（主线可见每层边界），然后删除功能分支，不留悬空引用。不用 GitHub PR 合并，避免自动 squash 抹去提交粒度。

2.1 编码规范（五条死规矩）

这五条规矩在 CLAUDE.md 中明确写出，在审查阶段会逐条核对：

规矩具体要求为什么
测试先行 新接口必须先写测试并看到失败，再写实现 测试先行逼迫你在写实现前想清楚接口契约，事后补测试往往只测已经工作的路径
%w 包装错误 所有 fmt.Errorf 必须用 %w，不用 %s 上层可以用 errors.Is/errors.As 精确匹配；%s 把错误变字符串，可达性丢失
context.Context 第一参数 所有 I/O 函数第一参数为 ctx context.Context 超时取消必须能传播；无 ctx 的函数在未来集成时几乎必然要破坏接口
小接口 包边界只暴露最小接口，内部实现不泄漏 小接口好 mock、好测试、好替换；接口太大在包间引入强耦合
表驱动测试 多场景覆盖用 []struct{name, input, want}，不重复写函数 新增测试用例只加一行结构体，不用复制整个测试函数体

规律

上面 9 个 bug 中，有 3 个（advisory lock 泄漏、ENGRAM_EMBEDDER 发散、GC 误删 embedding）是跨边界集成缺陷——单元测试覆盖的是单个函数的行为，无法感知两个独立进程或两个独立存储路径之间的约定。这是为什么两阶段审查（spec 合规 + 代码质量）不可省略的直接证据。

4 · 测试怎么互不打架

Postgres 依赖测试在 go test ./... 并发执行时，早期版本存在间歇性 flake。根因是多个 _test.go 共享同一张 refs 表，某个测试的 DELETE 或断言会与其他测试的写入产生竞争。

修复分两种策略，分别针对两类不同的操作：

4.1 普通 CRUD 测试：per-test agent id + scoped DELETE

对于普通的 refs / MemStore 测试（commit d3f4bba，merge 86f71dd）：

  • 每个测试生成唯一 agent id（t.Name() 哈希 + 随机后缀），杜绝两个并发测试操作同一行。
  • 清理只 DELETE 自己插入的行（WHERE agent_id = $1），而不是 TRUNCATE 整张表。
  • 移除所有全局 TRUNCATE：全局清空在并发测试下会把其他测试正在使用的数据一并删除。

4.2 全局扫描类操作：per-test schema 隔离

对于 ClaimJob、DrainJobs 这类不带 agent_id 过滤的全局扫描，仅靠 scoped DELETE 不够。扫描会看到其他测试插入的 pending jobs，破坏"exactly N jobs claimed"的断言。修复方案是 per-test schema 隔离：

  • 每个测试在事务内执行 SET search_path TO test_<unique_name>，所有 DDL 和 DML 都在独立 schema 下运行。
  • 全局扫描（SELECT ... FOR UPDATE SKIP LOCKED）只能看到同 schema 内的行——不同测试的数据完全不可见。
  • 测试完成后 DROP SCHEMA CASCADE 清理，无残留。

左：旧方案，多个测试写同一张表，全局扫描互相干扰。右：新方案，全局操作测试用独立 schema，扫描只看到本测试的数据。

成果

隔离改造后，ENGRAM_TEST_DB="..." go test -race ./... 连续多次运行零 flake。29 个测试文件、跨 10 个包并发执行，无需 -p 1 串行化。

5 · 几条铁律（守则）

CLAUDE.md 列出了四条不得违反的设计约束，每条背后都有明确的工程推导，不是品味问题。

守则工程理由违反的代价
不上 Temporal / Kafka 维护任务幂等、短小（秒级）、可重派；Postgres 同事务入队已消除 outbox 双写；reconcile loop 已够用；River 是 M5 级的 drop-in 引入 Temporal/Kafka 需独立运维、独立健康监控，在 Agent 记忆规模下净收益为负
对象不可变 内容寻址 = 哈希永远映射同一内容 = 缓存永不失效（L3 LRU 无需 invalidation）；GC 靠可达性判断；append-only 写入崩溃安全 任何 mutation 破坏缓存正确性，引入 invalidation 复杂度，GC 无法做简单标记-清除
并发控制只有 ref CAS 一个点 单一序列化点使推理简单；所有写者在同一个原子操作竞争，输者重试，无两阶段提交、无分布式锁协议 第二个并发控制点引入分布式事务，死锁概率指数级增长
system/ 精小；缓存/索引/工作树是派生视图 system/ 每轮推理必须塞进 LLM 上下文，字节直接消耗 token；派生视图可随时重建，Pod 崩溃不丢数据 system/ 膨胀导致每轮 token 爆炸；把缓存当主存在 Pod 重启时丢数据

6.3 唯一有意推迟的功能

Embedding store 的驱逐（eviction）逻辑被有意推迟，不是技术问题，而是优先级决策：

  • Embedding 是内容寻址的派生视图，可从 HEAD 全量重建（maintenance.Reindex）。
  • 在 Agent 记忆规模下，每个 agent 的 embedding store 体积很小（Markdown 文本，embedding 数量有限）。
  • 实现 LRU 驱逐会增加 ObjCache 的复杂度（需要跟踪访问时间、设计驱逐策略），在需求出现前不值得投入。

6.4 §15 研究 A/B 候选项

2026 年深度研究交叉验证了大部分设计决策。两个真实优化目标已在构建中实现，与研究推荐方向一致：

  • 混合召回（§15.2）：L4 已实现 trigram + semantic RRF fusion。
  • 队列卫生（§15.3）：L5b 的 SKIP LOCKED + 无 ORDER BY + L5d 的 stale reaper 对应"操作杠杆"建议；River 保留为 M5 drop-in。

存储底层（§15.4 Prolly Trees / lakeFS SSTable）和结构化记忆（§15.5 Zep/Graphiti）作为 A/B 候选项保留，当前规模不需要。
