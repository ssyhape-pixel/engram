存储与一致性

不可变对象 · gitfs · refs CAS · 任务队列 · advisory lock · GC · 迁移 — Engram 持久层从头到尾

覆盖层级：L1 MemStore core · L2 agent loop · L3 SHA read cache · L5a GC · L5b maintenance jobs

Engram 的持久层由三个协作部件构成：以哈希为键的不可变对象存储（ObjStore）、 把 git 当格式用的自定义 Storer（gitfs），以及存在 Postgres 里的唯一可变指针（refs）。 三者合力保证：对象不可变、历史可重放、并发写入不产生撕裂状态。

核心接口是 MemStore（internal/memstore/memstore.go:33–38）—— 四个方法：ResolveHead、Materialize、CommitWithCAS、TreeKeys。 下面逐层解剖每个方法背后的存储机制。

1 · 按指纹存的不可变对象

每一个存储单元——blob（文件内容）、tree（目录快照）、commit（版本节点）——都以其自身内容的 SHA-1 哈希字符串作为键。"内容相同"和"键相同"是一回事，因此：

  • 按 SHA 缓存，无需失效逻辑。 相同的哈希永远对应相同的字节；缓存命中时无需检查是否过期。
  • 写入天然去重。 两个 agent 若有完全相同的 system/persona.md，存储中只保留一个 blob。
  • GC 可用可达性推导。 从所有 agent HEAD 出发遍历对象图，未被引用的即是垃圾，可安全回收。

ObjStore 接口

接口定义于 internal/memstore/objstore/objstore.go:16–26，六个方法： 

type ObjStore interface {
    Has(ctx context.Context, key string) (bool, error)
    Get(ctx context.Context, key string) ([]byte, error)
    Put(ctx context.Context, key string, data []byte) error // idempotent
    Iter(ctx context.Context, fn func(key string) error) error
    Stat(ctx context.Context, key string) (time.Time, error) // 返回对象创建时间，GC 用
    Delete(ctx context.Context, key string) error             // 缺失键不报错（幂等）
}

key 是十六进制哈希字符串。Put 注释明确写着 idempotent： 相同的 key 意味着完全相同的字节，重复写入安全无害。Stat 返回对象的 最后修改时间——对不可变对象而言等于创建时间——GC 用这个值做宽限期判断。

Local 后端：原子写入

开发/测试环境使用本地文件系统后端（internal/memstore/objstore/local.go）。 对象按 <root>/objects/<key[:2]>/<key> 路径存放——两字符前缀分片， 与 .git/objects/ 布局完全一致，避免单目录条目过多。

Put 实现了"先写临时文件再 os.Rename"的原子可见性模式 （local.go:53–79）： 

func (l *Local) Put(ctx context.Context, key string, data []byte) error {
    p := l.path(key)
    if _, err := os.Stat(p); err == nil {
        return nil  // 已存在，幂等返回
    }
    os.MkdirAll(filepath.Dir(p), 0o755)
    tmp, _ := os.CreateTemp(filepath.Dir(p), ".tmp-*")
    tmpName := tmp.Name()
    defer os.Remove(tmpName)
    tmp.Write(data)
    tmp.Close()
    os.Rename(tmpName, p)  // 原子可见：观察者只见完整字节
}

os.Rename 在同一文件系统内是原子操作：任何读者要么看不到这个文件， 要么看到完整的字节。由于对象不可变，若并发写入同一 key，最后一次 rename 胜出——结果始终正确。

对象图：commit → tree → blob + 父链

图 1 · 对象图：commit→tree→blob + parent 父链。改一个文件 = 新 blob + 新受影响 tree + 新 commit；C0 及其所有对象原样保留。

不变量 对象永远不可变。 改变一个文件 = 新 blob + 新 tree（及所有受影响的父 tree）+ 新 commit。 旧对象保留至 GC 判定不可达且超过宽限期。绝不原地覆写。

2 · gitfs：把 git 当工作格式用

Engram 使用 git 的 blob/tree/commit 数据结构来表达内存版本，但从不 shell out 到 git 二进制，也不依赖 .git/ 目录。所有 git I/O 都通过 go-git 完成，配以自定义的 Storage 替换其默认的文件系统后端。

Storage：go-git 的自定义 Storer

internal/memstore/gitfs/storage.go 实现了 go-git 的 storage.Storer 接口，将对象层桥接到 ObjStore。它内嵌一个 memory.Storage，非对象数据（refs、config、index）委托给它——这些是 per-session 临时的，权威引用始终在 Postgres。

存储格式用 git loose-object 帧编码（storage.go:42–59）： 

// frame 编码: "<type> <size>\x00<content>"
// sha1(frame(o)) == o.Hash() — 这正是 git 的哈希定义
func frame(o plumbing.EncodedObject) ([]byte, error) {
    content, _ := io.ReadAll(r)
    buf.WriteString(o.Type().String())  // e.g. "blob"
    buf.WriteByte(' ')
    buf.WriteString(strconv.FormatInt(o.Size(), 10))
    buf.WriteByte(0)   // \x00 分隔符
    buf.Write(content)
    return buf.Bytes(), nil
}

因为用了 git 原生的 loose-object 编码，Engram 对象的 key 与真实 git 仓库的哈希完全一致—— 相同内容在任何 git 实现里得到同一个 SHA。这让未来可以用标准 git 工具直接验证或导出对象。

SetEncodedObject（storage.go:77–93）： 计算 hash → frame → objs.Put(h.String(), data)。 EncodedObject 反向：objs.Get → unframe → 返回 MemoryObject。

Materialize：将某个 commit 检出到目录

internal/memstore/gitfs/repo.go:37–59： 

func Materialize(ctx context.Context, objs objstore.ObjStore, at string, dir string) error {
    st := NewStorage(ctx, objs)
    seedRefs(st, at)                     // 在 in-memory Storage 中伪造 refs/heads/main 和 HEAD
    repo, _ := git.Open(st, osfs.New(dir))
    wt, _ := repo.Worktree()
    wt.Checkout(&git.CheckoutOptions{Hash: plumbing.NewHash(at), Force: true})
}

seedRefs 在内存 Storage 上设置一个临时 refs/heads/main 指向目标 commit， 让 go-git 能正常打开仓库，整个过程完全不接触磁盘 .git 目录。

Commit：将目录快照写入 ObjStore

repo.go:63–96： 

func Commit(ctx context.Context, objs objstore.ObjStore, parent string, dir string, msg string) (string, error) {
    st := NewStorage(ctx, objs)
    // parent == "" → git.Init（首次 commit）；否则 seedRefs + git.Open
    wt.AddWithOptions(&git.AddOptions{All: true})
    h, _ := wt.Commit(msg, &git.CommitOptions{Author: author(), AllowEmptyCommits: true})
    return h.String(), nil  // 十六进制 commit SHA
}

go-git 在 wt.Commit 内部为每个新 blob、tree、commit 调用 st.SetEncodedObject，最终通过 objs.Put 落盘。 返回值就是新 commit 的 SHA，随后被 CommitWithCAS 送进 Postgres CAS。

TreeKeys：给 L3 缓存提取不可变 key

repo.go:98–123 实现 TreeKeys： 直接解析 ObjStore 中的 commit 对象，提取 rootTree 哈希与 system/ 子树哈希，无需查询 Postgres。 这两个 SHA 是纯粹不可变的——任何文件变化都会产生新的 rootTree； system/ 子树 SHA 只在 system/ 目录内容变化时才变—— L3 缓存用它们作为永不过期的缓存键（详见② 请求路径）。

3 · 唯一可变指针 + compare-and-swap

整个系统中唯一被允许改变的指针是 Postgres 中的 agent_id → HEAD 行。 所有并发控制都收敛到对这个指针的一次原子 CAS。

agent_refs 表

-- 000001_init.up.sql
CREATE TABLE agent_refs (
  agent_id   text PRIMARY KEY,
  head       text NOT NULL,
  updated_at timestamptz NOT NULL DEFAULT now()
);

CommitRef：CAS 的 SQL 是什么

internal/memstore/refs/refs.go:259–265： 

ct, err := tx.Exec(ctx,
    `UPDATE agent_refs SET head=$1, updated_at=now()
     WHERE agent_id=$2 AND head=$3`,
    next, agentID, parent)
if ct.RowsAffected() == 0 {
    return ErrCASConflict
}

WHERE head=$3 是 compare 步骤：只有当数据库里存储的 HEAD 与调用方持有的 parent 一致，UPDATE 才命中一行（RowsAffected() == 1）。 如果另一个写者已先行提交，HEAD 已变更，RowsAffected() == 0， 函数返回 ErrCASConflict，调用方需重新读取最新 HEAD 后重试。

不变量 所有并发控制收敛于这一个 CAS。 系统没有乐观多写者合并、没有分布式锁、没有 CRDT。 Session Router 在上游确保同一 agent 同时只有一个写者； CAS 是最后一道防线，捕获任何意外竞争。 不存在的 agent 调用 CommitRef 也返回 ErrCASConflict（0 行更新）， 因此调用方必须先 Bootstrap。

ResolveHead 与 Bootstrap

ResolveHead（refs.go:36–46）： SELECT head FROM agent_refs WHERE agent_id=$1，缺失行返回 ErrAgentNotFound。

Bootstrap（refs.go:49–57）： INSERT … ON CONFLICT (agent_id) DO NOTHING，幂等设置初始 HEAD。

4 · 提交就是一次 Postgres 事务

CommitWithCAS（memstore.go:60–69）将写操作分为严格有序的两阶段：

  1. 对象优先（写 ObjStore） 调用 gitfs.Commit，把所有新 blob/tree/commit 对象写进 ObjStore。 写入幂等，可安全重试。崩溃只留下不可达的孤立对象，GC 宽限期后自动清理。
  2. 原子推进 ref（写 Postgres） 在一个 Postgres 事务内执行 CAS UPDATE agent_refs 并 INSERT INTO memory_jobs（reindex、reflect）。 要么整体成功，要么整体回滚——不存在"ref 已更新但任务未入队"的半提交状态。
设计亮点 同-tx 入队替代了 Kafka + outbox 模式。 任务入队与 ref 推进在同一个 Postgres 事务中，天然保证"任务入队 当且仅当 commit 已提交"， 无需独立的消息中间件，也无需 outbox 轮询。

5 · memory_jobs 队列表

memory_jobs 是一张 Postgres 表，充当 Engram 的维护任务队列——省掉了独立消息中间件。

表结构与关键设计

-- 000001_init.up.sql
CREATE TABLE memory_jobs (
  id         bigserial PRIMARY KEY,
  agent_id   text NOT NULL,
  kind       text NOT NULL,     -- "reindex" | "reflect" | "defrag" | "gc"
  from_sha   text,              -- 触发该任务的 commit SHA（用于增量处理）
  state      text NOT NULL DEFAULT 'pending',  -- pending | running | failed
  attempts   int  NOT NULL DEFAULT 0,
  created_at timestamptz NOT NULL DEFAULT now()
);

-- per-agent singleton：同一 (agent_id, kind) 最多一条 pending 行
CREATE UNIQUE INDEX memory_jobs_pending_uniq
  ON memory_jobs (agent_id, kind) WHERE state = 'pending';

-- 000002_claimed_at.up.sql  （第二次迁移）
ALTER TABLE memory_jobs ADD COLUMN claimed_at timestamptz;

三个关键字段：

  • partial unique index（WHERE state = 'pending'）： 同一 agent 同一 kind 最多一条待处理行。commit 重试时重复 INSERT 变成幂等空操作。
  • attempts：重试计数。RetryJob 在每次失败时递增； 达到 maxAttempts 时将 state 置为 'failed'，防止毒药任务循环崩溃 worker。
  • claimed_at（第二次迁移加入）：worker 认领时记录时间戳。 ReapStaleJobs 以此判断 worker 崩溃——凡 state='running' 且 claimed_at < before 的行，说明执行者已消失，安全地归还到队列。

6 · WithGlobalLock：Postgres advisory lock

部分维护任务（全局 GC、per-agent 单例保证）需要确保同一时刻只有一个 worker 在执行。 Engram 通过 Postgres session-level advisory lock 实现，无需独立的分布式锁服务。

实现

internal/memstore/refs/refs.go:84–109： 

func (r *Refs) WithGlobalLock(ctx context.Context, key int64, fn func(context.Context) error) (bool, error) {
    conn, err := r.pool.Acquire(ctx)  // 从连接池取出一个连接并持有
    defer conn.Release()

    var got bool
    conn.QueryRow(ctx, `SELECT pg_try_advisory_lock($1)`, key).Scan(&got)
    if !got {
        return false, nil  // 锁已被其他 session 持有 → 跳过本轮
    }

    // 解锁必须用 context.Background()，不能用请求 ctx
    defer func() {
        _, _ = conn.Exec(context.Background(), `SELECT pg_advisory_unlock($1)`, key)
    }()

    return true, fn(ctx)
}
坑点（真实 Bug） 解锁必须使用 context.Background()，而不是传入的请求 ctx。

Advisory lock 是 session 级的：它绑定在连接上，连接在 conn.Release() 后 归还连接池可被下一个请求复用。若解锁语句使用了已取消的请求 ctx， pg_advisory_unlock 不会执行，锁泄漏在连接上。 之后所有用同一 key 的 pg_try_advisory_lock 永远返回 false， 直到该连接被连接池回收。这是代码审查阶段捕获的真实 bug，已修复。

当前两处调用：

  • 全局 GC：保证同一时刻只有一个 worker 在执行 mark-sweep（key 固定）。
  • per-agent 维护（reflection、reindex）：传入不同的 key，实现 per-agent 单例。

7 · GC：可达性 mark-sweep + 宽限期

GC 是一个纯函数（internal/maintenance/gc.go），签名： 

func GC(ctx context.Context, objs objstore.ObjStore,
        reachable map[string]struct{}, grace time.Duration, now time.Time) (Stats, error)

它不查 Postgres、不看 refs——调用方负责预先算好 reachable 集合并传入。 这个设计让 GC 完全可以用 in-memory ObjStore 单元测试。

第一步：计算可达闭包（ReachableObjects）

internal/memstore/gitfs/reach.go:18–53 从所有 agent HEAD 出发做迭代 DFS，三个去重 map 防止重复遍历： 

func ReachableObjects(ctx context.Context, objs objstore.ObjStore, heads []string) (map[string]struct{}, error) {
    seenCommits := map[string]struct{}{}
    seenTrees   := map[string]struct{}{}
    stack := append([]string{}, heads...)

    for len(stack) > 0 {
        h := stack[len(stack)-1]; stack = stack[:len(stack)-1]
        if _, ok := seenCommits[h]; ok { continue }
        seenCommits[h] = struct{}{}

        c, err := object.GetCommit(st, plumbing.NewHash(h))
        if err != nil {
            return nil, fmt.Errorf("gitfs: reach commit %s: %w", h, err) // 失败 → nil
        }
        reachable[h] = struct{}{}
        addTree(reachable, seenTrees, st, tree)    // 展开整棵 tree
        for _, p := range c.ParentHashes {
            stack = append(stack, p.String())       // 保留完整历史
        }
    }
    return reachable, nil
}

seenTrees 的作用：多个 commit 共享同一子树（常见于小改动）时， 子树只被展开一次，大幅减少遍历量。遇到任何错误都返回 nil, err—— 调用方（GC runner）必须检测并中止，绝不能用不完整的 nil map 去 sweep。

第二步：sweep（按 key 逐一判定）

gc.go:32–68，删除条件：不可达 AND 超过宽限期： 

// 先快照 keys，再遍历（删除期间不能边 Iter 边 Delete）
objs.Iter(ctx, func(k string) error { keys = append(keys, k); return nil })

for _, k := range keys {
    if _, ok := reachable[k]; ok { s.Kept++; continue }  // 可达 → 跳过

    mt, err := objs.Stat(ctx, k)
    if errors.Is(err, objstore.ErrNotFound) { continue }  // 并发删除，跳过
    if err != nil { s.StatErrors++; continue }             // Stat 失败 → 保守保留

    if now.Sub(mt) > grace {
        if err := objs.Delete(ctx, k); err != nil {
            s.DelErrors++; continue   // Delete 失败 → 最佳努力，不中止
        }
        s.Swept++
    } else {
        s.Kept++  // 不可达但在宽限期内 → 保留（in-flight 写入保护）
    }
}

宽限期的作用

回顾两阶段提交：对象先写 ObjStore，然后才在 Postgres 推进 ref。 存在短暂时间窗口：对象已在 ObjStore，但 ref 尚未更新， 因此 ReachableObjects（从所有 HEAD 出发）看不到这些新对象。 宽限期（默认 1 小时）确保"年轻"对象不会被误删——它们很快就会被 ref 引用到。

GC 流程图

图 4 · GC 流程。可达集计算失败时直接中止，不进入 sweep；宽限期内的不可达对象保守保留。

不变量 永远不要对不完整的可达集合执行 sweep。 ReachableObjects 遇到任何错误都返回 nil, err。 GC runner 检测到 err 非 nil 时必须中止，不得调用 GC()。 这防止了"遍历中断→误判为垃圾→删除存活对象"的灾难。
边界 GC 只作用于 git 对象存储，从不触及搜索索引（embedding 存储）。 搜索索引有独立的生命周期管理（详见④ 搜索与索引）。 两者解耦，避免跨存储的可达性判断复杂度。

Stats 结构

type Stats struct {
    Scanned    int  // Iter 快照的 key 总数
    Swept      int  // 成功删除的不可达超龄对象数
    Kept       int  // 可达 或 宽限期内（保守保留）
    StatErrors int  // Stat 失败 → 保守保留，不计入 Kept
    DelErrors  int  // Delete 失败 → 最佳努力保留，不中止整个 GC
}

单个对象的 Delete 失败不会中止整个 GC（best-effort），记入 DelErrors 供 worker 日志/告警。GC 仅在 Iter 本身失败时返回顶层 error。
