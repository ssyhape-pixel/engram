Engram 深入解析

一座给 AI agent 用的"云端记忆系统"是怎么从零长出来的——从一块块不可变的存储,到在后台默默整理记忆的工人。

基于 main 分支 · 九层全部实现并合并 · 六份报告 + 本索引

想象每个 AI agent 都有一本只属于它的笔记本。它能随时翻看常驻的几页要点,需要时再去翻长尾的细节,还能把新学到的东西持久地写回去,下次开机依然记得。Engram 就是在服务端给成千上万个 agent 提供这种笔记本的系统。

它的做法刻意简单:写进去的每一块内容都不可更改(要改就写一块新的),而"这本笔记现在翻到第几页"这唯一会变的指针,放在一个数据库里、用一把"改之前先确认没人动过"的小锁守着。其余一切——缓存、搜索索引、工作副本——都是可以随时丢掉重算的派生物。整套系统因此能保持无状态、可水平扩展,却又强一致。

一句话抓住灵魂:记忆是"主动去取"的,不是"一股脑塞给模型"的。开口推理前只装载常驻要点和一份薄薄的目录;真正的细节,由模型在思考过程中自己决定要不要去翻。

六份报告

报告一

架构总览

四个区、三条贯穿全局的规矩、一次请求到一次提交的完整链路、为什么不上那些重型中间件。先读这份建立全局观。

报告二

请求路径

一个 agent 同时只有一个写者:进程内的小锁、跨机器的兜底、模型怎么"自己去翻"记忆、提交撞车了怎么办。

报告三

存储与一致性

系统的心脏:为什么每块内容都按指纹存、唯一会变的指针怎么守、"提交就是一次数据库事务"、垃圾怎么安全回收。

报告四

搜索与索引

怎么把"关键词"和"语义"两种找法揉到一起、昂贵的向量怎么算一次存下来反复用、后台怎么悄悄把它们备齐。

报告五

维护管线

后台那个工人每一轮都做什么:抢任务不打架、反思整理、拆大文件、回收卡死的活,以及它们如何复用同一套机制。

报告六

构建历程

它是怎么一层层盖起来的:每层都走"构思→设计→计划→实现→双重审查→合并",以及审查链路真正抓出来的那些坑。

九层地图

层加了什么带来的保证深读
L1存储内核:按指纹存对象 + 数据库守唯一指针内容不可变 + 唯一一处做并发裁决报告三
L2Agent 循环 + 多轮会话一个 agent 同时一个写者 + 模型自取记忆报告二
L3按指纹的读缓存命中即等于重算,永不会读到旧值报告四
L4关键词 + 语义混合搜索两种找法融合,缺一种也能降级工作报告四
L4b向量持久化 + 增量补算昂贵的向量只算一次、跨会话共享报告四
L5a后台垃圾回收只删"没人引用且足够旧"的对象报告三 / 五
L5b任务队列消费 + 反思多 worker 抢活不重不漏 + 不自我触发死循环报告五
L5c确定性拆分大文件按标题拆,且保证不会越拆越多报告五
L5d回收卡死的任务worker 崩在半路的活会被捞回来重做报告五

怎么读

  • 第一次看,想要全局观 → 报告一(架构总览)加报告六(构建历程),半小时建立框架。
  • 要接前台、改请求处理 → 报告二(请求路径),按需回看报告三的提交机制。
  • 关心一致性、存储、回收 → 报告三是心脏,最值得逐段读。
  • 做检索、向量相关 → 报告四。
  • 做后台、运维 → 报告五(队列、反思、拆分、回收、每轮循环)。
现状

规划中的整条路线已全部走完:10 个代码模块、2 个可运行进程(一个跑前台、一个跑后台),九次分层合并。全套测试串行跑全绿、关键模块并发检测干净。唯一刻意推后的是"向量库的淘汰"——因为向量是派生的、能随时重算、体量又小,目前没有压力。
