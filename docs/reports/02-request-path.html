② 请求路径 · L1–L4 · 基于 main 分支

请求路径：Router · Session · Agent Loop · Tools

从一条用户消息到一次持久提交——单写者保护的两层设计、上下文装配、工具驱动的记忆检索与写入，以及 CAS 提交协议。

1. 概览

Engram 的请求路径（架构区域 ①）是系统中唯一与 LLM provider 直接交互的部分。 它的哲学可以用一句话概括：计算无状态，状态在记忆仓库。

每一次对话对应一个 Session。Session 持有两样临时的东西——chat history 和物化的工作目录（workdir）——但这两样东西不构成持久状态。 真正的持久状态是 agent 的记忆仓库：内容寻址对象存储里的 git 树，加上 Postgres 里的 HEAD ref。 Session 结束时，工作目录被销毁。下次打开时，从 HEAD 重新物化——不会泄漏状态。

核心不变量 每个 agentID 同时最多一个活跃 Session。 Router 进程内用 sync.Mutex + active map 快速拦截同机并发； Postgres ref CAS（ErrCASConflict）是跨 pod 的最终兜底。 任何时候都不做有损的 Markdown 自动合并。

2. "唯一写者"：为什么 + 两层保护

为什么必须序列化写？

记忆文件是结构化散文 Markdown，不是数据库行。 两个写者并发修改同一文件时，任何自动三方合并（git merge 风格）都会产生语义错误——上下文被切断、事实被混淆，甚至凭空制造矛盾的句子。这类损坏是静默的、难以回溯的。

Engram 的选择是：彻底不走多写者路线，改用单写者序列化。 代价是并发度降低（每个 agent 同时只有一个 Session），但记忆的正确性得到了绝对保证。 两层保护各自管不同的危险面：

进程内锁 — 快路径

管什么：同一 pod 内的并发请求（同机多线程/多 goroutine）。

实现：sync.Mutex + active map[string]bool（router.go:32-33）。非阻塞准入：已占用立即返回 ErrAgentBusy，不排队等待。

价值：在昂贵的对象写入和 Postgres 事务发生之前就终止冲突，零额外延迟。

数据库 CAS — 跨机兜底

管什么：多 pod 场景下 sticky routing 失效、或进程外工具直接写 Postgres。

实现：refs.CommitRef 在单个 Postgres 事务里做 WHERE head=$parent 的原子更新；父 hash 不匹配则返回 ErrCASConflict（memstore/memstore.go:65）。

价值：即使进程锁被绕过，数据库层也保证 ref 不会被悄悄覆盖。详见 ③ 存储与一致性。

2.1 进程锁：claim / free

Router 的结构体（internal/agent/router.go:24-34）： 

type Router struct {
    store    memstore.MemStore
    prov     LLMProvider
    scratch  string
    cache    cache.Cache   // sysCache — L3 system/ 读缓存
    embCache cache.Cache   // embCache — L4 向量嵌入缓存（必须分离）
    emb      search.Embedder

    mu     sync.Mutex
    active map[string]bool
}

claim 与 free 都在 mu 下操作（router.go:45-60）： 

func (r *Router) claim(agentID string) bool {
    r.mu.Lock()
    defer r.mu.Unlock()
    if r.active[agentID] {
        return false          // 已被占用，立即拒绝
    }
    r.active[agentID] = true
    return true
}

func (r *Router) free(agentID string) {
    r.mu.Lock()
    delete(r.active, agentID)
    r.mu.Unlock()
}
设计选择 锁的语义是准入，而非等待。 ErrAgentBusy 的含义是"现在不行"——调用方决定重试策略（立即返回 429、短暂 sleep 重试，还是入上层队列）。 Router 本身不持有任何排队状态，与"无状态 worker"哲学完全一致。

2.2 sync.Once 防双重释放

Open 把 release 包裹在 sync.Once 里再传给 Session（router.go:125-131）： 

var once sync.Once
release := func() {
    once.Do(func() {
        os.RemoveAll(workdir)
        r.free(agentID)
    })
}

场景：调用方同时执行 defer sess.Close() 和显式 sess.Close()，或者错误路径与正常路径都触发了关闭。 没有 sync.Once 时，第二次 free 会删除 active map 里的条目——而这个条目可能已经被另一个新 Session 重新设置，造成它被错误地"释放"。 sync.Once 保证 RemoveAll + free 作为一个整体只执行一次，无论关闭被调用多少次。

2.3 两层保护各管什么（对比一览）

维度进程内锁（sync.Mutex）数据库 CAS（Postgres）
保护范围单 pod 内所有 goroutine所有 pod、所有进程外写者
触发时机Open() 入口，物化之前commit 时，对象已写完之后
冲突表现ErrAgentBusy，立即失败ErrCASConflict，提交失败
副作用无（什么都没写）只有 garbage 对象（GC 清理）
多 pod 能力不能（锁不跨进程）能（Postgres 是全局协调点）

3. 一次会话的生命周期

Session 是单次对话的完整生命周期容器（internal/agent/session.go:22-34）： 

type Session struct {
    agentID  string
    store    memstore.MemStore
    prov     LLMProvider
    tools    *Toolset
    head     memstore.CommitHash  // 当前已提交的 HEAD（随 commit 推进）
    workdir  string               // 物化的临时工作目录
    history  []Message            // 本 Session 的 chat history（纯内存）
    dirty    bool                 // 本轮是否有成功的 edit
    maxSteps int                  // 防失控循环，默认 16
    release  func()               // Close 时调用：RemoveAll + free
    cache    cache.Cache          // L3 SHA 键缓存；nil 则每轮重算
}

history 只活在本 Session 生命周期里。 dirty 是本轮（Send 调用）的写标记，commit 成功后归零。 head 在每次成功提交后推进，始终对应对象存储中一个真实存在的 commit。

3.1 Open 的完整步骤

  1. claim(agentID) 在 mu 下检查 active map；已占用则立即返回 ErrAgentBusy。
  2. ResolveHead(agentID) 从 Postgres 读取当前 HEAD commit hash（router.go:96）。失败则立即 free + 返回错误。
  3. MkdirTemp 在 scratch 目录下创建临时工作目录。agentID 中的 / 被替换为 _ 以兼容 MkdirTemp 模式（router.go:104-106）。
  4. Materialize(HEAD, workdir) 调用 gitfs.Materialize，将 HEAD commit 所指向的 git 树内容写入工作目录（router.go:111）。
  5. readWorkdirFiles + NewHybrid 读取 workdir 全部文件为 map[string][]byte，构造 search.HybridSearch 快照（router.go:116-122）。
  6. sync.Once release 闭包 + NewSession 组装 release 函数（Once 包裹），返回 *Session（router.go:125-132）。

3.2 Send：一个 Turn 的完整时序

Send（session.go:73-117）是请求路径的核心循环。进入前记录 base := len(s.history)，失败时回滚到 base，保证 history 不留悬空记录。

  1. history += user 消息 追加 Message{Role: RoleUser, Text: userMessage}。
  2. assembleSystem(ctx) 拼装系统提示：常驻 system/ 内容 + tree index。可能命中 L3 缓存（见§4）。
  3. Generate(System, history, Tools) 调用 LLMProvider。返回 ToolCalls（轮次未完成）或最终 Text。
  4. Dispatch 每个 ToolCall Toolset.Dispatch 路由到 list / read / recall / edit。edit 成功则 s.dirty = true（session.go:98-99）。
  5. history += assistant-toolcalls + tool-results 追加两条消息，再回到步骤 2 继续循环。至多 maxSteps=16 次迭代。
  6. 模型返回最终文本，跳出循环 len(resp.ToolCalls) == 0 时 break，history += assistant 文本。
  7. commit（仅 dirty=true 时） 调用 store.CommitWithCAS，先写对象后推 ref（见§7）。成功后 head = newHead, dirty = false。

失败回滚：步骤 2/3 出错，或循环耗尽 maxSteps，执行 s.history = s.history[:base]。 commit 阶段（步骤 7）失败时不回滚 history，dirty 保持 true——下一轮 Send 可重提交。

4. 上下文装配

每一步 Generate 之前，assembleSystem 构造完整的系统提示字符串。它由两部分拼接：

部分来源缓存键（L3）作用
常驻 system/ 内容 workdir 下 system/ 目录全部文件 sys:<systemSubtree-SHA> 模型每轮都能看到的核心记忆（persona、规则、关键事实）
Tree index walkDir 全树，读各文件 frontmatter description: idx:<rootTree-SHA> "路径: 描述"索引，让模型知道长尾文件的存在以便按需召回

4.1 缓存分支 vs 直接重算

完整逻辑（session.go:141-170）：

func (s *Session) assembleSystem(ctx context.Context) (string, error) {
    if s.cache == nil || s.dirty {
        return s.buildResident()  // 无缓存或有未提交修改：从 workdir 重算
    }
    rootTree, systemSubtree, err := s.store.TreeKeys(ctx, s.head)
    if err != nil {
        return s.buildResident() // TreeKeys 读失败不致命，退化为重算，不中断会话
    }
    sysKey := "sys:" + string(systemSubtree)
    sys, ok := s.cache.Get(sysKey)
    if !ok {
        built, berr := s.buildSystemContent()
        if berr != nil { return "", berr }
        sys = built
        s.cache.Put(sysKey, sys)
    }
    idxKey := "idx:" + string(rootTree)
    idx, ok := s.cache.Get(idxKey)
    if !ok {
        built, berr := s.buildTreeIndex()
        if berr != nil { return "", berr }
        idx = built
        s.cache.Put(idxKey, idx)
    }
    return sys + idx, nil
}
缓存正确性 缓存键是 不可变的 git tree hash，由 TreeKeys 从对象存储直接读取，不经过 Postgres。 只要 HEAD 未推进，system/ 子树内容不变，sys:SHA 就永远有效。 dirty=true 时（workdir 有未提交改动），缓存被完全绕过，直接从 workdir 重算——保证模型看到最新的编辑，而不是缓存里的旧版本。 TreeKeys 读失败时也静默退化为重算，不中断会话。 L3 缓存的底层实现详见 ④ 搜索与索引。

4.2 buildSystemContent — 常驻内容

（session.go:187-213） 遍历 workdir/system/，每个文件以 ## path\ncontent\n 格式追加到 builder。 若 system/ 目录不存在，WalkDir 触发 fs.ErrNotExist，被转换为 fs.SkipAll（静默处理，返回空字符串）。 真实 I/O 错误（权限问题、磁盘故障）则完整传播——防止把残缺内容缓存起来。

4.3 buildTreeIndex — 廉价目录

（session.go:217-238） 遍历 workdir 全树，每个文件调用 frontmatterDescription(path)（tools.go:172-192）——用 bufio.Scanner 逐行扫描，找到 YAML frontmatter 的 --- 分隔符，提取 description: 字段值。 这是只读 frontmatter，不读文件体，token 开销极小。 所有行排序后拼接，格式为 path: description。

设计亮点 记忆是 agentic 的，不是 RAG 的。 Tree index 只给模型"线索"（路径 + 描述），不把长尾内容全部塞入上下文。 模型在推理过程中根据 index 判断哪些文件相关，再通过 recall / read 工具精确拉取，既节省 token，又保持信息的语义准确性。

5. 工具：模型怎么"自己去翻"

Toolset（internal/agent/tools.go:18-25）绑定到单个 Session 的 workdir，是无状态的。 所有工具通过 Dispatch(ctx, ToolCall) ToolResult 路由。 Session 负责追踪 dirty；Toolset 本身不持有写标记。

5.1 工具定义（发给模型的 Schema）

（tools.go:30-35）四个工具：

// list — 列出记忆文件（内含 description，即 tree index）
{Name: "list",
 Description: "List memory files with their descriptions (the tree index).",
 InputSchema: {"type":"object","properties":{}}}

// read — 读取文件，可选 1-based 行范围
{Name: "read",
 Description: "Read a memory file, optionally a 1-based inclusive line range.",
 InputSchema: {"type":"object",
   "properties": {"path":{"type":"string"},
                  "start":{"type":"integer"},   // 1-based，含
                  "end":  {"type":"integer"}},  // 1-based，含
   "required": ["path"]}}

// recall — 在记忆中搜索，返回匹配行范围
{Name: "recall",
 Description: "Search memory for a query; returns matching line ranges.",
 InputSchema: {"type":"object",
   "properties": {"query":{"type":"string"}},
   "required": ["query"]}}

// edit — 写入记忆文件（新建或覆写）
{Name: "edit",
 Description: "Write content to a memory file (creates or overwrites).",
 InputSchema: {"type":"object",
   "properties": {"path":{"type":"string"}, "content":{"type":"string"}},
   "required": ["path","content"]}}

5.2 recall — 搜索长尾，返回行范围

（tools.go:140-153） doRecall 调用 t.search.Recall(ctx, t.agentID, query, 8)，返回至多 8 个 Hit{Path, LineStart, LineEnd, Snippet}，序列化为 path:start-end: snippet 格式传回模型： 

for _, h := range hits {
    fmt.Fprintf(&b, "%s:%d-%d: %s\n", h.Path, h.LineStart, h.LineEnd, h.Snippet)
}

L2 的后端是 GrepSearch（子串扫描 workdir）；L4 替换为 HybridSearch（trigram + 向量混合）。 Search 接口不变，工具代码不感知切换。详见 ④ 搜索与索引。

5.3 read — 按行范围读取

（tools.go:110-138）

start, end := intInput(in, "start"), intInput(in, "end")
if start <= 0 && end <= 0 {
    return okResult(callID, string(data))   // 无范围：返回全文
}
// 边界收紧后：
return okResult(callID, strings.Join(all[start-1:end], "\n"))
注意：1-based 行号 read 的 start/end 参数是 1-based 闭区间，与 recall 返回的 LineStart/LineEnd 完全对齐。 模型拿到 recall 返回的 notes.md:5-12，直接传 start=5, end=12 给 read 即可取回对应行，零转换。 内部实现是 all[start-1 : end]（Go 切片 0-based 开区间）。 若改为 0-based API，模型每次都需要做减一转换，极易引入 off-by-one 错误。 start<=0 && end<=0 明确表示"不限范围"，与 1-based 不产生歧义（合法行号 >= 1）。

5.4 edit — 写入 workdir，置 dirty

（tools.go:155-168） doEdit 先调用 safePath（tools.go:47-54）做 filepath.Clean + 前缀检查，拒绝 .. 逃逸路径。 然后 os.MkdirAll 创建父目录，os.WriteFile 覆写（或新建）文件。 Session 的 Send 在 Dispatch 返回后检查 call.Name == "edit" && !res.IsError，将 s.dirty 置为 true（session.go:98-99）。

6. 跟模型对话的抽象

6.1 接口

（internal/agent/provider.go:62-64）

type LLMProvider interface {
    Generate(ctx context.Context, req Request) (Response, error)

边界价值 这个接口是 agent loop 与任何 LLM provider 之间的唯一边界。 Session 从不 import Anthropic SDK，也不直接发 HTTP 请求。 FakeProvider 用同一接口驱动全部自动测试，零网络调用。 换提供商只需实现此接口，Session / Toolset 代码一行不改。

6.3 FakeProvider — 脚本化确定性驱动

（internal/agent/fake.go）

type FakeProvider struct {
    Steps []func(req Request) Response
    calls int
}
func (f *FakeProvider) Generate(ctx context.Context, req Request) (Response, error) {
    if f.calls >= len(f.Steps) {
        return Response{}, fmt.Errorf("fake: no scripted response for call %d (have %d)", f.calls, len(f.Steps))
    }
    step := f.Steps[f.calls]
    f.calls++
    return step(req), nil
}

每个 Step 接收完整的 Request（含 system 提示和 history），可在其中做断言——验证上下文装配是否正确。 典型测试脚本：Step 0 返回 recall ToolCall，Step 1 返回 edit ToolCall，Step 2 返回最终文本。 调用次数超出脚本长度时报错，防止测试漏写步骤时静默通过。

6.4 AnthropicProvider — 真实 API 接入

（internal/agent/anthropic.go） 将 provider 无关协议映射到 Anthropic Messages API（POST /v1/messages），默认模型 claude-sonnet-4-6，max_tokens=4096，版本头 anthropic-version: 2023-06-01。 HTTP client 可注入（anthropic.go:33-36），便于测试时替换 mock RoundTripper。

消息格式映射（anthropic.go:88-121）：

内部 RoleAnthropic 表示注意事项
RoleUser user + text block
RoleAssistant（有文本） assistant + text block + 可选 tool_use blocks text 为空时不加 text block
RoleAssistant（仅工具调用） assistant + tool_use blocks blocks 为空时跳过整条消息（anthropic.go:108-110）
RoleTool user + tool_result blocks Anthropic 要求工具结果作为 user 角色发回
Anthropic API 两个关键守卫

(1) tool_use block 的 input 字段不能为 null——nil map 被替换为空 map（anthropic.go:103-105）。

(2) assistant 消息至少要有一个 content block，否则 API 报 400——空 blocks 时直接 continue 跳过整条消息（anthropic.go:108-110）。

apiRequest 的 system 和 tools 字段均标注 omitempty，为空时不序列化到 JSON，避免 Anthropic API 对空字段的校验报错。

7. 提交与撞车

7.1 两步写序：对象先于 ref

CommitWithCAS（memstore/memstore.go:60-68）严格遵循"先写对象，再推 ref"：

gitfs.Commit 写 blob + tree + commit 对象到 ObjStore
内容寻址，幂等
成功后
refs.CommitRef 单一 Postgres tx：
CAS HEAD parent→new
+ 入队 reindex + reflect

若对象写成功但 Postgres tx 失败（或进程在此处崩溃），只留下不可达的 garbage 对象——GC（L5a）负责清理，不会产生 torn ref。 反向情况（ref 先推进而对象未落盘）在这个写序下不可能出现。

7.2 commit 同时入队两个维护 job

// session.go:121
jobs := []memstore.Job{{Kind: "reindex"}, {Kind: "reflect"}}

这两个 job 在同一个 Postgres 事务内随 ref CAS 一起入队（见 ⑤ 维护管线）。 这保证"job 入队当且仅当 commit 成功"——不需要 Kafka，不需要 outbox 表：Postgres tx 本身就是原子单元。

7.3 ErrCASConflict — 暴露，而非覆盖

（session.go:123-128）

if errors.Is(err, memstore.ErrCASConflict) {
    // Impossible under single-writer-per-agent (Router lock). If it occurs,
    // the invariant was violated; surface it rather than silently clobbering
    // the other writer's tree (no lossy merge). Multi-writer reconciliation is L5.
    return fmt.Errorf("agent: commit conflict — single-writer invariant violated for %s: %w",
        s.agentID, err)
}
设计决策：不重试、不覆盖

在正确的 Router 单写者保护下，ErrCASConflict 理论上不可发生。 一旦出现，意味着存在旁路写入——进程外工具直接修改 Postgres，或多 pod 下 sticky routing 失效。

曾被考虑但被否决的方案：有界重试 + clobber（重新 materialize 新 HEAD，重放本轮 edit，再提交）。 被否决的根本原因：

  • "重放 edit"需要将本轮的修改内容合并到对方改动后的文件树上——对散文 Markdown 这是有损操作，可能产生语义错误。
  • 丢弃对方的写入同样违反"无有损合并"原则。
  • 多写者协调（reconcile + reflection）是 L5 的职责，不应在 Session 层悄悄修复。

当前行为：将错误透明上报，让调用方知晓不变量被破坏，而不是静默"修复"一个不该在此层修复的问题。

7.4 commit 全流程小结

步骤位置失败后果
1. gitfs.Commit — 写所有对象到 ObjStore memstore.go:61 只有 garbage 对象，ref 未变，GC 清理，Session dirty 保持 true
2. refs.CommitRef — CAS HEAD + 入队 jobs（单一 Postgres tx） memstore.go:65 ErrCASConflict（单写者被违反）或通用 DB 错误，上报调用方
3. Session 更新 head，dirty=false session.go:132-133 commit 失败时 dirty 保持 true，下一轮 Send 可重提交

    prov = agent.NewAnthropic(key)
default:
    prov = &agent.FakeProvider{Steps: []func(agent.Request) agent.Response{
        func(r agent.Request) agent.Response {
            return agent.Response{Text: "(fake) received your message"}
        },
    }}
}

// 向量嵌入器选择
switch env("ENGRAM_EMBEDDER", "fake") {
case "voyage":
    emb = search.NewVoyage(vkey)
default:
    emb = search.NewFakeEmbedder(256)
}

// 嵌入缓存（分层：内存 LRU + 持久化 ObjCache）
embCache := cache.NewTiered(cache.NewLRU(4096), cache.NewObjCache(objstore.NewLocal(embObjRoot)))

// 防御性检查：sysCache 与 embCache 必须使用不同根目录
if filepath.Clean(embObjRoot) == filepath.Clean(objRoot) {
    log.Fatalf("ENGRAM_EMB_OBJ and ENGRAM_OBJ must be different directories")
}

// Router 装配
router := agent.NewRouter(store, prov, os.TempDir(), cache.NewLRU(1024), embCache, emb)

最后两行是关键：sysCache（cache.NewLRU(1024)）和 embCache（cache.NewTiered）是不同实例，底层对象根目录也不同。 若混用同一实例，system/ 的纯文本内容会被写入向量嵌入存储，导致类型混淆和后端污染。 embObjRoot == objRoot 时程序启动即 Fatal，是对这一不变量的主动防御。
