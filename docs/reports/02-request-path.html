请求路径：Router · Session · Agent Loop · Tools

从客户端消息到持久化记忆提交的全链路——单写者保护、上下文装配、工具调用与 CAS 提交。

1. 概览

Engram 的请求路径（区域 ①）是唯一与 LLM provider 直接交互的路径。 其设计哲学是：计算无状态、存储有状态。 一个 Session 代表一次与某个 agent 的对话；它持有临时的 chat history 和物化的工作目录，但真正的持久状态始终是记忆 repo（对象存储中的 git 树 + Postgres HEAD ref）。 Session 结束时工作目录被销毁——下次打开时从 HEAD 重新物化，天然无泄漏。

组件关系如下：

核心不变量 Session 是唯一写者。 Router 在进程内通过 sync.Mutex + active map 保证每个 agentID 同时只有一个活跃 Session。 跨 pod 的兜底是 Postgres ref CAS（ErrCASConflict）。 任何时候都不做有损的 markdown 合并。

2. Router：单写者

2.1 为什么需要单写者

记忆文件是结构化散文 Markdown。两个写者并发修改同一文件时，任何自动三方合并都是有损的——语义上下文会被丢失或混淆。 因此 Engram 的设计决策是：对同一 agent 的写操作必须完全串行化，而不是乐观并发。 Router 是进程内的第一道关口；Postgres ref CAS 是跨 pod 的最终兜底。

2.2 进程内锁：claim / free

Router 用一个 sync.Mutex 保护一张 active map[string]bool：

// internal/agent/router.go:24-34
type Router struct {
    store    memstore.MemStore
    prov     LLMProvider
    scratch  string
    cache    cache.Cache   // sysCache — L3 system/ 读缓存
    embCache cache.Cache   // embCache — L4 向量嵌入缓存（分离实例）
    emb      search.Embedder

    mu     sync.Mutex
    active map[string]bool
}

internal/agent/router.go:45-60 claim(agentID) 在锁下检查 active 映射：已存在则返回 false（非阻塞，不排队）；不存在则置为 true。 free(agentID) 在锁下删除条目。 Open 调用 claim，失败时立即返回 ErrAgentBusy，让调用方决定是否重试或排队。 

// internal/agent/router.go:45-53
func (r *Router) claim(agentID string) bool {
    r.mu.Lock()
    defer r.mu.Unlock()
    if r.active[agentID] {
        return false
    }
    r.active[agentID] = true
    return true
}
设计选择 锁的语义是准入，而非等待。 ErrAgentBusy 意味着"现在不行"，调用方决定重试策略。 这与整体"stateless workers"哲学一致：worker 不持有任何排队状态。

2.3 幂等释放：sync.Once

internal/agent/router.go:124-131 Open 把 release 包裹在 sync.Once 中再传给 Session： 

var once sync.Once
release := func() {
    once.Do(func() {
        os.RemoveAll(workdir)
        r.free(agentID)
    })
}

这保证即使 Session.Close() 被调用两次（defer + 显式调用），也不会错误地释放一个其他 Session 可能已重新获取的 claim。

2.4 sysCache vs embCache：两个独立缓存实例

internal/agent/router.go:38-42 NewRouter 接受两个分离的 cache.Cache 实例：

2.5 跨 pod 的兜底

进程内锁只能保护单 pod 范围。多 pod 部署时，两个 pod 各自持有不同 agent 的锁，不会冲突；但若 sticky routing 失效，同一 agent 可能被两个 pod 同时服务。 此时兜底是 Postgres ref CAS（见 ③ 存储与一致性）：后写者会收到 ErrCASConflict，而不是静默覆盖。 多 pod sticky routing 是 L5 的工作。

3. Session 生命周期

3.1 Session 的字段

internal/agent/session.go:22-34

type Session struct {
    agentID  string
    store    memstore.MemStore
    prov     LLMProvider
    tools    *Toolset
    head     memstore.CommitHash  // 当前已提交 HEAD
    workdir  string               // 物化的临时工作目录
    history  []Message            // 本 Session 的 chat history（临时）
    dirty    bool                 // 本轮是否发生过 edit
    maxSteps int                  // 防失控循环，默认 16
    release  func()               // Close 时调用：RemoveAll + free
    cache    cache.Cache          // L3 SHA 键缓存；nil = 每轮重算
}

history 是纯内存、仅在本 Session 生命周期内有效。 dirty 是本轮的写标记，commit 成功后归零。 head 在每次成功提交后推进，始终对应对象存储中的真实 commit。

3.2 一个 Turn 的完整时序

internal/agent/session.go:73-117

sequenceDiagram
  participant U as 调用方
  participant S as Session
  participant P as LLMProvider
  participant T as Toolset
  participant W as workdir
  participant M as MemStore

  U->>S: Send(userMessage)
  S->>S: history += user消息

  loop 至多 maxSteps=16 步
    S->>S: assembleSystem(ctx)
    Note over S: sys/ 常驻内容 + tree index
    S->>P: Generate(System, history, Tools)
    alt 返回 ToolCalls（未完成）
      P-->>S: ToolCalls
      S->>S: history += assistant-toolcalls
      loop 每个 ToolCall
        S->>T: Dispatch(call)
        T->>W: list / read / recall / edit
        T-->>S: ToolResult
      end
      S->>S: history += tool-results
      Note over S: edit 成功则 dirty=true
    else 返回最终文本（完成）
      P-->>S: Text
      S->>S: history += assistant-text
      Note over S: 跳出循环
    end
  end

  alt dirty=true
    S->>M: CommitWithCAS(head, workdir, reindex+reflect)
    M-->>S: newHead
    S->>S: head=newHead, dirty=false
  else 纯读轮
    Note over S,M: 不提交，HEAD 不变
  end

  S-->>U: 最终文本

3.3 历史回滚机制

internal/agent/session.go:74 Send 在进入循环前记录当前 history 长度 base := len(s.history)。 若 assembleSystem 或 Generate 失败，或 tool 循环超出 maxSteps， 都执行 s.history = s.history[:base] 回滚，避免在 history 中留下悬空的 user 消息。 commit 阶段失败则不回滚 history，保留 dirty=true，下一轮可重提交。

4. 上下文装配

每一步 Generate 前，assembleSystem 返回系统提示的完整字符串，由两部分拼接而成：

部分来源缓存键（L3）作用
常驻 system/ 内容 workdir 中 system/ 目录下全部文件 sys:<systemSubtree-SHA> 模型每轮都能看到的核心记忆（persona、规则、关键事实）
Tree index walkDir 全树，提取 frontmatter description: idx:<rootTree-SHA> "路径:描述"索引，让模型知道长尾文件的存在，按需 recall/read

4.1 缓存分支 vs 直接重算

internal/agent/session.go:141-170

func (s *Session) assembleSystem(ctx context.Context) (string, error) {
    if s.cache == nil || s.dirty {
        return s.buildResident()  // dirty 或无缓存：从 workdir 重算
    }
    rootTree, systemSubtree, err := s.store.TreeKeys(ctx, s.head)
    if err != nil {
        return s.buildResident() // TreeKeys 读失败不致命；退化为重算
    }
    sysKey := "sys:" + string(systemSubtree)
    sys, ok := s.cache.Get(sysKey)
    if !ok {
        built, berr := s.buildSystemContent()
        if berr != nil { return "", berr }
        sys = built
        s.cache.Put(sysKey, sys)
    }
    idxKey := "idx:" + string(rootTree)
    idx, ok := s.cache.Get(idxKey)
    if !ok {
        built, berr := s.buildTreeIndex()
        if berr != nil { return "", berr }
        idx = built
        s.cache.Put(idxKey, idx)
    }
    return sys + idx, nil
}
缓存正确性 缓存键是 immutable git tree hash，由 TreeKeys 从对象存储直接读取（不经 Postgres）。 只要 HEAD 未推进，system/ 子树的内容不变，sys:SHA 永远有效。 workdir 有未提交改动时（dirty=true），缓存被绕过，直接从 workdir 重算——保证模型看到最新编辑。 详见 ④ 搜索与索引（L3 缓存实现）。

4.2 buildSystemContent

internal/agent/session.go:187-213 遍历 workdir/system/，若目录不存在则返回空（通过 fs.SkipAll 静默处理，不报错）。 任何真实读取错误（权限、I/O）都被传播，以防缓存到部分内容。 

b.WriteString("# Resident memory (system/)\n")
// 对每个文件：
fmt.Fprintf(&b, "\n## %s\n%s\n", rel, string(data))

4.3 buildTreeIndex

internal/agent/session.go:217-238 遍历 workdir 全树，对每个文件调用 frontmatterDescription(path)（见 §5.4）， 构造 path: description 行，排序后输出。 这是一个廉价索引（只读 frontmatter，不读文件体），token 开销小。

设计亮点 记忆是 agentic 的，不是 RAG 的。 Tree index 只给模型"线索"（路径 + 描述），不把全部长尾内容灌入上下文。 模型在推理过程中用 recall / read 工具按需拉取长尾， 既节省 token，又保持信息的精确性。

6.3 FakeProvider — 脚本化确定性驱动

internal/agent/fake.go

type FakeProvider struct {
    Steps []func(req Request) Response  // 按调用序号取脚本
    calls int
}
func (f *FakeProvider) Generate(ctx context.Context, req Request) (Response, error) {
    if f.calls >= len(f.Steps) {
        return Response{}, fmt.Errorf("fake: no scripted response for call %d", f.calls)
    }
    step := f.Steps[f.calls]
    f.calls++
    return step(req), nil
}

每个 Step 接收完整的 Request（含 system 提示和 history），可在其中断言上下文是否符合预期。 典型测试脚本：Step 0 返回 recall ToolCall，Step 1 返回 edit ToolCall，Step 2 返回最终文本。 调用次数超出脚本长度时返回 error，防止测试写漏步骤时静默通过。

7. 提交与 CAS 冲突

7.1 commit 的两步写序

internal/agent/session.go:120-135 commit 调用 store.CommitWithCAS，后者内部严格遵循"先写对象，再推 ref"的顺序： 

flowchart LR
  A["gitfs.Commit\n写 blob+tree+commit 对象到 ObjStore\n(幂等，内容寻址)"]
  B["refs.CommitRef\n单一 Postgres tx:\nCAS HEAD parent->new\n+ 入队 reindex + reflect job"]
  A -->|成功后| B

internal/memstore/memstore.go:60-68 若对象写成功但 Postgres tx 失败（或崩溃），只留下不可达的 garbage 对象——GC（L5a）会清理，不会产生 torn ref。 反之不可能：ref 先推进而对象未落盘是不存在的状态。

7.2 commit 同时入队两个维护 job

// internal/agent/session.go:121
jobs := []memstore.Job{{Kind: "reindex"}, {Kind: "reflect"}}

这两个 job 在同一个 Postgres tx 内随 ref CAS 一起入队（见 ⑤ 维护管线）。 这保证"job 入队当且仅当 commit 成功"——无需 Kafka，也无需 outbox 表：Postgres tx 就是天然的原子单元。

7.3 ErrCASConflict — 暴露而非覆盖

internal/agent/session.go:123-128

if errors.Is(err, memstore.ErrCASConflict) {
    // Impossible under single-writer-per-agent (Router lock). If it occurs,
    // the invariant was violated; surface it rather than silently clobbering
    // the other writer's tree (no lossy merge). Multi-writer reconciliation is L5.
    return fmt.Errorf("agent: commit conflict — single-writer invariant violated for %s: %w",
        s.agentID, err)
}
设计决策：不重试、不覆盖

在正确的 Router 单写者保护下，ErrCASConflict 理论上不可发生。 一旦出现，意味着存在旁路写入（进程外的工具直接修改 Postgres，或多 pod 下 sticky routing 失效）。

曾被考虑但被拒绝的方案：有界重试 + clobber（重新 materialize 新 HEAD，重放本轮编辑，再提交）。 被拒绝的原因：rebase 需要将本轮 edit 的内容合并到别人修改后的树上，对散文 Markdown 而言是有损操作； 丢弃对方的写入同样违反"无有损合并"原则。 正确的多写者协调（reconcile + reflection）是 L5 的职责。

当前行为：将错误透明上报，让调用方知晓不变量被破坏，而不是静默修复一个不该静默修复的问题。

7.4 commit 全流程小结

步骤位置失败后果
1. gitfs.Commit — 写所有对象到 ObjStore memstore/memstore.go:61 只有 garbage 对象，ref 未变，GC 清理
2. refs.CommitRef — CAS HEAD + 入队 jobs（单一 Postgres tx） memstore/memstore.go:65 ErrCASConflict（单写者违反），或通用 DB 错误
3. Session 更新 head，dirty=false session.go:132-133 commit 失败时 dirty 保持 true，下一轮可重提交

    emb = search.NewFakeEmbedder(256)
}

// 嵌入缓存（分层：内存 LRU + 持久化 ObjCache）
embCache := cache.NewTiered(cache.NewLRU(4096), cache.NewObjCache(objstore.NewLocal(embObjRoot)))

// Router 装配（sysCache 和 embCache 必须分离）
router := agent.NewRouter(store, prov, os.TempDir(), cache.NewLRU(1024), embCache, emb)

cmd/api/main.go:81-83 代码在此处检查 embObjRoot 与 objRoot 是否相同，相同则 Fatal。 这是对 sysCache vs embCache 必须分离 这一不变量的主动防御。
