④ 搜索与索引

Trigram 倒排索引 × 语义向量检索，通过 RRF 融合；embedding 持久化让增量 reindex 免费获得。

1. 设计立场：记忆是主动的，不是 RAG

Engram 的 recall 是一个中循环工具调用，由模型在推理过程中按需发起，不是预推理阶段的 top-k 填充。 推理开始时只加载两类内容：

  • system/ 常驻文件——直接进入系统提示，由 L3 SHA 读缓存加速（见 §5）；
  • 树索引（文件名 + description 字段）——让模型知道长尾记忆的存在。

只有当模型判断某段长尾记忆与当前任务相关时，才调用 recall(query, k) 拉取精确的行范围， 类似"按需分页"而非"提前倒入"。

搜索变体 A（默认）：Trigram 索引直接存储文件行的文本内容，查询时零对象存储访问。 整个索引在 Session 打开时于内存中从已物化的工作目录重建，随 Session 关闭而丢弃——它是可再生的派生视图， 不是权威存储（符合核心不变量 ③）。

不变量 索引（trigram + semantic）是派生的、可丢弃的视图。真相永远在 object store 的 content-addressed 对象 + Postgres ref 里。 缓存崩溃、索引丢失，均可从对象重建，不影响正确性。

2. Trigram 倒排索引

TrigramIndex（trigram.go）是一个行级倒排索引， 以 3-rune 小写 shingle 为键，存储 (path, line) 的 posting 列表。 关键设计点：它保留行的原始文本（lines map[string][]string）， 因此查询时无需访问任何外部存储——纯内存 CPU 操作，零 I/O。

核心实现

// trigram.go:21–37
func trigrams(s string) []string {
    r := []rune(strings.ToLower(s))
    if len(r) < 3 { return nil }
    seen := make(map[string]struct{}, len(r))
    out := make([]string, 0, len(r)-2)
    for i := 0; i+3 <= len(r); i++ {
        t := string(r[i : i+3])
        if _, ok := seen[t]; ok { continue }
        seen[t] = struct{}{}
        out = append(out, t)
    }
    return out
}

去重是关键：同一个 trigram 在一行中出现多次只记一次，避免 posting 列表膨胀。

候选集求交

candidates(query)（trigram.go:65–102）：

  1. 将查询拆成 trigram 集合；
  2. 每个 trigram 的 posting 列表转为 map[posting]struct{}；
  3. 从最小的集合开始做交集（减少无效遍历）；
  4. 对于少于 3 个字符的查询（无 trigram），退化为全量扫描。

最终 Search 做了双重过滤：trigram 交集是预过滤器（可能有假阳性）， strings.Contains 子字符串检查是精确谓词。 

// trigram.go:107–135 (Search)
func (t *TrigramIndex) Search(query string, k int) []Hit {
    q := strings.ToLower(query)
    var matches []posting
    for p := range t.candidates(q) {
        if strings.Contains(strings.ToLower(t.lineText(p.path, p.line)), q) {
            matches = append(matches, p)
        }
    }
    // 按 path, line 排序保证输出稳定
    sort.Slice(matches, func(i, j int) bool { ... })
    ...
}
特性说明
粒度行级 posting，返回单行 Hit（LineStart == LineEnd）
大小写索引和查询均 lowercase，snippet 保留原始大小写
重建成本纯 CPU，Session 打开时与物化树同步完成
假阳性trigram 交集后有 substring 精确验证，不漏报也不错报
短查询长度小于 3 时全量扫描（退化为 grep）

3. 语义检索

语义索引（semantic.go）将文件按 Markdown 一级标题分段（sectionize）， 对每一段计算 embedding 向量，查询时用余弦相似度排名。 与 trigram 不同，语义检索的匹配单元是段落而非行， 因此能捕捉跨行、跨表达方式的语义相关性。

分段逻辑

sectionize（semantic.go:40–52）： 逐行扫描，遇到以 # 开头的行（即任意级别标题）就划出一个段落边界。 第一个标题前的内容作为独立段落，文件无标题则整个文件为一段。 返回的 section 携带 1-based 行范围，用于 Hit 定位。

Embedding 键设计

内容寻址是语义持久化的核心。每个段落的 embedding key： 

// semantic.go:63–66
func embKey(model, text string) string {
    sum := sha256.Sum256([]byte(text))
    return "emb:" + model + ":" + base64.RawStdEncoding.EncodeToString(sum[:])
}

model 前缀确保不同 embedding 模型的向量不会混用（如从 fake 切换到 voyage-3）。 sha256(text) 确保内容相同的段落复用缓存，内容变化则生成新 key。

BuildSemantic：缓存命中 → 复用，缺失 → 批量 Embed → Put

// semantic.go:107–156 (BuildSemantic，核心逻辑)
for _, p := range paths {
    for _, sec := range sectionize(string(files[p])) {
        ci := len(si.chunks)
        si.chunks = append(si.chunks, chunk{...})
        key := embKey(emb.Model(), sec.text)
        if c != nil {
            if enc, ok := c.Get(key); ok {
                si.chunks[ci].vec = decodeVec(enc)
                continue              // 缓存命中：直接复用，不调用 Embed
            }
        }
        pend = append(pend, pending{ci: ci, text: sec.text, key: key})
    }
}
if len(pend) > 0 {
    vecs, err := emb.Embed(ctx, texts)   // 批量调用一次
    ...
    for i, pp := range pend {
        si.chunks[pp.ci].vec = vecs[i]
        if c != nil {
            c.Put(pp.key, encodeVec(vecs[i]))  // 写入缓存
        }
    }
}

所有缺失的段落汇总后一次批量调用 Embed，最小化 API 往返。 写入缓存（c.Put）在拿到向量后立即执行。 如果 c == nil（未注入缓存），则跳过所有缓存逻辑，退化为纯内存模式。

向量编解码

向量存为 base64 编码的 little-endian float32 字节串（semantic.go:68–86）， 与 cache.Cache 的 string 接口直接兼容，无需额外序列化层。

4. RRF 融合

HybridSearch.Recall 调用 trigram + semantic 各取 max(k, 10) 条， 再通过 Reciprocal Rank Fusion 合并为最终 top-k 列表。

RRF 实现

// hybrid.go:56–85
func rrf(lists [][]Hit, k int) []Hit {
    const rrfK = 60
    type agg struct { hit Hit; score float64 }
    scores := map[string]*agg{}
    var order []string
    for _, list := range lists {
        for rank, hit := range list {
            id := fmt.Sprintf("%s:%d-%d", hit.Path, hit.LineStart, hit.LineEnd)
            a, ok := scores[id]
            if !ok {
                a = &agg{hit: hit}
                scores[id] = a
                order = append(order, id)
            }
            a.score += 1.0 / float64(rrfK+rank+1)
        }
    }
    sort.SliceStable(order, func(i, j int) bool {
        return scores[order[i]].score > scores[order[j]].score
    })
    ...
}

RRF 公式：每条结果在某列表中排名 r（0-based）对融合得分的贡献为 1 / (60 + r + 1)。常数 60 平滑了头部排名差异，是 RRF 的标准超参数。 命中 ID 以 path:lineStart-lineEnd 标识，出现在多个列表中的结果得分累加。 SliceStable + order（首次出现顺序）保证同分时稳定 tie-break。

降级策略

NewHybrid（hybrid.go:22–28）在构建时 best-effort 构建语义索引—— 若 BuildSemantic 报错（如 embedding 服务不可用），h.sem 保持 nil， Recall 仅使用 trigram 列表，不报错。查询时 semantic 失败也同样静默降级： 

// hybrid.go:41–47
if h.sem != nil {
    if semHits, err := h.sem.Search(ctx, query, n); err == nil {
        lists = append(lists, semHits)
    }
    // 查询时 embedding 失败：丢弃语义，保留 trigram（降级）
}
不变量 Recall 永远不因 embedding 服务故障而报错。最差情况退化为纯 trigram 检索， 覆盖精确词汇匹配场景。见测试 TestHybridDegradesWhenEmbeddingFails。

Trigram vs Semantic：互补的盲区

查询类型TrigramSemantic

跨行语义短语（如 "alpha beta gamma" 分三行）不命中强（段落向量捕捉词汇分布）
同义词 / 近义描述不命中强
无 embedding 服务正常工作降级（被忽略）

5. L3 SHA 读缓存

cache.Cache（cache.go:13–16）是一个极简的字符串 KV 接口： 

type Cache interface {
    Get(key string) (string, bool)
    Put(key, val string)
}

所有 key 都是内容哈希，因此命中 = 等价于重新计算，不存在失效问题。 "失效"的唯一形式是 LRU 淘汰（空间压力），不影响正确性，只影响缓存率。

LRU（cache.go:23–69）：

  • 单 sync.Mutex 保护所有操作——Get 需要修改 recency list（MoveToFront），无法用 RWMutex；
  • 容量上限为 entry 数量（默认 1024），超出时淘汰最旧项；
  • 同一 key 重复 Put 会刷新 recency，不增加 entry 数。

L3 缓存在 Session 层有两个独立实例：

  • sysCache（cache.NewLRU(1024)）——缓存 system/ 内容的组装结果，详见报告 ②；
  • embCache（Tiered LRU + ObjCache）——缓存 embedding 向量，详见 §6。
不变量 sysCache 与 embCache 是不同实例，且指向不同的后端 ObjStore 目录。 system 内容不会写入 embedding 存储，embedding 向量不会污染 system 缓存。 见 NewRouter 注释："they MUST be different instances"（router.go:39）。

6. Embedding 持久化（L4b）

Voyage API 调用昂贵（时延 + 费用），而段落内容一旦写入就不可变（content-addressed）—— 同一段落的 embedding 在任何时间、任何 worker 上的结果都相同。 L4b 将 embedding 向量持久化到独立的 content-addressed ObjStore， 使"昂贵的 Voyage 调用"退化为"廉价的对象存储读"。

ObjCache：ObjStore 上的 Cache

// objcache.go:21–24
func safeKey(key string) string {
    sum := sha256.Sum256([]byte(key))
    return hex.EncodeToString(sum[:])
}

embKey（来自 semantic.go）已经是 base64 字符串，但可能含 : 等字符， 不安全用于文件路径或 S3 键。safeKey 再次 sha256 + hex 编码，得到纯十六进制的 flat key。 这意味着对象键是键的哈希而非值的哈希，但唯一性由 embKey 的 sha256(text) 保证。

ObjCache 的所有错误都降级为 miss / no-op： 

// objcache.go:26–38
func (o *ObjCache) Get(key string) (string, bool) {
    data, err := o.objs.Get(context.Background(), safeKey(key))
    if err != nil { return "", false }   // error = miss
    return string(data), true
}

func (o *ObjCache) Put(key, val string) {
    _ = o.objs.Put(context.Background(), safeKey(key), []byte(val))   // ignore error
}

Tiered：LRU 前 + ObjCache 后

// tiered.go:11–25
func (t *Tiered) Get(key string) (string, bool) {
    if v, ok := t.front.Get(key); ok { return v, true }
    if v, ok := t.back.Get(key); ok {
        t.front.Put(key, v)    // 从持久层提升到内存层
        return v, true
    }
    return "", false
}

func (t *Tiered) Put(key, val string) {
    t.front.Put(key, val)
    t.back.Put(key, val)    // 双写：LRU + ObjStore
}

读取时先查 LRU（快），LRU miss 再查 ObjCache（磁盘/S3），命中则提升到 LRU。 写入同步双写，保证持久化与内存缓存一致。

注入方式：BuildSemantic 不感知持久化

// cmd/api/main.go:80–84
embObjRoot := env("ENGRAM_EMB_OBJ", "./engram-embeddings")
if filepath.Clean(embObjRoot) == filepath.Clean(objRoot) {
    log.Fatalf("ENGRAM_EMB_OBJ and ENGRAM_OBJ must be different directories")
}
embCache := cache.NewTiered(cache.NewLRU(4096), cache.NewObjCache(objstore.NewLocal(embObjRoot)))

BuildSemantic 接受 cache.Cache 接口，不知道背后是 LRU、ObjCache 还是 Tiered。 持久化完全通过依赖注入实现，semantic.go 零代码改动。

收益 Session 首次打开时的 BuildSemantic 调用，若 reindex worker 已预热， 所有段落均命中 ObjCache，Voyage API 调用数为零。 跨 Session 重启、跨 pod、跨进程，embedding 持久化透明生效。

7. 增量 Reindex

maintenance.Reindex（reindex.go:23–46）是 L4b 的维护端： 它物化 agent HEAD 的完整树，调用 BuildSemantic， 主要目的不是返回索引，而是触发 BuildSemantic 的 Put 副作用—— 将所有段落的 embedding 写入持久化 embCache。 返回的 *SemanticIndex 被丢弃。 

// reindex.go:41–44
// BuildSemantic persists missing embeddings into embCache as a side-effect.
if _, err := search.BuildSemantic(ctx, emb, embCache, files); err != nil {
    return fmt.Errorf("maintenance: reindex build semantic: %w", err)
}

为什么不需要 diff？

内容寻址使增量天然免费：

  1. BuildSemantic 对每个段落计算 embKey(model, text)；
  2. c.Get(key)——持久化 embCache 已有的段落直接命中，跳过 Embed；
  3. 只有新增或修改的段落（key 不存在）才会调用 Voyage API；
  4. 第二次 reindex 对同一 HEAD 调用 Voyage 次数为零（所有段落已持久化）。

这一行为通过 countingEmbedder 测试精确验证： 

// reindex_test.go:46–52
// Second reindex: every chunk's embedding is already persisted → 0 new embeds.
if err := Reindex(ctx, store, ce, embCache, agentID); err != nil { ... }
if ce.calls != first {
    t.Fatalf("second reindex embedded %d more texts, want 0 (incremental persistence broken)", ce.calls-first)
}

Reindex 与 Session 共享 embCache 的前提

cmd/api 的 ENGRAM_EMBEDDER 和 cmd/maintenance 的 ENGRAM_EMBEDDER 必须使用同一 embedding 模型（默认均为 "fake"，生产均为 "voyage-3"）。 两者的 embKey 都携带 model 前缀，模型不同则 key 不同，缓存无法共享。

潜在问题 若 ENGRAM_PROVIDER=anthropic（LLM 提供商）与 ENGRAM_EMBEDDER=fake（embedding 模型）配置不一致， Session 和 maintenance worker 使用不同 embedder model，embKey 前缀不同， reindex 预热的 embedding 对 Session 完全无效，退化为每次 Session 均重新调用 Voyage。 生产部署应确保两端 ENGRAM_EMBEDDER 配置一致。

8. GC 隔离

Engram 的 GC（L5a，maintenance/gc.go）通过可达性分析清除无引用的 git 对象。 GC 操作的是 git object store（ENGRAM_OBJ）， embedding 向量存储在独立的目录（ENGRAM_EMB_OBJ，默认 ./engram-embeddings）。 两者是不同的 objstore.NewLocal 实例，GC 的 reachable 集合计算完全不涉及 embedding store。

两个 cmd 均在启动时强制检查目录分离： 

// cmd/api/main.go:81–83
if filepath.Clean(embObjRoot) == filepath.Clean(objRoot) {
    log.Fatalf("ENGRAM_EMB_OBJ and ENGRAM_OBJ must be different directories (got %q)", embObjRoot)
}

// cmd/maintenance/main.go:133–135
if filepath.Clean(embObjRoot) == filepath.Clean(objRoot) {
    log.Fatal("ENGRAM_EMB_OBJ must differ from ENGRAM_OBJ (GC must never sweep the embedding store)")
}
不变量 GC 永远不会清除 embedding 向量。embedding store 是只增长的内容寻址存储： 旧段落的向量不会被引用，但也不会被 GC，因为它们不在 git object store 里。 embedding 的"过期"管理（若需要）是独立于 git GC 的问题，目前未实现（可接受：记忆体量小）。

关于 git GC 的完整分析见报告 ③ 存储与一致性。

设计总结

L4 核心

Hybrid = Trigram + Semantic

词汇精确性 + 语义模糊性，RRF(k=60) 融合；embedding 失败时静默降级为纯 trigram。

L4b 收益

Embedding 持久化

Tiered(LRU + ObjCache) 注入 BuildSemantic；逻辑零改动，Voyage 调用大幅减少。

增量免费

Content-addressed Keys

embKey = sha256(model+text)；已持久化的段落天然命中缓存，无需 diff。

安全隔离

GC 不碰 Embedding Store

独立 ObjStore 根目录 + 启动时路径相等检查，双重保障。
