00设计目标与不变量

目标是让一个 agent 在云端长期、跨 session 地拥有可自我编辑的记忆，同时满足多租户隔离、水平扩展、可审计、可控成本。整套设计反复复用同一组心智模型——把它们当作贯穿全文的不变量，后面每个环节都是它的推论。

三条不变量 · 整套设计的脊椎

objects 不可变

记忆的文件内容、目录树、提交都按自身 hash 做内容寻址。不可变意味着：能当缓存键、能跨租户去重、能做 mark-sweep GC、同一 hash 永不映射到不同内容。

refs 可变且强一致

唯一可变的是指针 agent_id → HEAD。所有并发、顺序、一致性全部收口在对它的原子 CAS 上。它住在 Postgres 里，并顺手兼任任务队列。

其余皆为可重建的派生视图

缓存、trigram 索引、worker 本地工作副本——丢了都能从权威 objects 重算。所以它们全部无状态、可随意扩缩与重启。

01总览架构

系统分四个区：① 请求路径处理用户回合，② 读加速层降低延迟与 token 成本，③ 权威存储是唯一有状态、保证持久与一致的层，④ 维护层在后台做记忆整理。三类流向在图中用样式区分——实线 = 数据读取，粗青线 = 写入（commit），虚线 = 派生更新（reindex）。

02① 请求路径

Session router — 按 agent 串行化

消息进来的第一站。它做的最重要的事是把同一 agent 的写串行化：保证任意时刻一个 agent 只有一个写者。这是整个系统唯一需要串行的点——因为客户端模式下「一个人一次一个 session」这个隐式串行器在云端没有了，必须显式地重建它。

实现可以很轻：把同一 agent_id 的请求路由到固定 worker（session 亲和），或者用一把 per-agent 锁（Redis 锁 / 数据库行锁）。默认选「单写者」而非「乐观并发」，因为记忆是散文 markdown，多写者并发的 git 自动 merge 是有损的。

Agent worker — 无状态 pod

真正跑 agent loop 的地方。它是无状态的：session 开始时从权威存储 checkout 该 agent 的记忆到本地临时盘，结束时提交回去；本地盘只是个可丢弃的缓存。这让它可以随 k8s 任意扩缩、被回收、重启。worker 持有 bash / 文件 / git 工具（指向本地工作副本），agent 直接编辑记忆文件。

LLM provider — 推理

纯外部依赖，只负责推理。worker 把组装好的 context（含 system/ 常驻记忆与召回片段）发给它。它不持有任何状态。

03② 读加速层

Memory cache — 按 commit SHA 做键

缓存的是物化好的 system/（每轮必进 context 的常驻记忆）与 memory tree 索引，省掉每轮回源对象存储的开销。关键的设计决定是用 commit SHA 当键，因为这把「缓存失效」这个最难的问题直接消解掉了。

为什么按 SHA

git 的 commit SHA 是内容寻址的——一个 SHA 唯一且不可变地对应整棵树的一个快照。于是 sha → 物化内容 是不可变映射：内容变了就是另一个 SHA、另一个键。所以缓存永不失效，传统的 invalidation 退化成纯粹为回收内存的 LRU/TTL 淘汰。它还顺带是个 ETag（比较版本只需比 SHA）、让多 worker 共享同一份字节、并保证后台改写记忆时前台仍读到一致快照。

真正需要保鲜的不是这份内容缓存，而是可变指针 agent_id → HEAD——那住在 Postgres 里（见 ③）。新内容 = 新 SHA = 缓存 miss = 重新物化回填，旧条目自然淘汰。

Search service — 本地 trigram 索引

负责语义/关键词召回。它持有一个 trigram 倒排索引（zoekt / bleve / tantivy 一类），这是「grep at scale」的成熟技术。变体 A（推荐）：索引里连文件内容一起存，查询完全在本地 shard 内完成、不碰对象存储——因为记忆是小文本，内容重复一份可忽略。变体 B：索引只存 (object hash, 字节 offset, 长度) 指针，查询时再对对象存储发一个很小的 Range GET 取命中字节，offset 因内容寻址而永远稳定。

04召回问题与 grep 复刻

客户端 agent 的 grep 之所以高效，本质是计算紧贴数据：在本地盘上扫、返回子文件的命中行、只有命中的几行进 context。云端把计算和数据分到两地后，「全量召回整个文件」会同时带来两笔成本——跨平台 egress（文件越大、跨 region 越疼）与更天天疼的 context 膨胀 + 二次扫描负担。

修法是把搜索搬到数据那边，只让命中区域跨线回来——也就是把过滤下推到存储（与 ClickHouse 的谓词下推同形）。query 出去很小，重活（扫描）留在数据侧，回来的只有命中行。

诚实的残留成本

grep 的行为可以完全复刻（lexical、子文件、查询廉价、regex 都行），复刻不了的是它的零基础设施属性——客户端就是文件 + grep 二进制，云上你要花一个索引 + 一个搜索服务 + commit 时的增量更新来买回同样体验。代价不在 egress（可解）也不在处理压力（可解），而在多出来的活动部件。务实做法：先把召回粒度从「文件」降到「行范围」（这步几乎不加组件、收益最大），trigram 索引按规模再上。

Search 与 Object storage 怎么交互

核心：它俩主要在构建时耦合，在查询时几乎不耦合。

构建时：一次 commit 后，search 拿到「HEAD 从 old 走到 new」，做 git diff old..new 得到精确的变更文件清单，只取这几个对象、增量更新 trigram postings——没变的文件因内容寻址而 postings 照旧有效。查询时：变体 A 完全本地作答、不碰对象存储；变体 B 多一个很小的 Range GET。索引是派生视图，丢了能从对象重建，按 agent_id 分片、冷 agent 按需加载。它最终一致——刚写的记忆有几秒搜不到，但你召回的多是旧记忆，通常无所谓。

05③ 权威存储

这是唯一有状态、保证持久与一致的层。它把两个性质对立的东西分开放：不可变的内容、可变的指针。

Object storage — 内容寻址的对象

每个文件内容、目录树、提交都按自身 hash 存为 objects/<hash>，幂等写（已存在就跳过）。内容寻址天然是对象存储的甜区：海量、廉价、高耐久（11 个 9）、可去重。严格租户隔离时按 per-tenant 前缀放，放弃跨租户去重以免泄漏「某文件是否存在」。它在查询关键路径上几乎不出现——读由缓存与索引挡住，写是幂等 PUT。

Postgres — refs（HEAD·CAS）+ 任务队列

可变指针 agent_id → HEAD 住在这里，因为我们需要对它做原子 CAS（HEAD: old → new，失败说明有人动过 HEAD）。整个系统的并发与顺序就收口在这一个点上。它还顺手兼任任务队列——这正是轻量化的关键（见 ④ 与 ⑦）。

写顺序

永远先写 objects，后动 Postgres。即便 CAS 前崩了，落下的 object 只是暂时不可达的垃圾（等 GC 收），绝不会出现 ref 指向一个没写全的 commit 的撕裂状态。

06④ 维护层

后台只有一个独立的 Go worker（与前台 worker 分开部署）。它把记忆维护的三件事做掉：reflection（反思近期对话、固化记忆）、defrag（整理记忆文件、拆大合重、重建层级）、GC（清理不可达对象、打包小对象）。reindex（更新 search 索引）也是它的一项任务。

拿活的两种姿势

  • 对账循环（最轻）：死循环 poll Postgres——「哪些 agent 的 HEAD 比我记录的 processed_sha 新？」refs 表本身就是工作清单，处理完更新游标。崩了重启从状态重新推导，天然崩溃安全。就是 k8s controller 的 reconcile 模式。
  • River 队列（要真队列时）：直接 dequeue commit 事务里入队的 job。重试退避、周期任务、per-agent unique（单例）都现成，且就跑在同一个 Postgres 里，没有额外集群。
不阻塞前台

维护任务改记忆时用 git worktree——在另开的工作目录里改、commit、merge 回主分支。前台 worker 同时仍读自己 SHA-pin 住的快照，互不干扰，只在最后 merge / CAS 时对账一次。per-agent 单例靠一把锁或 River 的 unique job 保证，别让两个 defrag 同时上同一个 agent。

07三条数据流

读（前台，实线）

router 按 agent 串行 → worker 组装 context：system/ 走 SHA-keyed cache，历史召回走 search service（本地作答），仅 miss 时回源对象存储 materialize → LLM 推理。全程无状态、可水平扩。

写（commit，粗青线）— 轻量化最漂亮的一处

worker 提交分两步：(1) 把内容寻址的新 object 幂等写进对象存储；(2) 在 Postgres 的同一个事务里 做 ref 的 CAS 外加 把维护任务入队。

一事务 = 干掉 Kafka + outbox

CAS 与 enqueue 在同一事务里原子完成，等价于「任务入队 ⟺ commit 成功」。这一下把整套 Kafka + outbox + event bus 全消解了——因为根本没有第二个系统要写，「双写不一致」这个问题不存在。worker 一提交就立刻返回，不等任何下游。

维护（后台，虚线 / 派生）

Go worker 通过对账循环或 River 拿到活，在 worktree 里跑 reflection / defrag / GC，并按 git diff 增量更新 search 索引。它读写对象存储、读 Postgres 拿工作，全部异步、不在用户回合的关键路径上。

08为什么不用 Temporal / Kafka

它俩回答的是两个正交问题：Temporal = 谁来跑长任务且抗崩溃（执行）；Kafka = 下游怎么知道发生了 commit 且不拖累前台（通知）。本设计两个都不用，原因如下。

组件它解决什么为什么这里不需要
Temporal多步、长命、需中途断点续跑的复杂 workflow记忆维护任务幂等、可从状态重推、不算长；接受「失败整段重跑」即可丢掉 durable resume，于是只需 队列/轮询 + 重试 + 单例 + 调度——全是轻量货
Kafka解耦写路径、多消费者扇出、持久重放下游消费者就一个（维护 worker）；用 Postgres 同事务入队即可，省掉 dual-write 一致性难题

判断标准很简单：任务的「重跑成本」和「中间状态」大不大——大才上 Temporal；消费者多不多、是否要重放——多才上 Kafka。本场景两者都不达标，所以用「Postgres 里几张表 + 一个 Go 进程」就够，透明、可控、无黑盒集群。

09落地路线

不要一开始就把四个区全建齐。按压力增量演进：

  1. 最小可跑：无状态 worker + 对象存储 + Postgres(refs) + 一个跑对账循环的 Go worker，GC 用 cron 兜底。没有队列、没有 Kafka、没有 Temporal。
  2. 读加速：加 SHA-keyed cache（先 per-pod 内存即可），降低回源。
  3. 召回升级：召回先从「整文件」降到「行范围」（search 工具返回区间 + Range GET）；有规模/延迟压力再上独立的 trigram search service。
  4. 队列升级：需要事务性入队与更体面的重试/调度时，把对账循环换成 River——还是同一个 Postgres，不引入新集群。
  5. 就近 grep：grep 保真度要求高时，引入 warm per-agent sidecar 持有工作树、本地索引/grep。
